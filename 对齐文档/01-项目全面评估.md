## 1. 架构与分层

### 1.1 分层结构(标准三层 + AI 子系统)
```
controller (42) ──REST/SSE──> service/impl (38) ──> mapper (55, MyBatis-Plus)
                                    │
                          ┌─────────┼──────────────────────────┐
                          ▼         ▼                          ▼
                    ai/* 子系统   event 监听器(6,@Async @TEL)   scheduler(5 个 @Scheduled)
                 (gateway/agent/
                  client/mode/router/
                  prompt/asr)
```
- 主轴 `controller → service → mapper` 整洁一致;`entity`(纯字段)与 `dto/vo` 分离;`common`/`exception`/`config` 集中。**Mapper 100% MyBatis-Plus `BaseMapper`,0 个 XML,仅 1 个 mapper 有自定义 `@Select`**。

### 1.2 越层 / 耦合热点(风险)
- **Controller 直连 Mapper 绕过 Service**:6 个 controller(`AuroraChatController`/`DiaryController`/`PortraitController`/`SocialController`/`UserController`/`UserPreferenceController`),跨 controller 共 **28 处** Mapper 字段。典型 `AuroraChatController.java:47-50` 直接持 `DialogMessageMapper` 并手写 `QueryWrapper`(:249-257)。业务与权限逻辑散落 controller,无法复用/测试。
- **god-class `AuroraAgentServiceImpl`(1401 行 / 66 方法 / 23 依赖 / ~15 职责)**:安全门 + 上下文装配 + Prompt 构建 + 结构化调用重试 + 输出消毒 + 异步画像反思 + 告别检测 + SSE 流式 + 手写 JSON 序列化 全聚于此(`service/impl/AuroraAgentServiceImpl.java`)。最长方法 `produceReply:166-320`(~155 行,混 DB 读 + 21 链 PromptBuilder + LLM 重试 + 循环 DB 写 + 异步调度)。
- **AI 子系统两套调用风格并存**:部分 agent 走门面 `StructuredAiService`(`MemoryExtractAgent:35`/`LetterGuardAgent:19`/`CapsuleAgent.converse:95`),部分直连 `LlmClient.chat`(`AuroraAgent:26`/`CapsuleAgent.generateUserPersona:53`/`ModeSwitchService:91`)。"agent" 抽象未统一,遗留薄壳 `AuroraAgent`(28 行)与真承载逻辑的 `AuroraAgentServiceImpl` 并存。

### 1.3 AI 子系统架构(亮点)
- **统一接口** `ai/client/LlmClient.java:5`(`chat`/`streamChat`,`RESPONSE_MAX_TOKENS=4096`)+ 三 provider(Glm/MiniMax/DeepSeek,各 ~230 行,**~80% 复制粘贴**,可收敛 ~400 LOC 到共享基类)。
- **Failover 链** `ai/client/FailoverLlmClient.java:12`(Chain 模式,逐 provider 尝试,末尾兜底 Mock);**仅 prod 模式启用**(`LlmConfig.java:289`),demo/dev 走"单 provider 内重试→自带 mock"。默认 active=minimax,链:minimax→mimo→glm→deepseek→mock。
- **Agent loop**(`produceReply :166-321`):同步安全门(:153)→ 硬边界(:169)→ 上下文装配(:193-211)→ PromptBuilder(:212)→ `callWithRetry`(:632,只对 timeout/502/503/connection 重试)→ `sanitizeLlmOutput`(:669)→ 每 5 轮异步画像反思(:300)。**注意:不是经典工具调用迭代,而是"带重试+兜底的结构化单次调用管线"**。
- **PromptBuilder**(416 行)是唯一 prompt 出口,含注入消毒 `sanitize:406`(折叠空白 + 剥离"忽略/你是/system"话术)、分级截断、置信度阈值。**系统设计优秀点**。
- **模型路由** `ai/router/SessionModelRouter.java:53`(session.preferred > profile.preferred > 默认;keyless provider 不进 `namedLlmClients` → 回落)。温度贯通:`AuroraAgentServiceImpl:243` → `StructuredAiService:135` → `GlmLlmClient:128`。
- **ASR** `asr/AsrClient.java:3` + 工厂 `AsrConfig:23`(MiMo/Glm/Mock 适配器,key 空落 mock)。`MimoAsrClient` 用 base64 data-URL + magic-byte 判 mime。
- **`MockLlmClient`(529 行)**:不是"假数据",是用 `PseudoSemanticAnalyzer` 做语义分析后按模块手搓结构化 JSON 的**确定性引擎**,是 demo/offline 的真后端。既是亮点(无 key 可跑可测),也是债(529 行 `String.format` 拼 JSON,脆弱)。

### 1.4 调度 / 事件 / SSE / 线程池
- **5 个 `@Scheduled`**:`AuroraProactiveJob`(90s,全表扫 user 串行 tick)、`LetterDeliveryJob`(60s,SENT→FLYING→DELIVERED 两段投递,带重试)、`CapsuleSyncRetryJob`(60s,续跑 FAILED 同步)、`NightlyMemorySettlementJob`(cron 0 0 2 * *,分页 200/批 重算引力/主题/基线/能量衰减)、`SessionIdleWatcher`(5min,空闲告别)。
- **SSE 主动推送** `ProactiveDeliveryChannel.java:19`:**进程内 `Map<userId, Set<SseEmitter>>`** + `SseEmitter(0L)` 无超时 → **多实例部署会丢推送**(无 Redis pub-sub 广播,无心跳)。
- **Aurora "假流式"**:`POST /stream-stage` 暂存富上下文 → 服务端先 `produceReply` 生成全文并落库 → 再分段 drip 给前端(`AuroraAgentServiceImpl:414-426`);只有 provider 侧 `streamChat` 才是真流。
- **线程池** `aiExecutor`/`taskExecutor` 同一 Bean(core 4 / max 8 / queue 100,`ThreadPoolConfig:13`),承载 SSE drip + 6 个 @Async 监听 + 画像反思 + ASR。**高并发 queue 满会 `RejectedExecutionException`(默认 Abort)**。
- **6 个 `@TransactionalEventListener(AFTER_COMMIT)+@Async`**(`MemoryExtractListener` 等):事务提交后异步抽取记忆/情绪/引力/Todo/capsule,失败仅记日志不阻断主流程。

### 1.5 前后端集成
- 全 `@RestController` + `/api/**`;42 controller 中 **38 个用 `ApiResponse<T>` 信封**(`{success,code,message,data}`),一致性好。
- **响应契约分裂(M-044 残留)**:错误路径返回裸 `Map`(键 `success/error/message/status/timestamp`,`GlobalExceptionHandler.java:86-94`)与成功 `ApiResponse`(键 `code/data`)**字段名不同**。前端只判 `json.success`/`json.message` 所以"能用",但契约事实上分裂。
- `AuroraSelfController` 全用 `ResponseEntity<?>` 裸响应,未收口到信封。
- 前端 `IC.*` namespace(`static/js/app.js` 831 行单体对象 + 50+ 页内联 `pages/*.html`),无构建链,可维护性随页面数线性下降。XSS 转义纪律良好(`IC.esc:333-340`,~199 处 `innerHTML` 均经转义)。

### 3.2 Schema 设计质量
- **主键**:全局 `id-type: auto`,`BIGINT AUTO_INCREMENT`;例外 2 张单例表(`tb_aurora_self_profile`/`constitution`,INT PK=1)。
- **外键**:仅 **9 条 FK**(对话消息/思绪碎片/边界/替身聊天/信件流水等 CASCADE);**所有 `user_id` 都是软外键**(无 DB 级 FK),删用户不级联清理其数据。
- **UNIQUE 约束(全库最强完整性防线)**:`tb_memory_card(user_id,source_session_id)`、`tb_emotion_trace(user_id,source_session_id)`、`tb_daily_record(user_id,record_date)`、**`tb_capsule_usage_quota(visitor,capsule,date)`**、`tb_user_portrait(user_id,dim)`、`tb_aurora_self_model(user_id,dim,status)`、`tb_agent_user_relationship(user_id)` 等,精准命中"去重/配额/每日/每维"语义。
- **索引缺口(中低危)**:`tb_dialog_message` 缺 `(session_id,created_at)` 复合;`tb_notification` 未读列表缺 `(user_id,is_read,created_at)`;`tb_persona_chat_message` 连 session_id 索引都没有;日志/时序表无 `created_at` 索引;`tb_capsule_sync_queue.findRetryable` 的 `next_retry_at` 未索引。
- **软删除**:全库**无软删除列**(`@TableLogic` 零命中),删除即物理删除 → 对情绪/记忆 PII 数据是合规风险(无回收站、无审计回溯)。
- **审计字段**:`created_at`/`updated_at` 几乎全表有,但 DDL **无 `ON UPDATE CURRENT_TIMESTAMP`**,`updated_at` 完全依赖 `MybatisMetaObjectHandler`(Java 侧填);原生 JdbcTemplate 路径需手写(目前一致,但隐式约束易在新代码破坏)。

### 3.3 MyBatis-Plus 用法
- `BaseEntity`(`id`+`createdAt`(INSERT fill)+`updatedAt`(INSERT_UPDATE fill))55 实体 100% 继承。
- **分页:全项目仅 1 处 `Page<>` 用法,且 `PaginationInnerInterceptor` 从未注册**(`MybatisPlusConfig.java` 裸 `@MapperScan`,无 `@Bean`)→ `selectPage` 不追加 LIMIT(见 §3.5 R1)。
- **逻辑删除 / 乐观锁**:`@TableLogic`/`@Version` 全库零命中,并发写为 last-writer-wins。
- 自定义 SQL 极少(54/55 mapper 纯 BaseMapper),0 个 XML。

### 3.4 H2-file vs MySQL 双模式
| profile | DB | sql.init.mode | LLM | seed |
|---|---|---|---|---|
| dev(默认) | H2-file `./data/innercosmos` MODE=MySQL | **always** | prod 真实 key | (开关失效,见 R2) |
| mysql | MySQL localhost:3306 | **never** | **mock**(profile 覆盖) | true |

## 5. 安全 / 测试 / 可运维性

### 5.1 认证 / 授权
- **Session-based(非 JWT,类名 `JwtAuthenticationFilter` 有误导)**:`JwtAuthenticationFilter.java:41-58` 从 `HttpSession.getAttribute("LOGIN_USER_ID")` 读 uid,仅 `status=ACTIVE` 建上下文,`role=ADMIN` 才 `ROLE_ADMIN`。
- ✅ 历史 `X-User-Id→ROLE_ADMIN` 漏洞已修(过滤器只认 session);登录防会话固定(`AuthController:34-40` invalidate+建新);BCrypt 强度 12;登录失败统一报错(无用户名枚举)。
- 角色仅 USER/ADMIN,register 始终 ROLE_USER,**无自助提权端点**;`AdminController` 全部 `requireAdmin`。
- 🔴 **CSRF 全局禁用**(`SecurityConfig.java:31`),仅靠 `SameSite=Lax` 补偿 → **对 session-cookie 应用是真实风险**(Lax 不防同站 CSRF)。`cookie.secure` 默认 false(需 `COOKIE_SECURE=true`)。

### 5.2 访问控制 / 越权(IDOR)— 2 处真实发现
已修复样板扎实:`MemoryServiceImpl.extractFromSession:83-127`(读消息前先 `verifyOwnership`)、`declineFriend/acceptFriend`(`SocialController:101-123` 内联校验)。但发现 **2 处同类漏修**(根因:鉴权 id 经 `@RequestBody` 传入,绕过 path-variable 的 `assertOwnsSession` 纪律):

| 级别 | 漏洞 | 机制 | 位置 |
|---|---|---|---|
| 🔴 **HIGH** | **`POST /api/aurora/goodbye` 跨用户会话接管+私信外泄** | `sessionId` 取自 body → `GoodbyeOrchestrator.start` 只判空**从不比较 `sess.userId`** → 篡改受害者会话 → `SessionCloser` 按受害者 sessionId 拉全部消息,摘要/LTM/画像写入**攻击者**账户 | `GoodbyeController.java:28-31` + `GoodbyeOrchestrator.java:42-47` |
