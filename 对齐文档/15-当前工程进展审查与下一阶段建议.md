## 4. 当前进行中工作审查边界

主工作树正在增加 ShedLock Redis provider，并给 `SessionIdleWatcher`、Aurora proactive、Capsule retry、慢信和夜间结算任务添加分布式 lease。方向正确，因为多副本下这些定时任务会重复执行。

验收时必须证明：

- 两个应用上下文/Pod 同时触发时只有一个获得 lease。
- lease owner 崩溃、Redis 短暂中断和 lock expiry 后能够恢复。
- 生产 Redis 不可用时 fail closed，不悄悄退回每 Pod 执行。
- lockAtMostFor 足以覆盖最坏执行时间，lockAtLeastFor 不阻塞合理重试。
- 长任务最好转为“scheduler 入队、worker 幂等处理”，ShedLock 只做过渡/轻调度。
- `academy-eks` 使用集群内 Redis，Redis 重启后的行为有明确测试。

未提交的连接指南包含实际环境标识并对 ALB/NLB、Fargate 和安全组权限有超出 Lab 证据的表述；在提交前应改为环境变量和发现命令，并引用 `14` 的能力矩阵。

## 5. 主要风险

### R1. 平台工作挤压产品创新

最近提交高度集中在基线、安全、PostgreSQL、OIDC、Redis。它们必要，但如果下一阶段继续扩充平台而没有 React Aurora 体验、真实 Provider 评测和双核切片，项目会成为“工程很扎实但 demo 普通”的反面。

### R2. Academy 环境被误当生产环境

无 StorageClass/EBS CSI、Pod 无 AWS 身份、4 小时临时凭据和固定区域意味着它只能证明部分 Kubernetes 能力。强行接 SQS/Secrets/RDS 会制造脆弱演示和凭据风险。

### R3. 测试数量掩盖真实 AI 质量

685 个自动测试主要证明行为回归与工程契约。日志仍大量显示 Mock/Fallback 和结构化输出修复；它们没有证明 Aurora 比通用聊天更好、共鸣体更像用户或心理 Skill 有效。

### R4. 文档状态漂移

work package evidence、总账和实时 HEAD 发展很快。任何 `PASS` 必须由当前 commit、环境和 reviewer 重新核验；旧阻塞摘要与新实现并存时应标明历史快照。

## 6. 下一阶段优先顺序

### P0. 关闭正在进行的多副本最低阻断

完成 SchedulerLease，保持 685+ 回归、production smoke 和 Redis failure contract；随后立即建立 `academy-eks` Kustomize overlay 和 capability preflight。

### P1. 建立两条可持续运行路径

- `local-complete`：Compose PostgreSQL/pgvector + Redis + 后端角色 + 真实 Provider 配置。
- `academy-eks`：API/Worker/Scheduler、静态 PV PostgreSQL、短状态 Redis、Envoy Gateway、探针/PDB/HPA。
- 同一镜像、Flyway 和 profile schema；SQS 通过 `EventDispatcherPort` 抽象，Academy 使用 JDBC outbox。

### P2. 立即转入 G3 + G4 纵向体验

首条纵向切片只做最核心的“Aurora 被打断后重新理解我”：React/TypeScript 对话页、typed SSE、真实多消息节奏、Stop/interrupt/replan、持久化 timeline、OTel trace、真实模型 pairwise 评测。

这条切片完成前，不继续引入 Kafka、Service Mesh、图数据库或更多托管服务。

### P3. Memory/Profile 纠正传播

完成 provenance claim、用户纠正、检索变化、星空变化和共鸣体失效/重编译的一条纵向链。这是 Aurora 与共鸣体质量共同依赖的数据壁垒。

### P4. Capsule Genome Compiler

用已建立的离线评测对比“长 Prompt baseline”与 Planner/Retriever/Critics/Reranker，先证明高保真与泄露边界，再扩展匹配策略和 simulator。

### P5. 课程展示收口

在同一脚本中切换本地完整体和 Academy EKS，展示产品差异 + Kubernetes 证据 + 对教育环境限制的正确解释。商业新加坡架构作为独立目标，不伪造当前部署。
