## 2. 冲突裁决

按以下次序处理冲突：

1. 安全事实、法律事实和当前代码行为必须以实时验证及有效证据为准，不能被愿景文本覆盖。
2. “最终要做成什么”以 L0、L1 和 L1-AI 为准。
3. 同层冲突时，范围更专门、日期更新、且明确写出替代关系的文档优先。
4. `08` 与旧文档在 Aurora 主动性、Self/Constitution/Emergence、动态共鸣体、匹配、慢信上的创新方向优先于旧的保守降级建议。
5. 当前实现与目标不一致时，应登记 gap 并迁移，不得反向修改目标来合理化旧实现。
6. 新发现若会改变核心产品承诺或技术边界，先写 ADR/RFC，并同步更新 L1 文档与验收清单，避免隐性漂移。

## 4. 专题文档阅读路线

- Aurora、主动性、连续自我、共鸣体、心理建模：先读 `08`，再读 `06` 的 AI 章节和对应 innovation work packages。
- Kubernetes、EKS、运行角色、Day-2：先读 `10`，再读 `02`、`06`。
- AWS Academy Learner Lab、SQS/存储/IAM 限制和课程演示：先读 `14` 与 `docs/goal/aws-academy-capabilities.yml`；`13` 只作为连接操作参考。
- App、PWA、Capacitor、推送与移动权限：先读 `11`，再读 `03`。
- 新加坡发行、PDPA、跨境 LLM、商店、HSA、税务和支付：先读 `04`，最终由 `12` 的发布门禁验收。
- 近期安全和基线工作：读 `07`、`evidence/m1` 及最新 Git/CI 结果。
- 自主研发流程：读 `12`，再用 `05` 获取更细的 Agent 协作和质量环设计。
- 当前实现优先级、是否允许推翻旧界面/架构、产品战役与测试频率：读 `16`；其执行裁决优先于旧的微切片或兼容性习惯。
- 单会话连续自主执行：读 `17` 和 `docs/goal/single-session-state.yml`；内部提交与检查点不得结束顶层 Goal。

## 5. 文档与实现的双向追溯

每个新能力应形成以下链路：

```text
产品承诺（09/11）
  → 架构责任与数据契约（10/08）
  → Goal/Work Package/ADR
  → 代码、迁移、IaC、测试与评测
  → evidence
  → complete-product-acceptance.yml 状态
```

禁止出现只有愿景没有验收、只有代码没有产品承诺、只有单测没有真实体验、只有 demo 没有安全与恢复证据的“伪完成”。

## 6. 文档维护规则

- L1 文档描述稳定目标和跨模块契约，不记录每日流水账。
- 具体执行拆到 `docs/work-packages`；关键取舍写 ADR；实验写 `docs/research`；结果写 `evidence`。
- 任何验收状态必须指向真实文件、命令输出或人工签字，不得仅写“已完成”。
- 当前态变化后更新机器清单和事实基线；愿景只有在产品决策确实变化时更新。
- 中文是主要设计语言；用户可见文案、API 错误和产品内容必须逐步形成中英双语资产。
