# Inner Cosmos 体验优先的完全体重构策略与产品战役

> 文档性质：L1-EXEC 产品实现战略；补充并具体化 `09`—`12`
> 状态：AUTHORITATIVE
> 生效日期：2026-07-15
> 核心原则：Think big while carefully——大胆重做最终体验，谨慎保护数据、安全与可验证性

## 0. 本文解决什么问题

此前的纵向切片帮助项目跨过了 Java 作业原型、单机状态和静态页面的阶段，但继续把每个表、接口或调度器当成独立目标，会出现三种偏差：

- Agent 不断完成容易验收的工程小项，却回避 AI 质量、动态交互和真实用户体验。
- 为了前向兼容旧原型，保留了没有产品价值的页面、API 和技术选择。
- 每个小提交都运行全项目重型回归，时间消耗与风险不成比例，反而减少真实体验和实验次数。

从本文生效起，长期 Goal Mode 的主要执行单元由“微型切片”升级为“产品战役”。切片仍可作为战役内部的实现手段，但不得再成为产品完成叙事。

## 1. 最高优先级：最终效果，而不是旧实现

### 1.1 必须保护的内容

- 用户数据的完整性、归属、授权、删除权、来源和可迁移性。
- 安全、隐私、危机响应、生产密钥和真实外部操作的人类门禁。
- 已被用户实际依赖且仍有产品价值的关键旅程；替换时必须有数据迁移和明确切换方案。
- 可复现证据、可观测性、回滚能力和对已知风险的诚实表述。
- `09` 定义的核心产品承诺：Aurora、长期自我理解、共鸣体与慢社交的完整飞轮。

### 1.2 默认不再保护的内容

- 物理 HTML 页面、旧导航、旧视觉、静态卡片和旧信息架构。
- 旧 API 形状、内部类结构、MyBatis 映射方式、数据库表形状和 Mock 演示流程。
- 旧的模式名称、表单步骤、固定轮次、机械额度、重复确认和妨碍情绪连续性的限制。
- 为 Java 作业展示设计的模式落点、接口/实现形式和技术栈叙事。
- 尚无真实用户和外部消费者依赖的内部向前兼容性。

Agent 可以删除、合并或替换这些实现。若会影响已有数据，必须给出一次性迁移、导出或重建策略；若只是内部代码和未发布界面，不需要为了形式上的 N/N-1 兼容保留双轨垃圾。

### 1.3 技术升级的判断方式

技术栈不是宗教，也不是自由换栈竞赛。保留或替换某项技术时，回答四个问题：

1. 它是否阻碍目标体验、AI 质量、开发速度或可靠运行？
2. 替代方案能否带来可观察的改善，而非只让架构图更现代？
3. 迁移是否能在一个产品战役内形成可用闭环，而非长期半成品？
4. 数据、安全和运维风险是否有明确控制？

满足前两项且后两项可控时，允许重构甚至重写。Spring Boot、模块化单体、PostgreSQL、React、Kubernetes 是当前优选基线，不是不可修改的终身契约；微服务、图数据库、流平台、原生重写也只有在真实瓶颈或体验收益出现后才采用。

## 2. 产品完成的判断顺序

每项工作按以下顺序判断价值：

1. **用户时刻**：用户具体看见、听见、操作和感受到什么？
2. **差异证明**：它为什么明显优于普通聊天、AI 日记或静态社交卡片？
3. **智能闭环**：哪些 AI、记忆、画像、Self、Skill 或策略真正参与决策？
4. **体验编排**：界面、动画、文案、等待、取消、错误和跨端细节是否共同服务这一时刻？
5. **工程成立**：数据、并发、恢复、权限、观测和部署是否使效果不只是一次性演示？
6. **证据成立**：真实 Provider、场景回放、pairwise、Playwright/设备体验和关键工程测试是否支持结论？

只完成第 5 项，不代表产品进步；只完成第 1—4 项而无法重复运行，也不代表可交付。

## 3. 五场产品战役

战役不是串行瀑布。Agent 可以并行推进无冲突部分，但任何时刻至少一半的有效投入应直接落到用户可感知的体验或 AI 效果上。

### Campaign A — Living Aurora：让 Aurora 真正“活起来”

#### 目标体验

用户进入的不是聊天框，而是一个有状态、有节奏、有时间感的关系空间。Aurora 可以思考、停顿、连续发送多条不同作用的消息；用户能自然打断、补充、让她停下或改变方向。她记得合适的事，承认不确定，能在承诺的时间回来，并说明这次主动联系的来由。其 Self、关系状态和表达会在长期互动中连续演化，而不是随机漂移或永远不变。

#### 必须完成的能力群

- 双核/多阶段 runtime：理解与规划核、表达与关系核、按需 critic/repair，不依赖单一超长 Prompt。
- Conversation Choreography：多气泡、停顿、typing/considering/listening 状态、打断、停止、重规划、断线恢复和跨端续接。
- Temporal Cognition：用户时区、自然语言时间、安静时段、承诺、WakeIntent、临近交付再决策、语义相关性、过期/被新对话替代、Push/站内/深链和疲劳反馈。
- Self/Constitution/Emergence：版本化 Self Genome、关系连续性、候选演化、回放评测、激活/回滚和可理解的用户侧表达。
- 上下文装配：当前话题、未解决问题、相关记忆、用户画像、关系轨迹、环境与工具结果按任务动态选择，不做生硬拼接。
- 真实模型质量环：至少一个高质量 Provider 配置，对单核/双核、普通 Prompt/完整 runtime 做盲测和失败分析。

#### UIUX 方向

- 允许彻底替换当前 React 原型和所有物理 HTML；Aurora 空间采用动态、响应式、状态驱动的界面，而非把后端实体做成管理面板。
- 动效表达“正在靠近问题、停下来听、形成想法、回来履约”等状态，可使用 Framer Motion、WebGL/Canvas、Rive/Lottie 或自研物理动画，但必须有语义、性能预算、reduced-motion 等价路径。
- “回来约定”应由对话自然生成和协商：为什么回来、何时合适、会继续什么、如何改期/取消；不能长期停留在固定“一小时后”按钮。
- 主动消息到达后直接恢复相关上下文，并让用户能反馈“正合适 / 晚一点 / 不再提醒这类事”，反馈进入策略学习。

#### 战役验收场景

连续三天的回放中，用户经历倾诉、打断、纠正记忆、约定回访、临时改变计划和 Aurora 主动续接。非实现者能在不看说明的情况下感受到 Aurora 的连续性和主动性；pairwise 显著偏好完整 runtime 而非普通聊天基线。任何 WakeIntent 只有“成功投递通知”而没有相关性、深链续接和反馈闭环，只能记为基础设施进度。

### Campaign B — Living Inner Cosmos：把理解变成可看见、可纠正的宇宙

#### 目标体验

每次对话不是消失在历史记录里，而是逐渐形成用户能理解的星空、轨迹和个人模型。系统能指出“我为什么这样理解你”，用户一句自然语言即可纠正；纠正会真实改变后续 Aurora、分析、星空和共鸣体，而不是只改一张展示卡。

#### 必须完成的能力群

- 原始证据、结构化 claim、置信度、有效时间、冲突、来源、用途和同意状态的统一模型。
- 情景记忆、语义记忆、关系/人物记忆、程序性偏好、未解决问题和自我叙事的分层生命周期。
- 混合检索与 rerank：全文、结构化条件、pgvector、多样性、时效、关系相关性和用户当前意图。
- Consolidate、更新、冲突协调、遗忘、归档、撤销、删除传播和质量监控。
- 高维用户画像：事实、偏好、价值、关系、情绪模式、习惯、表达风格、需求、边界、变化趋势和不确定性。
- Starfield V2：数据驱动的空间/时间/主题可视化，以及可访问列表视图；不是随机点阵。

#### 战役验收场景

用户纠正一个系统误解后，可以看见其来源、影响预览和版本变化；随后同一条纠正在 Aurora 检索、周变化、星空和共鸣体编译中传播。以标注集评测 claim precision、检索 relevance、冲突处理和删除完整性，并用真实界面证明“看懂自己”而非“看懂数据库”。

### Campaign C — Resonance Network：释放高保真共鸣体与慢社交

#### 目标体验

用户能把经授权的自己编译成一个可信、细腻、有边界的共鸣体，在沙盒中体验“像不像我”，再决定发布。陌生用户遇到的不是统一语气的 Prompt 机器人，而是具有事实、价值、节奏、习惯和时间一致性的动态人格；匹配既能寻找相似共鸣，也能寻找有意义的互补和阶段性同行者。

#### 必须完成的能力群

- Capsule Genome Compiler：授权快照、特征提取、风格样例选择、冲突处理、场景索引、Prompt/context program、critic、泄露评测、版本和撤回。
- Prompt 由编译器与运行时共同生成；信息按场景选择、压缩和组织，而非越多越好。
- 保真评测：事实、价值、语气、习惯、边界、时序、陌生场景稳定性和反提示攻击。
- 用户向量与匹配策略：相似、互补、需求—支持、阶段、探索、多样性和安全过滤；策略可解释、可切换、可实验。
- 宽松防滥用、不打断情绪高潮的会话政策；共鸣验证、慢信、双向同意、真人连接、退出和撤回完整闭环。
- 隔离的 Simulator 能力合同：只有明确授权的数据才能用于特定测试或研究，不默认复用个人共鸣体。

#### 战役验收场景

同一用户的普通长 Prompt baseline 与 Genome 版本做盲测，后者在保真和边界上同时胜出。用户发布、被发现、完成一次有质量的对话并进入慢信；之后撤回授权，所有公开运行和派生索引停止使用旧版本。

### Campaign D — Psychology Skills & Trust：把心理学变成可靠能力

#### 目标体验

Aurora 不只是“安慰得像”，还可以在合适时机调用有证据来源的反思、心理建模或量表 Skill。用户也可显式打开某项能力，知道它能做什么、依据什么、有什么局限，并能查看和撤回结果。能力帮助用户理解自己，但不冒充诊断、治疗或专业人员。

#### 必须完成的能力群

- 版本化 Skill/Plugin runtime：manifest、适用条件、输入最小化、工具权限、结构化输出、引用、审计、评测和禁用/回滚。
- 首批 Skill 选择少而深：例如情绪辨识与需求澄清、认知重评辅助、价值/目标澄清、经许可的标准化自评。
- 心理模型与画像分离“用户自述、模型推断、量表结果、行为证据”，明确置信度和不可推断领域。
- 风险分层、危机资源、拒绝边界和新加坡本地化资源；安全措辞不应在普通对话中机械打断体验。
- 由领域专家审阅内容与高风险场景；未经审阅只能标为研究/原型能力。

#### 战役验收场景

同一个 Skill 可由 Aurora 按适用条件建议，也可由用户显式调用；结果带来源、解释、置信和下一步选择，并能进入或不进入画像。错误、不适用、用户拒绝和高风险场景均有自然且安全的体验。

### Campaign E — Complete Product & Cloud Showcase：跨端完整体与云原生证明

#### 目标体验

用户可以在 Web/PWA 和移动容器中完成同一个连续旅程；通知、深链、语音、离线草稿和安全存储在移动端形成增量价值。课程展示既让观众先看到令人眼前一亮的产品，也能进一步证明多副本、调度、恢复、观测和 Kubernetes 设计真实成立。

#### 必须完成的能力群

- 五空间统一 AppShell、设计系统、响应式/触控、i18n、a11y、性能与真实设备打磨。
- Capacitor 应用与通知、深链、麦克风、安全存储、权限和商店候选资产。
- `local-complete` 展示完整 AI/数据/动态 UX；`academy-eks` 展示受限环境下真实 Kubernetes 语义；`commercial-sg` 保持新加坡生产蓝图和发布门禁。
- API/Worker/Scheduler 多角色、迁移、outbox、Redis 协调、观测、备份恢复、故障演练和供应链证据。
- 三条英雄链路：Aurora SSE Pod 故障无损恢复、outbox 业务积压驱动 KEDA worker 弹性、OTel 跨 AI/异步/主动回访语义 trace。
- 在英雄链路稳定后主动扩展云原生能力展柜：Argo Rollouts/Argo CD、Kyverno/Cosign、Cilium/Hubble、cert-manager/External Secrets、Falco、Chaos、Velero、OpenCost、OpenFeature、SPIFFE/mesh、Indexed Jobs，以及有独立产品收益的 Dapr/Knative/Crossplane/GraphQL BFF。复杂度可以接受，但每项必须真运行、可失败验证、可关闭并有 Academy fallback。
- 8—12 分钟叙事：先展示差异化用户时刻，再逐层揭示 AI runtime、数据飞轮和云原生支撑。

#### 战役验收场景

同一 commit 在 local-complete 和 Academy EKS 上运行相同核心领域语义；本地展示完整体验，EKS 展示扩缩容、Pod 重建、任务不重复、日志/指标/trace 和受限服务适配。不得为了 EKS 截图牺牲本地完整产品，也不得用本地运行冒充 EKS 证据。

## 4. UIUX 的新授权范围

### 4.1 可以采用的先进范式

- 状态驱动的界面编排：同一空间根据 listening、thinking、speaking、interrupted、remembering、returning 等事件改变布局、动效和可操作性。
- Progressive disclosure：默认保持情绪沉浸，用户需要时再展开记忆来源、画像证据、策略解释和权限。
- Spatial/temporal interface：星空、轨迹、回声、涟漪和关系距离来自真实数据映射，不是装饰性随机动画。
- Optimistic/local-first interaction：草稿、打断、反馈和视觉状态立即响应，后台以持久化事件校准；断线后诚实恢复。
- Generative but governed copy：微文案可由策略生成，但核心权限、风险、删除和支付文案使用版本化资产。

### 4.2 每个细节都要被设计

关键旅程必须覆盖：首次进入、空状态、输入中、等待、长响应、多气泡、打断、停止、恢复、错误、Provider 降级、离线、权限请求、拒绝、撤销、删除、长文本、中英文、窄屏、键盘、屏幕阅读器和 reduced motion。微交互不是最后统一“美化”，而是战役定义的一部分。

### 4.3 动态不等于炫技

每个动画必须回答它在表达什么状态、是否帮助理解、是否阻碍输入、低端设备如何降级。核心对话的首要性能指标是可输入、可停止、可阅读和状态可信；视觉表现服务于情感质感，而不是抢夺注意力。

## 5. Agent 的执行方式

### 5.1 每场战役先建立 Experience Contract

在开始大量编码前，Agent 需在战役资产中写明：

- 3—5 个关键用户场景和可观察结果。
- 一段从进入到离开的 experience film，包括消息、时间、界面变化和失败分支。
- baseline、期望差异和质量指标。
- 需要的 AI/数据/前端/后端/移动/云能力图。
- 可以删除或替换的旧资产，以及必须迁移的数据。
- 关键风险、人工门禁和非目标。

这不是长篇空文档；它是 Agent 在数周自主实现中做取舍的裁决器。

### 5.2 战役内部自主拆解

Agent 可自行拆成实现批次、使用 worktree 或子 Agent、做 PoC、选择库和调整顺序。默认以“可体验的完整场景”收敛，而不是以“类和测试数量”收敛。遇到实现困难时可更换内部方案，但不得悄悄缩减 Experience Contract。

### 5.3 体验债务与工程债务共同记账

每场战役维护一个简短 backlog，按以下顺序处理：

1. 阻断核心旅程或造成数据/安全错误。
2. 伤害信任、人格连续性或情绪节奏。
3. 让产品与普通聊天/静态页面没有差异。
4. 高频摩擦、跨端问题和明显视觉不一致。
5. 低频美化与内部洁癖。

不能因为某项更容易写单测，就跳过排在前面的真实体验问题。

## 6. 测试与验证节奏

减少重复全量测试不等于降低质量。采用风险分层：

### 6.1 每个实现批次

- 编译/类型检查。
- 被修改模块的 focused tests。
- 至少一个相关用户场景的快速 smoke；UI 改动做目标页面交互或截图检查。
- 数据、安全、并发、调度或迁移改动增加对应负向/故障测试。

### 6.2 战役检查点

以下时机运行跨模块集成和选定 E2E：一条 experience film 首次贯通、跨域契约改变、合并多个实现批次、准备组员体验。AI 变更同时运行固定小型 eval/regression set，不能只看单元测试。

### 6.3 全量质量门

只有以下节点默认要求 Maven 全量、前端全量、Playwright 核心旅程、扫描和部署验证：

- 战役候选完成并准备验收。
- 进入共享主干/发布候选前。
- 数据迁移、身份权限、全局 runtime、依赖基线或部署架构等高爆炸半径变更。
- 距离上一次全量门已经积累多个相互依赖的实现批次。

纯文档、局部样式、孤立算法实验不重复运行 700+ 全量回归。测试资产继续保留，失败必须修复；只是把运行频率与风险匹配。

## 7. 当前优先级（自 2026-07-15 起）

## 8. 对当前 WakeIntent 切片的产品解释

`INNO-TEMP-001` 建立了耐久 intent、所有权、租约竞争和站内通知的基础，属于 Campaign A 的必要工程底座。但它尚不等于“Aurora 在正确时间回来”：固定一小时按钮、静态文案、单一站内通知和硬编码自主时区不能证明时间认知、相关性、自然协商或关系连续性。

因此：

- 允许将该提交验收为首条耐久基础切片。
- `AURORA-TEMPORAL` 必须保持 `IN_PROGRESS`。
- 后续不再为每个调度子能力单独开启碎片化 Goal；由 Campaign A 统一推进并在战役检查点验收。

## 9. 给 Goal Mode Agent 的最终裁决规则

当两个方向冲突时，依次选择：

1. 不破坏安全、数据权利和不可逆人类门禁的方向。
2. 更能实现关键用户时刻和差异化体验的方向。
3. 能在当前战役内形成真实闭环、可被体验和评测的方向。
4. 更简单、可观测、可恢复的内部实现。
5. 最后才考虑旧代码兼容、模式数量、类结构整齐或测试数量增长。

Agent 的目标不是把每份旧设计实现一遍，而是依据权威目标持续取舍、重构、实验和打磨，直到五场战役共同构成一个用户愿意使用、组员愿意展示、工程上能够真实运行的 Inner Cosmos。
