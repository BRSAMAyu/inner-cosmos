# Inner Cosmos 完全体 UIUX 与交互设计规格

> 文档性质：L1 体验与界面目标规范
> 状态：AUTHORITATIVE
> 设计母题：流动（Flow）
> 目标：让用户真实感到“被一个持续存在的对象理解，并能看见自己的内在世界”，而不是浏览一组 AI 功能页

## 0. 体验愿景

Inner Cosmos 的体验应同时具有三种质感：

1. **亲密而不黏腻**：Aurora 像真正理解节奏的朋友，不像客服、疗愈文案生成器或强行卖萌角色。
2. **深邃而不晦暗**：星空、时间和内在世界带来想象力，但信息结构清晰、行动可发现。
3. **智能而不炫耀机器**：用户看见系统在倾听、组织、等待和回访，却不被技术术语、假思维链或仪表盘淹没。

旧 UIUX 资产中的暖色、流动、呼吸、星尘、涟漪、半透明、昼夜与环境感继续作为设计基因；旧物理页面和不一致组件不是目标实现。

## 1. 设计宪法

### 1.1 五个视觉与动效母题

- **Flow**：曲线、路径和连续转场表达思绪与关系的流动。
- **Breath**：低频、克制的明暗和尺度变化表达系统正在陪伴，而非吸引注意力。
- **Stardust**：记忆、主题与偶遇的粒子语义；粒子必须承载信息，不只装饰。
- **Ripple**：用户输入、洞察形成、关系触达后的扩散反馈。
- **Translucence**：层次和边界的半透明表达；不能牺牲文字对比度和性能。

### 1.2 禁止的视觉方向

- 冷蓝霓虹、赛博朋克、纯黑高对比“AI 控制台”。
- 全站粉紫渐变、廉价玻璃拟态和无意义大光晕。
- 大量通用卡片网格、功能宫格、模板 SaaS dashboard。
- 以 emoji/机器人头像替代精细状态和角色表现。
- 所有页面都用星空背景，导致内容与核心星空场景失去层级。

## 2. 适应时间的主题系统

产品不是固定浅色或深色。它根据当地时间给出默认氛围，同时允许用户锁定主题并遵循系统 reduced motion/contrast 设置。

### 2.1 核心色板

| Token | 日间/晨昏 | 夜间 | 语义 |
|---|---|---|---|
| `surface-canvas` | `#F7F2EC` | `#211A18` | 主背景，暖纸/深暖褐 |
| `surface-raised` | `#FFFDFC` | `#2A2220` | 对话和内容层 |
| `surface-glass` | rgba warm white | rgba warm charcoal | 浮层，需 backdrop fallback |
| `accent-aurora` | `#B69A74` | `#C79A68` | Aurora、烛光、关键动作 |
| `accent-sage` | `#AAB8A6` | `#7E9281` | 平静、恢复、确认 |
| `accent-sky` | `#A5B3C2` | `#74869B` | 反思、时间、信息 |
| `accent-plum` | `#B59CA9` | `#8D7482` | 关系、共鸣、夜间情绪 |
| `text-primary` | `#302A26` | `#F0E8DF` | 正文 |
| `text-muted` | `#756C64` | `#B7AAA0` | 次要信息 |
| `danger` | 柔和但可辨识红 | 提升亮度的暖红 | 高风险/破坏动作 |

具体色值进入 design token 后需以 WCAG 对比度和真实设备校准；上表是设计方向，不允许随意换成冷科技主题。

### 2.2 七个时间段

黎明、早晨、正午、午后、黄昏、夜晚、深夜可改变背景温度、Aurora 光晕、星体可见度和环境音建议。变化必须缓慢、不中断操作；时间感也体现在文案与主动性，不只是换壁纸。

天气或位置仅在用户明确授权并确有价值时作为环境层，不能成为核心逻辑依赖。无位置权限时使用时区和系统主题即可完整体验。

## 3. 字体、图形与声音

- 中文正文优先清晰的人文宋/黑体组合；品牌标题可使用霞鹜文楷或经打包许可确认的同类字体。
- 英文展示可使用 Cormorant Garamond，正文使用高可读 sans-serif；代码/系统标识使用 JetBrains Mono。
- 字体必须本地化托管或符合隐私/性能要求，定义中英 fallback，避免布局跳动。
- 图标采用一套线性/柔和填充系统，重点状态不只靠颜色。
- 音频是可选环境层：默认不开启自动播放，有独立音量、静音、字幕/视觉替代和节电策略。
- Aurora 不应始终用固定人物脸；优先使用可呼吸的光、轨迹和抽象存在感，未来角色形象需单独用户研究。

## 4. 信息架构与导航

### 4.1 五个用户空间

| 空间 | 用户问题 | 核心内容 |
|---|---|---|
| Today / Aurora | “现在谁在这里陪我？” | 对话、主动消息、当天状态、继续事项 |
| Inner Cosmos | “我最近发生了什么变化？” | 星空、时间线、画像、洞察、Skill |
| Resonance | “谁可能真正理解我？” | 共鸣体、发现、匹配、连接 |
| Inbox | “有什么在等待我回应？” | 慢信、请求、通知、系统解释 |
| Me | “系统如何理解和使用我的数据？” | 契约、隐私、设备、导出删除、语言外观 |

桌面用克制的侧栏或顶部+侧栏组合；移动端用不超过五项的底部导航。对话时导航可弱化，但退出、停止、隐私和安全入口始终可达。

### 4.2 路由与连续性

- 页面切换保留未发送草稿、当前滚动和流式会话。
- 深链可进入一封信、一个星体、共鸣邀请或隐私动作，认证后返回原目标。
- 所有异步任务有统一 Activity/Task 状态，不让用户猜“处理中是否丢了”。
- 不再维护多套重复 Header、toast、modal、表单和请求逻辑。

## 5. Aurora 对话体验

### 5.1 主画面

对话画面以内容和存在感为中心，减少传统气泡墙的社交软件感：

- 顶部只显示 Aurora 当前关系/时间状态和必要控制。
- 中部消息在窄阅读宽度内自然排布；多条 Aurora 消息形成有节奏的组，而非每句独立重头像。
- 底部 composer 支持文字、按住/点击语音、附件（若启用）、当前意图和停止控制。
- 桌面可有轻量右侧 Context Veil，显示本次使用的记忆主题、待办或 Skill，但默认不抢焦点。

### 5.2 状态语言

禁止用通用 spinner 代表所有智能过程。采用类型化状态：

| 系统事件 | 用户感知 | 可操作性 |
|---|---|---|
| listening | 输入被接收、Aurora 在听 | 继续补充/发送 |
| reflecting | Aurora 在梳理较复杂内容 | 可打断、可要求简短回答 |
| composing | 第一条消息即将到达 | 停止生成 |
| between_messages | 她停一下再继续 | “继续”“等一下”或直接发新消息 |
| revising | 新消息导致计划改变 | 清楚显示已听见新内容 |
| settling | 会话正在沉淀记忆 | 可离开，后台继续 |
| scheduled_followup | Aurora 打算稍后回来 | 查看原因、改时间、取消 |
| recoverable_error | 网络/Provider 暂时失败 | 重试且不重复用户消息 |

这些状态只能表达可公开的工作阶段，不展示模型隐藏推理链。

### 5.3 打断、停止与多消息

- 用户在 Aurora 思考或输出期间发消息，立即取消旧输出的后续计划；已发出的内容保留，明确进入“重新听你说”。
- Stop 是一等操作：停止 token、未发送消息和延迟 follow-up，并让用户决定是否保留已生成部分。
- 多消息不是把段落硬拆开：每条承担独立社交动作，例如承接、澄清、补充或轻问句；由 Choreographer 决定节奏。
- 弱网重连后按 event sequence 恢复，不把同一条消息重复显示或落库。

### 5.4 用户意图与可控性

不要求用户学习大量“人格模式”。输入附近提供低认知负担的意图：陪我待一会、帮我理清、一起想办法、只听不建议、睡前慢一点。Aurora 仍保持同一个连续人格，意图只调节当前协作方式。

长期关系契约在 Me 中可视化：主动程度、直接程度、幽默、挑战/支持比例、安静时间、可触及主题和不希望出现的表达。

## 6. 主动消息体验

- 主动消息在视觉上说明“为什么现在出现”，如“你昨天希望我今晚问问结果”，而不是神秘地装作随机想起。
- 用户可从消息本身选择：有帮助、时机不对、别再问这个、今天安静、调整频率。
- 推送只显示隐私安全预览；锁屏默认不暴露敏感主题。
- 点击进入时恢复完整上下文，Aurora 知道这是一次 follow-up，不重新问背景。
- 被延迟或取消的主动意图在内部可观测；用户只看有意义的“计划”，不看工程队列。

## 7. Inner Cosmos 与记忆星空

### 7.1 首页层次

1. 当前情绪天气与一句非武断解释。
2. 可交互星空：近期星体、主要星座、未解决引力、人物轨道。
3. “本周变化与未解决问题”。
4. 可选入口：时间旅行、画像、Skill、全部记忆。

### 7.2 星空视觉编码

| 视觉属性 | 数据语义 |
|---|---|
| 尺寸 | 综合情感重力/长期重要性，不等同消息长度 |
| 亮度 | 当前活跃/复现程度 |
| 色相 | 主题/情绪族，提供图例且不只靠颜色 |
| 距离 | 时间、关系或语义投影，当前模式明确标注 |
| 轨道/连线 | 可解释的主题、人物或因果关联 |
| 脉动 | 新变化、冲突或待确认，不能无限动画 |
| 雾化/淡出 | 低置信、衰减或即将归档 |

用户切换时间/主题/人物模式时，空间平滑重排并保留焦点。提供列表、键盘和屏幕阅读器等价视图；reduced motion 下用渐变/静态变化。

### 7.3 星体详情

详情采用渐进披露：摘要 → 为什么重要 → 来源与时间 → 与其他记忆的关系 → 系统推断 → 修正/私密/删除。任何画像结论都能跳回来源，任何来源变更都显示可能影响哪些派生资产。

### 7.4 用户画像

不做“人格测试结果墙”。按“我确认的 / 系统正在理解的 / 有冲突的 / 已改变的”组织。每个条目显示：内容、置信度、最后证据、变化趋势、被哪些功能使用。用户可写自然语言纠正，系统生成结构化变更预览后再提交。

## 8. Psychology Skill Studio

- Skill 首页以用户问题组织，如“我为什么总在这件事上卡住”，不是插件技术目录。
- 进入前显示来源、预计时长、是否保存、是否进入画像、风险和适用边界。
- Skill 可使用对话、分步反思、量表、二维图或时间回顾，但共享统一 shell 和无障碍控件。
- 结果避免绝对标签，采用“你提供的信息更支持… / 也可能是… / 可以如何验证”。
- 结果页有三个出口：与 Aurora 继续、保存到 Inner Cosmos、做一个小行动；高风险时增加真人资源。
- 用户与 Aurora 都能调用 Skill，但 Aurora 只能建议，不能在未同意时暗中完成敏感心理评估。

## 9. 共鸣体创建与沙盒

### 9.1 创建向导

步骤不按数据库字段，而按用户心智：

1. 想让共鸣体表达“哪一部分的我”。
2. 选择用途与受众。
3. 选择可用信息：经历、价值、兴趣、风格、习惯、情绪/关系等。
4. 设置不可谈、不可推断、不可泄露与慢信边界。
5. 编译与覆盖报告。
6. 沙盒试聊、修正、发布。

每个授权项可预览例子，并清楚区分“公开摘要”“对话时按需使用”“绝不使用”。

### 9.2 编译反馈

Compiler 过程用有意义的阶段和可后台化任务表达：整理你选择的片段、理解表达节奏、检查事实冲突、测试边界。无需把模型链路术语暴露给普通用户。完成后展示覆盖雷达、可能不像的部分、高风险信息和版本差异。

### 9.3 沙盒与质量反馈

- 提供系统建议的典型问题、用户自定义问题和边界攻击问题。
- 每次回应可标“像我/不像我/事实不对/太暴露/语气不对”，反馈更新 Genome 提案而非直接在线漂移。
- 可比较当前版与新版本，但不能以真实用户身份把沙盒内容发送给其他人。

## 10. Resonance、匹配与慢信

### 10.1 发现页

- 首先选择或混合 Mirror、Complement、Growth Edge、Serendipity、Contextual 策略。
- 共鸣体不是漂亮头像卡堆叠；卡片以一句真实表达、共鸣维度、边界和为何相遇为核心。
- 不显示具有社会比较伤害的公开热度/关注者排行。
- 筛选可控制语言、时区、互动节奏、主题和安全边界。

### 10.2 共鸣体会话

- 固定但克制地表明这是授权 AI 共鸣体，不伪装真人正在输入。
- 聊天保持高质量自然节奏，陌生阶段防刷额度采用软提示和冷却，而非在情绪高潮硬锁死。
- 用户随时查看其可用信息边界、举报输出或停止使用自己的数据。
- 达到条件时邀请发慢信，但不强迫把每次对话转社交。

### 10.3 慢信

- Compose 强调意图、节奏和边界，提供 Aurora 辅助但不代写成千篇一律。
- 飞行状态用视差/轨迹表达等待，不制造不可取消的焦虑；允许发送方在规则允许时撤回。
- 接收者先看到安全预览和为什么收到，能接受、延后、拒绝、屏蔽或举报。
- 双方明确同意后才逐步开放真人连接；慢信不是泄露个人联系方式的后门。

## 11. 首次使用、信任与隐私中心

### 11.1 渐进式 onboarding

- 第一屏说明真实价值，不使用“治愈一切”或“永远陪你”。
- 只收集开始体验必需信息；心理、位置、通知、联系人等权限在价值出现时再问。
- 通过第一段对话完成关系偏好和记忆授权，不先填几十项人格表。
- 第一次落记忆时出现清楚的 before/after 预览。

### 11.2 隐私中心

用户可一处完成：

- 查看 P0—P3 和每项数据的用途。
- 管理 Provider/跨境处理同意与功能影响。
- 查看/撤销共鸣体授权、主动消息和 Psychology Skill 数据。
- 下载、删除账户或删除某个主题/时间段的数据。
- 管理设备、会话、通知预览和锁屏隐私。
- 查看最近的安全相关访问和派生重算状态。

破坏性动作使用清楚后果、可恢复窗口（若合法适用）与最终确认，不使用暗黑模式阻止离开。

## 12. 响应式与移动端

### 12.1 断点不是简单缩放

- Small phone：单列、底部导航、全屏 sheet、触控优先。
- Large phone/fold：对话与轻量上下文可分区。
- Tablet：双栏，星空与详情并行。
- Desktop：稳定侧栏 + 主内容 + 可选 context veil，控制最大阅读宽度。

### 12.2 移动原生增量

- Push：主动消息、慢信与任务完成，支持类别与安全预览。
- 麦克风：明确录音态、波形、取消、权限失败和后台中断。
- 安全存储：token/设备凭据，不把敏感明文放 Web storage。
- 深链：通知到具体上下文，认证后恢复。
- App lifecycle：后台/前台、网络切换、SSE 恢复、草稿保护。
- Haptic：仅用于发送、停止、抵达等少量事件，可关闭。

PWA 必须独立好用；Capacitor 不是把不适配页面包进 WebView。

## 13. 无障碍、包容与国际化

- 目标 WCAG 2.2 AA；关键流程支持键盘、屏幕阅读器、200% 缩放和高对比。
- 动画、星空和音频均有 reduced motion/sound 等价方案。
- 点击目标至少满足移动可用尺寸；错误不只靠红色。
- 中英文案使用语义 key，不拼接句子；布局承受英文扩张和中文断行。
- 时间、数字、时区和危机资源按 locale；新加坡默认英文可完整使用，中文保持一等质量。
- 情绪和心理文案避免性别、文化和诊断偏见；量表适用性需按语言版本验证。

## 14. 设计系统与实现合同

### 14.1 Design tokens

统一维护 color、type、spacing、radius、elevation、blur、motion、z-index、breakpoint、sound/haptic tokens。产品代码不得散落任意色值和动画时长；数据可视化 token 与普通 UI token 建立映射。

### 14.2 核心组件族

- AppShell、AdaptiveNavigation、ContextVeil。
- AuroraPresence、ConversationStream、MessageGroup、ThinkingState、Composer、StopControl。
- StarfieldCanvas、StarLegend、MemoryOrbit、InsightDrawer、ProvenanceTrail。
- ConsentScopePicker、ProfileClaim、Confidence/ChangeIndicator、DataImpactPreview。
- SkillCard/Runner/Result。
- CapsuleCard、GenomeCoverage、SandboxChat、BoundaryEditor、MatchReason。
- SlowLetterComposer、FlightPath、DeliveryState。
- AsyncTask、OfflineBanner、PermissionPrompt、SafeError、CrisisResourcePanel。

每个组件在 Storybook 中覆盖默认、空、加载、失败、弱网、无权限、长文本、中英、键盘、reduced motion 和高对比状态。

### 14.3 动效预算

- 交互反馈 120—220ms；页面过渡 240—420ms；呼吸和环境动效 4—12s 且低振幅。
- 输入响应不得被环境动画阻塞；星空在中端移动设备保持流畅并按性能降级。
- 不用无限 shimmer 掩盖未知等待；超过阈值说明正在做什么并允许后台化/取消。

## 15. 前端迁移策略

1. 盘点现有 30+ 物理 HTML 和重复 JS/CSS，建立路由/功能/后端契约矩阵。
2. 先建 tokens、组件系统、认证、类型化 API、错误/任务/遥测和五空间 AppShell。
3. 以纵向旅程迁移：首次体验 → Aurora → Inner Cosmos → 共鸣体/慢信 → 设置与后台。
4. 每迁移一条旅程，旧入口重定向或删除，避免长期双 UI。
5. 不机械复刻旧页面；按本规范重构信息架构，同时保留已验证功能。
6. 关键阶段用真实设备和不同网络体验，视觉回归不能代替人类体验评审。

## 16. 体验验收

一项界面能力只有同时满足以下条件才算完成：

- Happy path、错误、空、等待、权限、取消、离线/重连均有设计与测试。
- 中英双语、移动/桌面、键盘/屏幕阅读器和 reduced motion 通过。
- 操作结果在数据层真实成立，刷新、重登和另一设备后仍一致。
- 用户理解 AI 状态、数据用途和下一步，不依赖开发者讲解。
- 关键路径经至少一轮组员盲体验；问题进入可追踪清单并完成复验。
- Lighthouse/性能预算、自动 a11y、视觉回归和 Playwright 核心旅程通过。

最终的“优秀”不由组件数量决定，而由用户是否感到流畅、真实、可控、愿意继续探索，并能准确说出 Inner Cosmos 与普通 AI 聊天的差异决定。
