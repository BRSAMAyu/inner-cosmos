# Inner Cosmos 完全体实现路线与最终验收规约

> 文档性质：L1 自主实现与最终验收规范
> 状态：AUTHORITATIVE
> 执行者：Goal Mode Codex/Agent 团队
> 机器账本：`docs/goal/complete-product-acceptance.yml`

## 0. 这不是固定瀑布计划

G0—G9 是结果门，不是要求按编号做完所有内部任务才进入下一编号。安全基线、平台迁移、AI 创新、前端体验和研究评测应在依赖允许时并行，以纵向闭环持续集成。

Agent 自己决定具体工作包、先后、并发、PoC 和修复方式；本文件只固定：最终必须交付的结果、不可破坏的依赖、质量门和真实人类权限边界。

## 1. 自主研发操作系统

### 1.1 每次接管循环

```text
读取目标与权威层级
 → 检查分支/工作树/CI/证据/阻塞门禁
 → 选择当前最高价值且无阻塞的纵向差距
 → 建立或更新 work package 与验收映射
 → 实施 + 测试 + 体验 + 安全复核
 → 产生可复现 evidence
 → 更新机器账本和决策记录
 → 提交小而完整的变更
 → 继续下一差距，直到所有目标门关闭
```

### 1.2 自主权限

Agent 可以自主：

- 阅读、重构和实现仓库代码、测试、迁移、IaC、脚本和文档。
- 建分支/工作树、拆工作包、使用子 Agent 做独立复核、运行本地与授权 CI。
- 选择同一目标栈内的库和补丁版本，写 ADR，完成 PoC 后取舍。
- 创建 mock/fake、测试数据、离线评测与本地 Kubernetes/EKS staging 资产。
- 在不涉及真实用户和不可逆外部操作时持续推进，不等待微观确认。

需要人类的事项见 `goal-objective.md`。Agent 应把人类门禁做成一页可执行 checklist，只请求必要签字；等待期间继续其他路径。

### 1.3 三种完成证据

任何重要能力至少同时有：

1. **实现证据**：代码、迁移、配置、API/事件 schema。
2. **机器证据**：测试、评测、扫描、运行日志、截图/trace、数据核对。
3. **体验/责任证据**：人工盲测、设计验收、安全/隐私审查或有权限签字。

只有文档、只有单测或只有演示视频都不是完整证据。

## 2. 全局工程纪律

- 主干始终可构建；每个提交只包含自己负责的变更，保留无关工作树内容。
- 新能力先定义 observable outcome 和失败模式，再决定实现。
- 所有数据/schema/API/event/AI state 变更版本化并可迁移。
- 所有异步和外部 Provider 路径必须测试超时、重复、乱序、取消、重启和降级。
- 所有用户域资源在读取内容或调用 LLM 前完成 owner/purpose/consent 校验。
- 新 Prompt、模型、Self、Compiler、Skill 或匹配策略有版本、离线评测和回滚。
- 新页面覆盖 loading/empty/error/offline/permission/cancel/long content/i18n/a11y。
- 不以降低创新能力作为默认修复；用架构、契约、观察和测试解决风险。
- 若证据推翻某一设计，写 ADR/RFC 并同步产品、架构、UIUX 和验收账本。

## 3. G0—G9 结果门

### G0 — Governance & Security Foundation

**结果**：仓库不再携带可用凭据，生产启动 fail-fast，已知 P0/IDOR 越权修复，研发/评测/部署有责任边界。

必须具备：

- 当前树 secret scan 0 finding；历史泄露清单不暴露值。
- GLM、MiniMax、MiMo/ASR 等真实旧密钥由授权人员吊销、轮换并独立签字。
- Goodbye、Aurora Self 及全仓用户资源所有权负向测试。
- 生产禁 Mock/fallback/demo seed/弱 Cookie/弱 TLS；Secret Manager 注入。
- acceptance ledger、创新语义 ADR、数据/AI 责任角色和人类门禁。

当前已存在大量实现与证据，但外部密钥轮换签字仍是硬门禁；必须按实时 evidence 复核，不能仅抄历史结论。

### G1 — Modern Platform Baseline

**结果**：Java/Spring 基线、供应链、容器和 CI 可作为后续产品化稳定地基。

- Java 21 编译/运行；Spring Boot 3.5.x 受支持补丁。
- 全量测试、依赖差异、SBOM、SCA/SAST/secret/container/IaC scan。
- 可复现、非 root、最小化、按 digest 的 OCI 镜像与启动 smoke。
- CI 缓存和并发合理，失败门禁明确；发布物签名/provenance。
- 本地开发一条命令启动必要依赖，README 与真实行为一致。

依赖：外部密钥轮换完成后才能越过既定 Boot 基线门；不依赖 G1 的产品研究和非基线代码可继续。

### G2 — Contracts, Data & Runtime Roles

**结果**：系统从单机单副本原型变为可多副本运行的数据与任务底座。

- Spring Modulith/ArchUnit 约束领域边界。
- Spring Security + OIDC/OAuth2.1 路径；Web/移动会话安全。
- OpenAPI v1、统一错误、幂等、并发版本和 SSE 恢复契约。
- PostgreSQL + Flyway + Testcontainers；MySQL/H2 数据迁移与 reconciliation。
- Redis 外置 session/短状态/限流/租约。
- Transactional outbox + SQS/DLQ + 幂等 consumer。
- API/AI worker/event worker/scheduler/migration job 独立运行和伸缩。
- provenance、consent、删除传播和多向量基础 schema。

### G3 — Unified Product Shell

**结果**：React/TypeScript PWA 成为统一产品，不再由孤立物理页面定义体验。

- design tokens、Storybook、五空间 AppShell、类型化 API client。
- 认证、任务、错误、offline、telemetry、i18n、a11y 基础设施。
- 首次使用和 Aurora 基础旅程迁移，旧入口无重复实现。
- PWA 安装、离线 shell/草稿、弱网 SSE 恢复。
- Playwright 核心旅程、视觉回归、axe/Lighthouse 预算。

G3 可以与 G2 并行，通过 mock contract server 开发，但合并门以真实 API 契约为准。

### G4 — Aurora Living Mind

**结果**：Aurora 的“生命感”由可维护系统能力支撑，并在体验评测中显著优于单轮基线。

- 双核、Context Assembly、Conversation Choreographer 与类型化 stream events。
- 多消息、停顿、打断、停止、重规划、断线恢复真实工作。
- Temporal Cognition、WakeIntent、quiet hours、主动预算与用户反馈闭环。
- Self Genome、Constitution、Relationship State、Emergence Proposal 的版本/评测/激活/回滚。
- 事实、隐私、安全和依赖风险 critics；Provider gateway 支持取消、路由和诚实降级。
- 固定纵向场景和盲测证明连续性、共情/帮助性、主动恰当性和打断恢复提升。

不得用 UI 假动画模拟不存在的系统状态，不得用 Prompt 中的角色描述冒充 Self/Emergence 已实现。

### G5 — Memory, Profile & Inner Cosmos

**结果**：数据管线能准确获取、更新、淘汰和复用长期信息，用户能美观且可解释地看见变化。

- 记忆生命周期、冲突/合并/衰减/遗忘与用户纠正。
- 高维画像 schema、多向量表示、来源 DAG、置信度、时间和用途。
- Retrieval 按任务、权限、相关性与 token budget 选择，不以“更多上下文”为目标。
- 本周变化/未解决问题、主题/人物/时间轨迹与影响预览。
- Starfield V2 的数据投影、WebGL/Canvas 呈现、列表等价视图和性能降级。
- memory precision、contradiction、retrieval relevance、correction propagation 评测达门槛。

### G6 — Capsule Intelligence & Resonance

**结果**：共鸣体从保守静态卡片释放为高保真、可授权、可评测的动态人格系统，并产生有质量的连接。

- Capsule Genome Compiler、授权快照、覆盖报告、版本和撤回传播。
- Planner/Retriever/Speaker/Critics/Reranker 运行时与沙盒反馈。
- 事实、风格、习惯、癖好、边界和长期一致性评测；泄露攻击集。
- Mirror/Complement/Growth Edge/Serendipity/Contextual 多策略匹配。
- 多空间候选、结构化约束、重排、解释、多样性和反馈学习。
- 宽松防滥用而非伤害体验的会话策略；慢信与双向真人连接闭环。
- 可隔离、经授权的 simulator 能力演示，不复用个人数据做隐性训练。

### G7 — Psychology Skills, Mobile & Experience Completion

**结果**：心理学能力、原生增量与完整 UIUX 共同形成可真实体验的产品。

- Skill manifest/runtime/registry/sandbox/评测/风险分级和至少三类高质量 Skill。
- 明确非诊断边界、来源、多语言、结果解释和真人转介路径；专家门禁有记录。
- Capacitor Push、麦克风、深链、安全存储、lifecycle/网络恢复。
- 五空间全部核心旅程完成旧 UI 退场，移动/桌面一致。
- 中英双语、WCAG 2.2 AA 目标、reduced motion、真实设备性能。
- 组员盲体验、用户任务观察与问题复验，证明体验差异不是开发者自述。

### G8 — EKS, Operations & Singapore Release Readiness

**结果**：产品可在 AWS Singapore 区域以生产方式部署、运行、恢复，并具备发布候选责任材料。

- Terraform 建 VPC/EKS/RDS PostgreSQL/Redis/SQS/S3/KMS/Secrets/观测与最小 IAM。
- Kustomize 环境 overlay、Gateway/WAF/TLS、NetworkPolicy、PDB、HPA、spread、probes、graceful shutdown。
- OTel dashboards/alerts/runbooks，SLO、成本预算、Provider/queue/data health。
- 滚动/canary、schema migration、回滚、Pod/节点/Provider 故障演练。
- RDS PITR/S3 lifecycle/恢复演练达到 RPO/RTO 目标。
- 新加坡英文体验、时区、本地危机资源和数据流映射。
- PDPA/跨境 LLM、商店、HSA 边界、ACRA/GST/支付等按发布时最新政策复核并由授权人员签字。

### G9 — Complete Product Acceptance

**结果**：不是“功能都写了”，而是一个外部用户无需讲解即可使用、团队可以运营、课程可以证明技术与创新的发布候选。

- `complete-product-acceptance.yml` 所有 required 项 PASS 且有证据。
- 全新环境从 IaC/迁移到应用可复现；核心 demo 与生产配置分离。
- Web/PWA/移动完整 E2E；关键 AI 纵向评测和盲测通过。
- 安全/隐私/灾备/性能/可访问性/多语言门禁通过。
- 8—12 分钟展示脚本同时证明产品差异、AI 技术、数据管线与 Kubernetes 运行。
- 运维、事故、数据权利、Provider 故障、模型回滚和发布 runbook 经非作者演练。
- 所有 known limitation 有 owner、风险、到期和用户影响；无被隐藏的 P0 阻塞。

## 4. 纵向切片策略

Agent 应优先完成能跨越产品—AI—数据—云—体验的薄切片。例如：

### Slice A：被打断后仍然理解我

真实消息 → Choreographer plan → SSE event → 用户打断 → cancel Provider/未发消息 → 新计划 → 对话落库 → trace + Playwright + blind quality review。

### Slice B：Aurora 在正确时间回来

用户约定 → WakeIntent → durable schedule → quiet/risk/relevance decision → Push/in-app → deep link → 上下文恢复 → 反馈更新契约 → 多副本与重启测试。

### Slice C：纠正我的画像并传播

对话提取 → provenance/claim → 用户纠正 → memory/profile version → retrieval 变化 → 星空变化 → capsule invalidation/recompile → audit 和删除测试。

### Slice D：从共鸣体到慢信

授权 → Genome compile → sandbox → publish → 多策略发现 → 高保真对话 → 慢信 → 双方同意 → 撤回/举报路径 → 隐私与质量评测。

纵向切片减少“后端完成但没有体验”“页面完成但数据是假”“K8s 完成但应用仍单机语义”的局部最优。

## 5. 评测与体验闭环

### 5.1 三层 AI 评测

- Component：提取、检索、分类、结构化输出、policy 和 critic。
- Trajectory：多轮、多日、打断、主动、纠正、矛盾、关系演化。
- Product：盲测单核/双核、普通 Prompt/Genome、相似/多策略匹配，测真实偏好和任务结果。

每个评测集固定版本、样本来源、语言、隐私等级、模型参数和 judge 协议。LLM-as-judge 只能与规则、人工 pairwise 和故障分析组合，不可单独证明优越。

### 5.2 组员体验协议

每个 G4—G7 里程碑至少执行：

1. 非实现者按任务脚本盲体验，不先听功能讲解。
2. 记录屏幕、时间、阻塞、误解、情绪反应和自由评论。
3. 对关键对话采用 pairwise 隐去版本来源。
4. 问题按“阻断/伤害信任/破坏差异/一般摩擦/美化”排序。
5. 修复后由原体验者或另一人复验，不能由作者自证关闭。

## 6. 验收矩阵

| 维度 | 最终必须回答的问题 | 代表性证据 |
|---|---|---|
| Product | 用户能否说出独特价值并完成完整闭环？ | 任务录像、访谈、漏斗、问题复验 |
| Aurora | 是否真的具有连续、主动、可打断的生命感？ | trajectory eval、pairwise、stream trace |
| Memory/Profile | 获取是否准，纠正/撤回是否传播？ | golden set、reconciliation、provenance test |
| Capsule | 是否像、是否守边界、是否可撤回？ | fidelity/leakage eval、沙盒与版本证据 |
| Psychology | 是否有证据、非诊断、可解释且安全？ | manifest、专家审阅、风险场景测试 |
| UX | 是否跨端流畅、美观、可访问、可理解？ | Playwright、视觉/a11y、真实设备盲测 |
| Security/Privacy | 越权、密钥、同意、删除和跨境是否受控？ | scans、negative tests、签字、删除演练 |
| Cloud | 多副本、故障、伸缩、迁移和恢复是否成立？ | EKS trace、chaos、HPA、restore report |
| Operations | 团队能否发现、处置和回滚问题？ | dashboard、alert、runbook 演练 |
| Singapore | 是否达到当地发布候选条件？ | 最新政策核验、数据流、人工签字 |

## 7. 机器账本使用规则

`docs/goal/complete-product-acceptance.yml` 是状态索引，不是自动真相：

- `UNASSESSED`：尚未用当前 HEAD 和环境核验。
- `IN_PROGRESS`：已有工作但结果门未满足。
- `BLOCKED`：有具体不可绕过的外部/依赖条件，必须给 blocker evidence。
- `FAIL`：已核验但不达标准。
- `PASS`：当前 HEAD/环境满足，证据可复现且独立复核要求已完成。

状态变化必须同时更新 `verified_at`、`commit`、`evidence` 和 `reviewer`（如需要）。禁止批量把 `UNASSESSED` 改 `PASS`，禁止以某个测试类替代完整 gate。

## 8. 发布与回滚规则

- 数据迁移与应用部署分离；先备份/预检，再 migration job，再 canary。
- 发布前冻结 API/event/Prompt/Self/Skill 版本清单。
- canary 观察技术 SLO、AI 质量/安全信号和产品错误；超过阈值自动或人工回滚。
- 回滚包括应用、feature flag、Prompt/model/Self/Genome Compiler；数据库使用向前修复或已演练兼容方案，不盲目 down migration。
- Provider 故障时可路由经批准的等价模型或诚实降级，不允许生产静默 Mock。
- 任何 P0 越权、敏感泄露、危机流程严重错误或不可删除数据均停止扩大发布。

## 9. 成本与性能

效果是核心目标，但“不惜模型成本”不等于不测成本：

- 记录每条核心旅程的 token、模型调用数、延迟和失败率。
- 高成本双核/多 critic 用在能显著提升体验的路径；由评测证明收益。
- 可缓存的是稳定的结构化派生物和编译结果，不缓存敏感即时回答造成错用。
- 建立预算告警和 per-user abuse guard，但 demo/研究可有经批准的高质量配置。
- 优化顺序：减少无效上下文和重复调用 → 模型路由/并行 → 缓存 → 才考虑降级质量。

## 10. 文档完成不等于项目完成

本体系刻意把愿景写完整，是为了让 Agent 不再依赖人类逐条提示。它同时要求 Agent 持续用实现和证据修正自己：

- 文档未覆盖的小决策由 Agent 自主解决并记录。
- 文档与真实技术约束冲突时做实验，不机械服从不可能条件。
- 核心承诺变更必须显式升级给产品所有者，不可暗中删减。
- 直到 G9 成立前，所有“完成”只能描述具体 gate 或 release candidate 状态，不能宣布整个目标结束。
