# Inner Cosmos 云原生高级能力展柜与课程评分设计

> 层级：`L1-CLOUD-SHOWCASE`，服从文档 24 的最终战役与环境边界。
>
> 目的：把“为了产品真正需要的云原生能力”和“为了课程展示主动扩展的高级能力”组织成同一套可运行、可评分、可降级的技术叙事。

## 1. 设计立场

项目允许有意识地采用高于当前 demo 规模所需的技术。复杂度不是主要否决理由；真正的否决条件是：没有任何产品连接、无法真运行、无法证明成功/失败、破坏主演示、泄露数据或把环境限制说成成功。

能力组合采用三层：

- **Foundation**：生产级 Kubernetes 基础语义，必须严谨，不允许包装性夸大。
- **Hero**：直接保护用户体验的三条核心链路，是课程技术叙事的主角。
- **Extension Gallery**：主动扩展的插件和高级能力，用于展示广度、工程判断和生态掌握。

每个扩展都以一个 `Showcase Card` 管理：

```yaml
id: CN-...
product_hook: 它改善哪个 Inner Cosmos 场景
why_cloud_native: 为什么 Kubernetes/生态能力优于应用内硬编码
environment: [local-kind, academy-eks, commercial-sg]
success_demo: 可观察成功动作
failure_demo: 可观察失败、拒绝或回滚动作
metrics: 定量信号
kill_switch: 如何关闭而不破坏核心产品
fallback: 环境不支持时如何保持语义
evidence: manifest/code/test/runtime/screenshot/trace
resource_budget: CPU/内存/存储/时间
claim_boundary: 演示能证明什么、不能证明什么
```

## 2. 三环境分工

| 环境 | 角色 | 允许结论 |
|---|---|---|
| `local-kind-showcase` | 完整插件实验场；可建多节点 kind/Cilium profile | 证明插件、故障与业务语义真运行；不证明 AWS 托管生产 |
| `academy-eks` | 课程集群的原生 K8s/EKS 证据；每次 session fresh probe | 只证明当次权限、容量、CNI/controller 下实测通过的能力 |
| `commercial-sg` | 新加坡生产蓝图/授权环境 | 证明 RDS/ElastiCache/SQS/Secrets/IAM/灾备等生产责任；未获账号时保持设计或 external gate |

同一 immutable image、领域语义、Flyway 和 API contract 贯穿三环境；差异通过 adapter、config、Kustomize/Helm values 和 capability detection 表达。

## 3. Foundation：先把基础能力做得专业

### F1 Workload 与生命周期

- API、AI/event worker、scheduler、migration、batch role 分开；service account、resources、probes、priority 和扩缩策略按角色配置。
- startup、liveness、readiness 分离；liveness 只代表进程是否应重启，readiness 才包含依赖/角色接单能力。
- API 先 readiness false，再 drain SSE；`preStop`、grace period、PDB、spread 和客户端 durable replay 共同保证连续性。
- scheduler 保持应用租约；migration 使用 sync wave/Job，业务 Pod 在兼容 schema 范围内启动。

### F2 Data 与事件一致性

- PostgreSQL 是事务真相；outbox/inbox、lease、SKIP LOCKED、idempotency key 和 dead/replay 状态可观察。
- Academy/local 使用 JDBC transport；commercial-sg 使用 SQS/DLQ adapter。两者运行同一事件 handler contract。
- schema 采用 expand→dual-read/write（必要时）→backfill→contract，不使用 exact version 阻断 canary 共存。

### F3 Security 与供应链

- non-root、read-only rootfs、seccomp、drop capabilities、bounded resources、least-privilege RBAC、no automount token（不需要时）。
- image digest、SBOM、Trivy、Cosign/provenance；生产禁 mock/local endpoint/debug CA/裸凭据。
- P0 内容不进入 label、trace attribute、Prometheus high-cardinality dimension、日志或 Kubernetes event。

### F4 Observability 与运营

- metrics/logs/traces 以 deployment/runtime/model/prompt/skill version 关联；用户只用短期不可逆相关键。
- SLO 同时覆盖 TTFT、turn completion、SSE resume、outbox age、WakeIntent lateness、fallback/structured-output、projection lag 和通知投递。
- runbook 的每个告警都对应一次可执行诊断和恢复动作。

## 4. Hero：三条必须现场破坏的链路

### H1 Aurora zero-loss drain

在 Aurora 多气泡输出中删除正在服务的 Pod。Gateway 停止新流量，服务器优雅终止，客户端携 `Last-Event-ID` 重连另一副本并从 PostgreSQL/Redis timeline 恢复。验证消息顺序、内容 event id、LLM/记忆副作用幂等、恢复时间和用户可继续打断。

技术点：Gateway API、readiness gate、termination lifecycle、PDB、topology spread、SSE、Redis fan-out、durable PostgreSQL timeline、idempotency。

### H2 Business-metric autoscaling

批量结束对话产生记忆/画像/共鸣体 projection outbox。KEDA 根据 ready count/oldest age 扩 worker；多个 worker 竞争 lease，inbox 保持 exactly once；积压清空后按 cooldown 缩容。KEDA 不可用时 Prometheus Adapter/native HPA 或固定 worker 保持正确性。

技术点：custom/external metrics、KEDA、transactional outbox、SKIP LOCKED、lease、HPA behavior、backpressure、DLQ/replay。

### H3 Semantic distributed trace

从一次用户 turn 看到 HTTP/SSE、context assembly、vector/full-text retrieval、Provider、outbox、worker、memory/profile projection、WakeIntent/notification。异步部分用 context propagation 或 span link，不伪造一个同步父子树。

技术点：OTel SDK/agent、Collector、Tempo/Jaeger、W3C Trace Context、tail/head sampling、exemplars、privacy-safe attributes、Grafana correlation。

## 5. Extension Gallery

### E1 Progressive delivery：Argo Rollouts/Flagger + Gateway API

产品连接：Aurora prompt/runtime、共鸣体 compiler 或 memory ranker 可能不 crash 却伤害体验。

- stable/canary 两套 workload，HTTPRoute 权重逐步提升；
- analysis 使用 5xx、TTFT、SSE reconnect、fallback、structured parse、outbox age 与合成 AI quality/safety probe；
- 失败自动 abort、权重归零和 rollback；
- Academy 插件不可用时，用原生 weighted backend + PromQL gate 脚本展示相同语义。

### E2 GitOps：Argo CD

产品连接：三 profile 配置容易 drift，migration/策略/观测需有顺序。

- App-of-Apps 或 ApplicationSet 管理 app、data-demo、observability、policy、showcase；
- sync wave：policy/CRD → migration → stable/canary → smoke；
- 演示手改 Deployment 后 drift 检测、自愈或受控回退；
- secrets 只引用外部对象，不进入 Git。

### E3 Policy-as-code：Kyverno + Cosign

产品连接：保护心理数据、Provider key 与发布真实性。

- verifyImages、digest-only、non-root、seccomp、drop caps、resources、approved registries；
- production namespace 禁 hostPath、privileged、local endpoint、mock provider、可变 tag 和明文 Secret manifest；
- Academy static PV 例外必须限定 overlay/namespace/期限并有审计；
- 演示安全 manifest 通过、不安全 manifest 被 admission 拒绝。

### E4 Network observability：Cilium/Calico + Hubble

产品连接：P0 数据、IdP、Provider 与管理面需要最小路径。

- Gateway→API、API/worker→Postgres/Redis、Prometheus→metrics、受控 egress→IdP/Provider；
- default deny，验证跨 namespace DB、metadata endpoint、未授权 Pod 被拒；
- Hubble 展示一次 Aurora turn 的网络依赖但不暴露正文；
- Academy 不能替换 CNI，只在实测 enforcement 后声明。

### E5 TLS/Secret lifecycle：cert-manager + External Secrets

产品连接：移动 PKCE/API TLS、Provider 凭据和轮换。

- cert-manager 管理本地/商业证书生命周期；
- External Secrets 连接 commercial Secrets Manager，refresh/rotation 后 workload 受控 reload；
- Academy 使用 operator 当次创建的 Kubernetes Secret，保留相同 Secret contract；
- 演示过期前续期或 secret version 变化，不输出 secret 值。

### E6 Runtime security：Falco

产品连接：防止容器 shell、敏感挂载读取、异常出站和运行时篡改。

- 规则覆盖 shell in container、写只读路径、访问 service-account token、可疑进程/端口；
- 告警进入统一 dashboard/runbook；
- 使用无敏感数据的测试事件证明检测与降噪。

### E7 Chaos：Chaos Mesh/Litmus + native fallback

产品连接：陪伴与回访在 Provider/Pod/Redis/网络失败时仍应可信。

- Pod kill、network delay/loss、Provider 429/timeout、Redis interruption、DB read-only、worker kill；
- 每次实验声明 steady state、fault、business invariants、abort condition、recovery time；
- 插件不可装时使用 kubectl/Toxiproxy/测试 Provider 注入，证据格式相同。

### E8 Backup/restore：Velero + database-native copy

产品连接：用户长期记忆和同意记录不可因集群丢失。

- namespace/config backup、off-cluster object storage、PostgreSQL logical/managed backup；
- 清洁环境 restore 后验证 schema、owner scope、outbox/inbox 和关键旅程；
- Academy 静态 PV/同节点副本只算课堂恢复演练，不称灾备。

### E9 Cost intelligence：OpenCost + AI cost telemetry

产品连接：双核/多模型、embedding、trace backend 和 worker 弹性成本高。

- 展示 namespace/workload 成本、请求/turn 成本、Provider token bucket、quality-per-cost；
- 将低优先 batch 延后或 scale-to-zero，同时保证 Aurora/WakeIntent priority；
- 不记录稳定 user id，不用成本指标自动伤害高风险用户体验。

### E10 Feature management：OpenFeature

产品连接：Aurora kernel、prompt、Capsule compiler、matching strategy 和主动性实验需要 kill switch。

- flag evaluation 与 deployment/config version 进入 trace；
- 支持合成 cohort、百分比、平台和紧急关闭；
- 不用脆弱性/心理特征做未同意分群；
- 与 canary 分工：feature flag 管语义/人群，Rollout 管 workload 发布。

### E11 Workload identity：SPIFFE/SPIRE 或 bounded mesh

产品连接：API、worker、scheduler 访问数据/事件的权限不同。

- 短期 workload certificate、mTLS、role identity 和 rotation；
- mesh 只在独立 showcase profile 用于 mTLS/telemetry/受控 traffic shadow；
- 证明 sidecar/ambient 模式不会破坏 Aurora SSE；否则自动回退非 mesh 主路径。

### E12 Cloud-native batch：Indexed Jobs

产品连接：embedding backfill、画像重算、Capsule benchmark、AI eval。

- shard、DB checkpoint、idempotent retry、parallelism、TTL 和低 priority；
- 在线 API/WakeIntent 在资源压力下保持优先；
- 失败 shard 单独重跑，不重算成功数据。

### E13 Portability demo：Dapr

产品连接：展示 event/secret/config adapter 的跨环境切换。

- 仅封装 transport 边界，不替换 PostgreSQL 事务真相或现有 owner/security；
- 在独立 profile 演示 JDBC/local 与受控 pub/sub component 的切换、trace 和重试；
- sidecar 故障时核心 profile 仍可运行。

### E14 Elastic skills：Knative

产品连接：低频、高成本、可隔离的心理教育 skill 或 simulator/eval 批次。

- CloudEvents、scale-to-zero、concurrency、cold-start 指标；
- Aurora 主对话/SSE 不经 scale-to-zero 服务；
- 高风险 skill 仍受 consent、policy 与专业门禁。

### E15 Infrastructure composition：Crossplane

产品连接：commercial-sg 的 bucket/queue/secret policy 形成可审计环境组合。

- 本地 control plane 演示 Composition/claim/policy；
- 未获 AWS 账户时不创建真实资源，不把 dry-run 称生产 IaC；
- Terraform 仍可作为商业主路径，Crossplane 是平台工程扩展。

### E16 Aggregated product API：GraphQL BFF

GraphQL 不是 CNCF/Kubernetes 特性，但可作为 API 技术展示。只有当 Cosmos/画像工作台能明显减少瀑布请求并改善字段选择时实施：

- BFF 聚合只读 projection，不绕过 domain service/owner scope；
- schema/codegen、depth/complexity/cost limit、persisted query、DataLoader、field auth；
- subscription 不替换 Aurora SSE；mutation 不复制现有核心 REST；
- 用首屏请求数、payload、TTI 与 IDOR negative test 证明收益。

## 6. 选择与排期

### Wave 0：事实可信

修 probes、metrics discovery、NetworkPolicy enforcement、backup/evidence/schema compatibility。任何插件不得绕过这些缺陷。

### Wave 1：英雄链路

并行不超过三支：H1 lifecycle/SSE、H2 KEDA/outbox、H3 OTel/trace。完成后形成课程主叙事。

### Wave 2：发布与安全组合

优先 E1 Argo Rollouts、E2 Argo CD、E3 Kyverno/Cosign、E4 Cilium/Hubble。它们共同说明“如何发布、如何约束、如何观察网络、如何自动回滚”。

### Wave 3：运营纵深

从 E5—E12 选择资源允许的能力，建议 cert/secret、Falco、Chaos、OpenCost/OpenFeature、Indexed Jobs；Velero 在有真正 off-cluster target 时加入主演示。

### Wave 4：扩展包装

从 E13—E16 选择能独立运行且不会污染主链的 1—3 项。它们进入技术附录或备用演示，不强迫 8—12 分钟全部展示。

## 7. 评分与退出

每项 10 分：产品关联 2、Kubernetes/云原生深度 2、真实成功证据 2、故障/安全证明 2、演示可理解性 2。

- 9—10：主演示候选。
- 7—8：技术展柜/答辩备用。
- 6：只进入附录。
- <6：不继续投入或返工，不以“已安装”计完成。

一项能力退出实现不等于删除全部资产：保留 ADR、实验结果和失败边界可展示工程判断。但退出项不得占用主应用资源、改变默认 profile 或提高教师演示失败率。

## 8. 课程证据包

最终每条主能力都交付：

1. 一页因果图：用户场景 → 应用机制 → Kubernetes/CNCF 能力 → 指标/故障 → 用户结果。
2. version-pinned manifest/Helm/Kustomize 与 image digest。
3. 一键 setup/verify/teardown；Academy 脚本不得依赖长期凭据。
4. 成功与失败时间线、截图/trace/dashboard、机器断言和原始摘要。
5. 资源占用、兼容矩阵、kill switch、fallback 与 claim boundary。
6. 60—90 秒现场片段和失败时的离线 evidence fallback。

这样既能展示生态广度，也能让老师看见真正的 Kubernetes 推理、生产规范和应用耦合，而不是一串无法解释的产品名。
