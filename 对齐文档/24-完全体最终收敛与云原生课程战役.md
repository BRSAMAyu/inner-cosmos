## 1. 最终裁决

Inner Cosmos 的完全体不是“功能数量最多的原型”，而是同时交付两种能被直接证明的卓越：

1. **可感知的产品卓越**：Aurora、记忆、画像、共鸣体、匹配与慢社交形成一个连贯、可信、细腻、可纠正的体验；用户无需听技术讲解，也能明显感到它比通用聊天工具更理解自己、更会长期陪伴、更能促成人与人的高质量连接。
2. **可解释的云原生卓越**：Kubernetes 能力不是贴在应用外面的清单，而是直接保护 Aurora 的连续性、AI 数据管线的吞吐与一致性、主动回访的准时性、心理数据的隔离以及高风险 AI 版本的可回滚性。课程展示必须能观察、施压、破坏并证明恢复。

这两条主线同等必要。基础设施不能代替体验，视觉效果不能掩盖数据或并发错误，AI 创意也不能以不可维护、不可观测、不可回滚为代价。

## 2. 合法完成状态

只有以下两种终态合法：

- `COMPLETE`：全部机器门和全部必需人工门均为 `PASS`。
- `RELEASE_CANDIDATE_BLOCKED_BY_HUMAN_GATE`：全部机器可执行任务均已完成并有可复现证据；剩余项只能是 Provider/Apple/Google/AWS/域名/法务/心理专业审查/真实用户同意等确需授权或实体参与的门禁，并明确列出责任人、输入和关闭方法。

以下情况不得称为完成：只构建成功、只有 mock、只有单元测试、只有静态截图、只有作者机器可跑、只有 manifest、只创建了 Kubernetes 对象却未验证语义、只让 Agent 自评质量、或用 Academy 证据冒充新加坡商业生产。

## 4. 产品完全体：六条黄金闭环

所有功能、页面和 API 都必须服务至少一条闭环。孤立 CRUD 不算产品完成。

### 4.1 Aurora：有时间感、关系感和行动能力的陪伴者

目标不是更长的 prompt，而是可观察的 Agent runtime：

- speaker 与 planner/critic 可单核或双核运行；模型、策略、预算和超时可配置并可降级；
- 支持多气泡、思考停顿、补充、打断、停止、重规划和主动消息；这些是事件状态机，不是前端伪动画；
- `Self / Constitution / Emergence / Relationship State` 保留创新自由度，同时版本化、可回滚、可解释，用户能调节主动性、亲密度、节奏和边界；
- WakeIntent 具备时区、安静时段、最新偏好、风险复核、幂等投递和反馈学习；
- 所有 turn 可在断网、前后台、刷新、进程终止和 Pod 替换后从 durable timeline 恢复，不重复气泡或副作用；
- 真实 Provider 以安全的本地环境变量注入，完成固定 golden conversations、干扰测试、失败注入、延迟/成本记录和盲评。Mock 只保证可复现，不证明质量。

验收至少覆盖：首次倾诉、长期回访、打断后重规划、记错后纠正、主动陪伴、危机边界、Provider 超时/429、跨设备恢复。每条旅程记录 baseline 与当前版本，不能只给主观描述。

### 4.2 记忆—画像—星空：同一份可纠正的数据真相

数据链必须是：授权收集 → 来源标注 → 抽取候选 → 用户确认/纠正 → 记忆巩固/衰减/合并/冲突 → 多视图投影 → 检索使用 → 反馈更新。

- 传统结构化表、全文检索、pgvector 和时间/关系权重共同工作；向量库不是唯一真相。
- 检索按当前任务组织有限上下文，兼顾相关性、时效、显著性、来源可信度、多样性与隐私目的；信息越多不等于更好。
- 用户画像区分事实、偏好、价值、关系、目标、情绪模式、沟通风格、边界与置信度；展示来源、更新时间、冲突和修改入口。
- timeline、daily record、weekly review、todo、heart diary、belief、thought shredder 必须回流同一 lineage，而不是平行小工具。
- 记忆星空首先是理解工具：视觉层级表达时间、主题、情绪重力与关系；支持筛选、来源、纠正、合并、淡化和 reduced-motion，不能只做漂亮粒子。

### 4.3 共鸣体：高拟真、受授权、可撤回的人格编译系统

共鸣体不是静态卡片，也不是把全部画像拼进 prompt。它由可版本化 compiler 构建：

- `source selector` 仅取用户授权且适合 P2 的信息；
- `trait/style model` 表达措辞、节奏、习惯、观点、矛盾与边界，不复制 P0 原文；
- `prompt/program composer` 针对当前场景选择证据和行为规则；
- `speaker` 生成回应，`critic` 检查人格一致、隐私、越权、虚构和重复；
- 发布前 sandbox/preview，发布后可追踪版本、立即撤回、删除传播有 durable receipt；
- simulator 资产必须用匿名/合成 benchmark、消融和一致性指标证明，而不能以真实用户私密数据外泄换取拟真。

陌生共鸣体对话可有宽松防滥用额度，但不得用僵硬轮次伤害正常关系发展；限制必须由风险、同意、成本或滥用证据解释。人工评价只决定默认上线/PASS，不阻止 Agent 完成 compiler、retrieval、speaker、critic、基线对照和自动评测。

### 4.4 匹配与人与人连接：从相关到安全而有意义

匹配不是单一余弦相似：

- 候选召回使用画像/记忆/意图向量，并先执行屏蔽、授权、年龄/地域等合法约束；
- 排序支持相似共鸣、互补探索和惊喜发现三种意图，结合置信度、活跃度、边界兼容、MMR 多样性与反馈；
- 每个推荐给出可理解原因，但不泄露他人敏感特征；
- 共鸣体沙盒 → 双方显式同意 → 慢信/连接 → 反馈闭环，拒绝与撤回立即生效；
- 用固定匿名用户对、负例、冷启动、公平性、重复推荐和屏蔽传播测试证明质量。

不发展公开流量广场或无限社交图谱。小组能力只能是明确同意、目的限定、可退出的受控空间。

### 4.5 慢社交：节奏本身就是价值

慢信和星海广场要让用户感到安全、期待和被认真对待：草稿保护、到达时间解释、发送前预览、状态清晰、撤回/屏蔽、拒绝不羞辱、通知不过载、失败可恢复。每个状态都由后端事实驱动，刷新和重试不得重复发送。

### 4.6 心理技能：可扩展能力而非诊断包装

Aurora 可通过受控 skill/plugin 注册心理教育、反思、量表或建模能力；每个能力包含 schema、适用/禁用条件、来源版本、同意、输出边界、审计与撤销。用户可以显式调用，Agent 也只能在许可和适用条件满足时调用。未经专业审查的内容保持实验标签，不宣称诊断、治疗或医疗效果。

## 5. UIUX 完成定义

“精美”必须转化为可验收的系统，而不是添加更多天气、BGM 或自定义鼠标。

### 5.1 统一体验语言

- 五空间 IA 保持唯一主导航；新增能力归入既有空间或二级流程，不增加第六主 tab。
- 使用共享 token、字体、间距、色彩、层级、motion 与组件状态；Web、Android、iOS、Windows/macOS 共用产品核心与平台适配器。
- 动效只表达状态、因果和空间关系；支持 `prefers-reduced-motion`，低端设备自动降低复杂度。
- 中文和 en-SG 均无硬编码、截断或布局破坏；日期、时区、数字和语气本地化。

### 5.2 每个组件的强制状态矩阵

关键组件必须在 `empty/loading/streaming/success/partial/offline/error/retry/permission-denied/expired/long-copy` 下验收，并覆盖：

- 390px 小屏、主流桌面、横竖屏；
- 深/浅色、200% 字体、reduced motion、高对比；
- 鼠标、键盘、触摸、TalkBack/屏幕阅读器；
- 慢网络、重复点击、刷新、后台、杀进程和权限拒绝。

验收采用真实浏览器/模拟器点击、视觉回归、axe/语义检查和关键旅程，而不只查询 DOM。48dp 触控目标、可见焦点、键盘不遮挡输入、错误就地解释、危险操作可撤销/二次确认是硬门。

### 5.3 自主体验评审

Coding Agent 对每条黄金旅程完成：运行 → 截图/录屏 → 逐组件点击 → 记录摩擦 → 修复 → 重跑。Agent 评审可以关闭机器可验证缺陷；审美或情感偏好由非作者/用户最终签字，但不能把可自动发现的断链、错位、空状态或不可点击留给用户。

## 6. Cloud-Native Showcase：应用驱动的高级 Kubernetes

### 6.1 先修可信度缺口

在增加插件前必须：

1. 将 startup、liveness、readiness 分开；liveness 不因 PostgreSQL/Redis/Provider 暂时失败触发重启风暴。
2. API、worker、scheduler 暴露 role-specific health/metrics，并在 base/Academy 中具有正确的 Prometheus 发现配置。
3. NetworkPolicy 用 deny/allow probe 验证实际 CNI enforcement；只创建对象不能记 PASS，并收紧跨 namespace 数据端口。
4. 明确 backup CronJob 是否属于 overlay；同节点静态 PV 备份不得冒充 off-cluster disaster recovery。
5. 清理公开 evidence 中违反仓库脱敏合同的 AWS account、registry、ELB 等标识，并检查当前树/历史边界。
6. 渐进发布前把 exact schema gate 改为版本兼容区间和 expand-contract 迁移合同。

### 6.2 三条必做英雄链路

| 链路 | 产品因果 | 实现重点 | 必须展示的证据 |
|---|---|---|---|
| `CN-ZERO-LOSS-DRAIN` | Aurora 不能因 Pod 更新丢失正在发生的陪伴 | readiness 摘流、graceful shutdown、PDB、topology spread、durable timeline、`Last-Event-ID` | 在输出多气泡时删除 API Pod；客户端恢复、无重复气泡/副作用，记录恢复时间 |
| `CN-EVENT-DRIVEN-AUTOSCALING` | 对话结束会触发记忆/画像/共鸣体投影；瓶颈是 backlog 而非 API CPU | outbox ready/oldest-age/dead 指标；KEDA Prometheus/Postgres scaler；worker 竞争、幂等、cooldown；失败 fallback | 批量结束会话→worker 1→N→积压清零→scale-in；杀 worker 后 lease 重领且 inbox exactly once |
| `CN-OTEL-SEMANTIC-TRACE` | 要解释“为什么记得慢、回访迟、成本高” | W3C context/span links 贯穿 API/SSE→LLM→outbox→worker→memory/profile→WakeIntent；OTel Collector+Tempo/Jaeger | 从一次 turn drill-down；可见 TTFT、provider、retrieval、outbox lag、projection、notification，标签不含正文/用户标识 |

本地 kind 是三条高级能力的完整实验与演示场。Academy 每次会话先做 CRD/RBAC/容量 capability probe；KEDA/trace backend 不可安装时，使用 Prometheus Adapter/native HPA、Collector 导出或固定 worker 作为明确降级，不能伪称插件成功。`commercial-sg` 保留正式托管目标。

### 6.3 次硬门

- `CN-PROGRESSIVE-DELIVERY`：stable/canary + Gateway API weighted route；以 5xx、TTFT、SSE reconnect、fallback、结构化输出、安全 synthetic probe 和 backlog 自动停止/回滚。Argo Rollouts 可用则采用；不可用时使用 HTTPRoute + PromQL gate 脚本。
- `CN-POLICY-AS-CODE`：Kyverno 或 ValidatingAdmissionPolicy/CI policy 强制 digest、non-root、seccomp、drop caps、resources、approved registry、生产禁 local endpoint/mock/裸 Secret；kind 展示坏 manifest 被拒，Academy 只在 webhook/权限实测后声称 enforcement。
- `CN-NETWORK-ISOLATION`：Gateway→API、API/worker→data、Prometheus→metrics、受控 Provider/IdP egress 精确 allow；本地用支持 NetworkPolicy 的 CNI 并记录 deny/allow，Academy 单独声明能力。
- `CN-PRIORITY-AND-BATCH`：在线 Aurora/SSE 与到期 WakeIntent 优先于 embedding backfill/评测；画像重算/embedding/eval 使用可重入 Indexed Job、分片 checkpoint 和 TTL。
- `CN-CHAOS-RECOVERY`：Pod/node/provider 429/timeout/Redis/DB/queue故障脚本；验证无丢失、无生产 Mock fallback、可告警、可回放，并量化恢复时间。
- `CN-PROFILE-PORTABILITY`：同一 image digest 在 `local-kind`、`academy-eks`、`commercial-sg` 合同下运行；环境差异只在 adapter/config/overlay。

### 6.4 高级能力展柜：允许为课程深度扩展

技术复杂度不是主要否决理由。三条英雄链路保证“真的有用”，其上允许建设一个更宽的 Cloud-Native Showcase：只要能力与产品存在正向关系、能真实运行并能讲出证据，即使当前规模尚不强制需要，也可以作为课程技术深度实现。每个扩展必须同时具备：`product hook`、可运行的最小纵向切片、一个成功与一个失败/拒绝证据、资源/权限边界、关闭开关和 Academy fallback；只画架构图或只安装 chart 不计分。

建议能力池按价值和依赖选择，完成 Tier A 后尽可能扩展 Tier B；Tier C 只在不会破坏主线时采用：

| 层级 | 技术/能力 | 与 Inner Cosmos 的自然连接 | 可展示的技术点 |
|---|---|---|---|
| A | KEDA | memory/profile/capsule outbox backlog | 非 CPU 业务指标扩缩、cooldown、幂等竞争 |
| A | OpenTelemetry + Collector + Tempo/Jaeger | Aurora 跨 LLM、检索、异步投影、WakeIntent | span link、采样、隐私安全属性、端到端 SLO |
| A | Argo Rollouts/Flagger + Gateway API | prompt/runtime/model 版本可能语义退化 | weighted canary、PromQL/AI quality analysis、自动回滚 |
| A | Kyverno/ValidatingAdmissionPolicy + Cosign | P0 数据和 Provider key 的供应链边界 | 签名镜像、digest、Pod Security、策略拒绝证据 |
| A | Cilium/Calico + Hubble | P0/API/data/Provider 的最小网络路径 | eBPF/NetworkPolicy、流量可视化、deny/allow probe |
| B | Argo CD | 多 profile 的声明式发布与 drift | GitOps、健康检查、sync wave 先 migration 后 workload、回滚 |
| B | cert-manager + External Secrets | OIDC/API TLS 与外部密钥轮换 | 自动证书生命周期、SecretStore adapter；Academy 用模板/手工 Secret |
| B | Falco | 防止容器读取敏感路径或异常 shell | runtime detection、自定义心理数据规则、告警链路 |
| B | Chaos Mesh/Litmus | Provider、Redis、网络、Pod 故障 | 可重复 fault workflow 与业务不变量；原生脚本为 fallback |
| B | Velero + 对象存储 | PostgreSQL/配置/命名空间灾备 | schedule、restore drill、RPO/RTO；静态 PV 限制如实披露 |
| B | OpenCost | AI worker、API、观测栈的资源归因 | namespace/workload cost 与 LLM token 成本联合看板 |
| B | OpenFeature | Aurora runtime/prompt/capsule compiler 实验 | vendor-neutral flags、分群、kill switch、版本证据 |
| B | SPIFFE/SPIRE 或 mesh mTLS | API/worker/scheduler 工作负载身份 | workload identity、mTLS、短证书；本地/商业优先 |
| B | Indexed Jobs | embedding backfill、画像重算、离线 AI eval | shard、checkpoint、retry、TTL、在线/离线优先级 |
| C | Dapr | JDBC/SQS transport 与 secret/config adapter 的可移植展示 | component swap、pub/sub tracing；不得取代既有事务真相 |
| C | Knative Eventing/Serving | 低频高成本心理技能或离线模拟任务 | scale-to-zero、CloudEvents；冷启动与 SSE 不用于 Aurora 主链 |
| C | Crossplane | `commercial-sg` 托管资源声明与策略 | composition、environment promotion；Academy 只离线/本地展示 |
| C | Service Mesh | 内部 mTLS、AI worker 流量镜像、细粒度遥测 | 只用于独立展示 profile，不让 sidecar 伤害 SSE 主路径 |
| C | GraphQL BFF | 星空/画像工作台一次取得多种聚合视图 | schema/type/defer/cost limit；它是产品 API 展示，不冒充 CNCF/K8s 特性 |

Kafka、CloudNativePG、Karpenter、每用户一个 CRD/Pod 等仍不是默认目标，但不永久禁止：Agent 只有在完成 Tier A、资源允许且能提交独立纵向证据时才可加入。任何扩展不得改变下列基础事实：Academy 保持 JDBC outbox，禁止把短期个人 AWS key 注入 Pod 调 SQS；SQS/DLQ 属于 `commercial-sg` adapter；WakeIntent 保持应用 scheduler + lease，不改成粗粒度 CronJob；商业 PostgreSQL 目标仍是 RDS 而不是为展示强行自运维数据库。

扩展评分按五项各 0—2 分：产品关联、Kubernetes 深度、真实运行证据、故障/安全证明、演示可理解性。低于 6/10 不进入最终演示；已实现但未入选的能力可保留在技术附录，不挤占 8—12 分钟主叙事。

## 7. 五个执行战役

### W0 — 集成与事实基线

精确集成本轮四分支；生成一次前端 bundle；修复集成错误；运行 focused + Web build + Playwright，再跑集成全门。对账当前 HEAD、测试数、移动/桌面设备证据、未提交用户改动。重写机器状态，禁止把旧分支 green 直接记作 main PASS。

### W1 — Living Intelligence 与数据闭环

并行推进 Aurora runtime/质量、记忆画像 lineage、共鸣体 compiler/eval、匹配/慢社交反馈。先建立 scorecard 与 baseline，再改实现；每次改动都用固定旅程、真实 Provider（有凭据时）、自动评测、延迟/成本/隐私/失败恢复分层证明。

### W2 — 完整体验与多端精修

把新页面从“可访问”提升为六条黄金闭环；执行组件状态矩阵、i18n/a11y/performance/visual regression；Android 模拟器与 Windows Tauri 跑同一旅程；Apple 签名/APNs/真机保留外部门禁，代码与交接资产不能留空。

### W3 — Cloud-Native Showcase

先完成 6.1 可信度修复，再按“三条英雄链路 → 次硬门”实施。每项必须同时提交：业务因果、manifest/code、自动验证、运行证据、失败边界、Academy fallback、8—12 分钟演示步骤。不得以安装插件作为完成。

### W4 — 收敛与非作者验收

运行全栈清洁启动、黄金旅程、全项目测试、安全/IaC/供应链门、kind 故障演练、可用时 Academy rehearsal；生成需求→实现→测试→证据矩阵、演示脚本、运行手册、SBOM/hash。修复所有机器可执行问题后，才列剩余人工门。

## 8. 多 Agent 协作协议

### 8.1 并发模型

最多 2—3 个并发 Agent：

- Integrator/主 Agent：唯一修改全局权威、ledger、generated bundle、共享 API types 和迁移序列；负责合并、冲突和最终门。
- Product Agent：W1/W2 的一个边界清晰纵向切片。
- Cloud Agent：W3 的一个独立英雄链路或验证环境。

子 Agent 必须使用独立分支/worktree，从 Integrator 锁定的 commit 开始；不得改全局 ledger/state，不得提交别人的脏文件，不得同时改相同核心文件。任务可以再细分调查与实现，但总并发不得超过 3。

### 8.2 Agent 自主权

Agent 可以自主修复测试、设计、交互、数据和部署缺陷，替换低收益实现，添加必要测试/观测/文档，并在证据表明原方案不成立时选择更好的实现。以下情况才停：需要真实不可逆外部操作、缺少用户持有的账户/设备/法律或专业签字、将显著改变产品定位、或发现可能泄密/毁损数据的操作。

不把可通过阅读代码、运行系统、创建临时账号、模拟用户、浏览器/模拟器点击、故障注入或官方文档查证的问题问回用户。

### 8.3 Git 与冲突纪律

- 开始前记录 `git status --short --branch`、HEAD、目标文件和所有现有脏改；用户/其他 Agent 的改动必须保留。
- 一个可回滚的纵向能力一个 commit；commit 信息写 outcome，不写“misc/final”。
- generated bundle、lockfile、全局状态、Flyway version、共享 locale/API registry 由 Integrator 串行处理。
- 禁止 `reset --hard`、覆盖脏文件、通配 cherry-pick、重写历史、提交密钥或在证据中记录敏感标识。
- 分支完成必须提供 commit、diff scope、测试、运行证据、已知边界；Integrator 独立读 diff 和复跑高风险门后才合并。

### 8.4 测试节奏

采用风险分层，避免每个小改动都跑全量：

1. 编辑循环：静态检查 + focused unit/component test。
2. 纵向切片 checkpoint：模块测试 + 一条真实 journey/smoke。
3. 共享合同、迁移、安全、并发、runtime role、K8s 生命周期变更：相关集成/故障测试。
4. 分支合并、战役候选和最终 release：Java/Web/Playwright/i18n/secret/IaC/image 全门。

测试数字只能从当前集成 HEAD 的命令输出写入状态文件，禁止复制旧报告。

## 9. 完成验收矩阵

每个 acceptance item 必须包含：`target`、`status`、`implementation`、`automated_tests`、`runtime_evidence`、`quality_evidence`、`known_boundary`、`owner/reviewer`、`commit`。

AI/产品项额外包含 baseline、数据集版本、Provider/model/prompt/runtime 版本、样本总数（含失败）、延迟/成本、隐私与人工评价边界。云原生项额外包含环境身份、manifest digest、负载、故障动作、时间线、不变量、恢复时间和 fallback。

最终至少证明：

- 六条黄金闭环在 Web 与 Android 模拟器完成，Windows Tauri 完成核心闭环；
- UI 状态矩阵和核心无障碍/国际化/性能门通过；
- Java/Web/Playwright/移动/桌面测试与安全供应链门在当前 HEAD 通过；
- 三条 Kubernetes 英雄链路在 kind 真运行，Academy 能力有实测或明确限制；
- 8—12 分钟脚本由非作者从清洁 checkout 完成；
- 所有剩余项都满足“确实只能由人类/外部账户关闭”的定义。

## 10. 最终课程叙事

1. 同一签名 image 和配置合同运行于本地/Academy。
2. 用户与 Aurora 对话、打断并收到多气泡；删除服务 Pod，客户端无损恢复。
3. 对话结束生成记忆、画像与星空；批量任务触发 outbox，worker 随 backlog 弹性并保持 exactly once。
4. 用户纠正记忆，变化传播至画像/共鸣体/匹配；展示来源、授权与撤回。
5. 共鸣体沙盒、匹配解释、慢信和 WakeIntent 形成慢社交闭环。
6. Grafana/Tempo 从一个 turn 追踪至 LLM、outbox、projection 和通知，且无敏感标签。
7. 发布一个变慢/高 fallback 的 canary，质量门自动回滚；不安全 manifest/越权流量被策略拒绝。
8. 明示 Academy 静态存储和凭据限制，以及 `commercial-sg` 的 RDS/ElastiCache/SQS/Secrets/灾备目标，不夸大证据。

这段演示同时回答三个问题：用户为什么喜欢、系统为什么可信、Kubernetes 为什么在这里不可替代。
