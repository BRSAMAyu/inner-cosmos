> 产品策略：本地完整效果优先；Academy EKS 证明云原生能力；商业新加坡保留发布架构与人工门禁。

## 1. 本轮目标

从同一基线启动两个长时并行 PR，把“很多能力已经存在”的 Inner Cosmos 收敛为“效果突出、完整可用、可由非作者运行和演示”的产品候选。

本轮不是平均分配零散 issue。每条轨道都拥有一个完整结果：

- **Track A — Living Intelligence & Data Truth**：把 Aurora、记忆画像、共鸣体、心理 Skill 和数据权利真正做强，并用真实 Provider 与冻结评测证明效果。
- **Track B — Complete Experience & Delivery**：把这些能力收拢成自然、惊艳、可访问、双语、跨端且可一键运行的完整产品与 Demo。

两条 PR 合并后，集成审查者只做跨轨契约接线、全量回归、非作者验收和发布候选收尾，不应再发现某个核心产品域整体未做。

## 2. 共同工作原则

1. **从效果倒推实现**：先写用户可感知的验收，再决定代码；“类/端点存在”不是完成。
2. **允许推翻原型**：旧页面、旧交互和旧实现没有天然兼容权；保护用户数据、安全边界和可迁移证据即可。
3. **真实 Provider 但零密钥落盘**：密钥只由操作者通过环境变量或本机忽略文件注入，不得进入 Git、PR、日志、截图、测试夹具或 evidence。
4. **阶段性测试，关键节点全量回归**：开发中跑最小相关测试；每个可恢复 checkpoint 跑域内门禁；PR 前跑全量门禁。禁止为了追求数字在每次小修改后重复完整套件。
5. **持续循环**：Agent 在一个会话/Goal 中依次执行“观察—设计—实现—运行—体验—评估—修复—提交”，直到轨道完成或只剩真实人工门禁。
6. **诚实证据**：Mock、kind、Academy、真实 EKS、真实 Provider、自动评分和人工盲评必须分开标注。
7. **新发现自治**：发现新问题时可以改计划，但必须记录用户影响、根因、方案、验收、文件所有权和跨轨依赖。

## 5. Track A 结果契约

Track A 完成后，必须交付以下产品效果，而非孤立组件：

### A1. Aurora 质量闭环

- 建立真实、多轮、纵向、冻结的开发集与 held-out 集，覆盖关键对话和故障场景。
- 单核/双核/记忆/Self/关系/Skill 可消融、可回放、可比较。
- 双核采取自适应路由；planner、responder、critic 的职责与停止条件清晰。
- 多消息、停顿、打断、停止、重规划、主动回访有统一决策轨迹和时序语义。
- 对真实 Provider 反复优化，达到预登记质量阈值；不允许以“优势未显著”作为最终交付。

### A2. 高精度记忆与用户画像

- 画像 Claim 覆盖事实、偏好、关系、价值、情绪、习惯、语言风格、需求、目标、边界等维度。
- 每个 Claim 有来源、置信度、时间、同意、冲突、纠正、失效与下游使用记录。
- 检索按任务选择，处理新旧矛盾、重复、衰减和预算；真实 embedding 质量、延迟和成本有基准。
- 用户纠正/撤回会传播到 insight、vector、cache、capsule、matching 等下游。

### A3. 高保真共鸣体与画像匹配

- 动态 Genome 编译与运行时选择取代超长 Prompt 拼接。
- 具备候选生成、特征选择、事实/风格/边界 critic 和必要的 rerank。
- 相似、互补、探索、反回音室等策略建立在授权画像向量和显式特征上，有可解释理由。
- 真实 Provider 盲测比较动态 Genome、长 Prompt 和静态卡片；同时验证事实、风格、隐私和时序一致性。

### A4. 心理 Skill 产品化

- 至少三项非诊断、证据可追溯、可版本化、可显式调用/由 Aurora 建议的 Skill。
- 输入输出、适用范围、禁忌、同意、保留、风险升级和中英内容完整。
- 自动评测完成；专业人员审阅若未获得，明确保留为人工门禁，不得冒充通过。

### A5. 数据权利与 AI 可观测

- 建立派生物登记与纠正/撤回传播，覆盖缓存、prompt snapshot、队列、分析、通知及备份策略。
- 记录不含 P0 明文的模型/Prompt/记忆/Skill/成本/延迟/风险/反馈 trace。
- Provider 超时、限流和失败可降级且不伪装真实回复。

完整执行规格见 `docs/tracks/TRACK-A-LIVING-INTELLIGENCE.md`。

## 6. Track B 结果契约

### B1. 黄金用户旅程

- 将产品收敛为一条无需开发者解释的旅程：被理解 → 可纠正理解 → 合适的主动回访 → 看见变化 → 授权共鸣体 → 发现共鸣 → 慢信连接。
- 首登、演示数据和真实数据模式清楚分离。
- 每一步都有空、加载、成功、部分完成、失败、离线和恢复状态。

### B2. 信息架构与动态交互

- 将 `AuroraApp.tsx` 拆为路由、领域状态/hooks、页面与组件，不再把所有业务编排堆在单文件。

### B3. 视觉、双语、无障碍和性能

### B4. PWA / Capacitor 完整体验

## 7. 提交、证据与 PR 规范

每条轨道：

1. 开始前记录分支、base SHA、运行环境和基线测试。
2. 每完成一个可独立恢复的纵向 checkpoint 就提交；提交信息表达用户/系统结果，如 `feat(ai): improve adaptive dual-kernel continuity`。
3. 每次提交前跑相关最小门禁；每个阶段跑域内套件；PR 前跑轨道要求的全量门禁。
4. 禁止把构建缓存、密钥、本机配置、大模型原始敏感响应或无关工作树修改提交。
5. 每个证据目录至少包含：基线、命令、环境、输入身份/脱敏说明、结果、失败样例、限制、commit SHA 和复现步骤。
6. 状态文件必须区分 `DONE`、`PARTIAL`、`BLOCKED_HUMAN`、`FAILED`；不得把人工门禁写成 PASS。
7. PR 描述必须包含：用户可感知变化、架构变化、迁移/回滚、验证结果、已知差距、跨轨请求、Demo 步骤和 evidence 索引。
8. 禁止 force push、改写共享历史或自动公开部署。推送和创建 PR 可以由有权限的组员执行。

## 8. 测试节奏

| 时机 | 必须运行 |
|---|---|
| 小修改 | 受影响模块的单元/组件测试、类型检查或快速评测 |
| 纵向 checkpoint | 该旅程的集成测试/Playwright/AI replay + Secret scan |
| 数据迁移/契约变化 | PostgreSQL + Flyway、OpenAPI diff、兼容/回滚验证 |
| PR 前 Track A | Maven 全量、AI Lab 全量、真实 Provider 冻结集、Secret scan、diff check |
| PR 前 Track B | Vitest、TypeScript/Vite、Playwright 完整旅程、a11y/visual/perf、Kustomize/Compose smoke、Secret scan |
| 合并后 | Maven + 前端 + AI Lab + local-complete E2E + Kustomize + Secret + 非作者 Demo |

随机模型输出不应造成普通 CI 抖动。离线回归使用冻结响应/判定规则；真实 Provider 评测单独记录模型版本、温度、运行时间、失败与成本。

## 9. 合并顺序与最终收尾

1. 两条 PR 都从相同基线完成，不互相 merge 未完成分支。
2. 推荐先审查并合并 Track A，固定最终服务端契约和迁移。
3. Track B 在最终审查前基于合并后的 `main` 刷新 typed client、移除临时 fixture/adapter，完成接口接线；若用户选择直接合并两个 PR，则由集成者完成同样步骤。
4. 集成者统一更新 `complete-product-acceptance.yml`、`single-session-state.yml` 和本文的状态，不接受两轨对全局账本的乐观修改。
5. 合并后只允许剩下：跨轨接线、小型体验打磨、非作者/专家/账号人工门禁和真实发布动作。若仍缺一个完整核心能力域，应退回对应 PR，不应把大块未完成工作藏进“收尾”。

## 10. 轨道终止条件

Coding Agent 不应因一次提交、一次测试绿色或上下文压缩而停止。只有下列情况可以结束轨道会话：

- 轨道任务书中的机器可执行验收全部满足，证据和 PR 已准备好；或
- 连续尝试并记录替代方案后，只剩不可伪造的人工权限/设备/专业评审门禁，且所有不依赖门禁的工作已经完成；或
- 出现会破坏数据、公开系统或改变产品核心承诺的重大决策，需要用户授权。

普通实现困难、测试失败、Provider 波动、方案需要重构都不是停止理由，应继续诊断和迭代。
