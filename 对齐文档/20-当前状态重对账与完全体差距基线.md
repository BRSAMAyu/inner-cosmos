## 1. 结论

Inner Cosmos 已经不是 Java 作业，也不再只是一个有若干页面的 AI 原型。当前仓库已经具备成熟产品工程的主要地基：Java 21 / Spring Boot 3.5、PostgreSQL 16 + pgvector + Flyway、Redis、多运行角色、可靠事件、React 19、Capacitor、Kubernetes、供应链门禁、安全加固，以及 Aurora、记忆画像、共鸣体、心理 Skill 的大量真实纵向切片。

真正的瓶颈已经从“能否实现”转为四个更难的问题：

1. **能力是否真的比通用聊天产品更好**：双核、长期记忆、Self、共鸣体等代码存在，但尚未形成稳定、可重复、可被盲评认可的体验优势。
2. **庞大能力是否被收拢成一个自然产品**：五个空间已有差异化视觉与功能，但信息架构仍偏工作台，多个长页面把内部机制平铺给用户，核心旅程缺少导演式节奏。
3. **数据全链路是否真的可控**：来源、同意、纠正和部分遗忘已经存在；缓存、派生数据、异步队列、分析、通知和备份仍没有统一的撤回传播证明。
4. **是否能由非作者稳定演示和接手**：工程测试很强，但“干净机器一键启动—真实 Provider—完整旅程—故障恢复—8 至 12 分钟演示”仍缺一次非作者闭环。

因此，接下来不应继续按极小技术切片逐项堆叠，而应以两条并行、端到端的产品收敛线推进：A 负责“智能与数据真相”，B 负责“完整产品体验与交付”。

## 4. 仍未解决的完整差距

以下不是普通 backlog，而是“如果不解决，就无法成为惊艳产品”的差距树。

### 4.1 P0：AI 优势尚未成立

当前真实 DeepSeek 对比中，确定性词法评分出现单核 `7/11`、双核 `6/11`。这不能证明双核更差，也不能证明双核更好；它证明**架构复杂度尚未转化为稳定体验收益**。

必须完成：

- 建立覆盖倾诉、犹豫、矛盾、被打断、长期回访、边界拒绝、危机降级、共鸣体拟真等真实轨迹的开发集与冻结评测集。
- 把单核、双核、是否检索记忆、是否使用 Self/关系状态做成可消融配置。
- 优化 planner / responder / critic 的职责、共享状态、何时启用双核、何时停止思考；避免“每次都多跑一个模型”却没有质量收益。
- 将多消息、停顿、打断、主动回访做成可观察的对话导演，而不是后端字段。
- 对每个输出记录模型、prompt 版本、使用的记忆/画像/Skill、延迟、token、成本、风险判定与用户反馈。
- 以盲评胜率、严重错误率、连续性、主动性适当度、延迟和成本为门槛；没有效果就继续迭代系统，而不是只把评测标记为完成。

完成标准：在冻结样例上相对强单核基线达到预先登记的胜率/非劣门槛；无 P0 事实幻觉、越权记忆或不适当主动唤醒；再由非作者盲评确认。

### 4.2 P0：共鸣体“治理强、拟真弱”

当前 Genome 和授权治理很完整，但运行时仍缺少真正的候选特征生成、上下文编译、critic/reranker、长 Prompt 对照与纵向拟真证明。匹配仍主要是主题族/词法特征 + MMR，不等于基于高维用户画像的匹配。

必须完成：

- 把事实、语言风格、节奏、习惯、价值、边界、关系倾向、矛盾和时间演化编译成可追溯 Genome，而非简单拼接 Prompt。
- 为每轮会话动态选择最小必要特征，控制矛盾、时效和信息密度。
- 引入生成前 planner、生成后事实/风格/边界 critic，并通过缓存和分层模型控制延迟。
- 实现画像 embedding、多策略召回（相似、互补、探索、反回音室）和可解释 rerank。
- 将拟真与隐私分开评测：高拟真不等于泄露 P0；所有语料必须来自授权数据并能撤回。
- 用长 Prompt、静态卡片、动态 Genome 三组盲测证明动态方案的价值。

### 4.3 P0：完整旅程仍像“能力陈列”，不是“自然产品”

当前 `AuroraApp.tsx` 约 1292 行，集中大量状态、异步函数和 API 编排；五空间由查询参数与条件面板驱动。共鸣页和内宇宙页把 Genome、授权、沙盒、边界、广场、对话、慢信或纠正、星空、心理 Skill 堆在同一长页面。用户看到了很多能力，却不一定知道下一步应该做什么。

必须完成：

- 把核心旅程确定为：首次被理解 → 得到可纠正的记忆/画像 → Aurora 在正确时间回来 → 看见自我变化 → 授权生成共鸣体 → 遇到有意义的人 → 慢信连接。
- 将内部机制渐进披露：普通用户看到结果与理由，高级用户再展开来源、Genome、策略和授权。
- 将 AppShell 拆为稳定路由、领域 hooks/store、页面和可复用状态组件；避免继续扩大单文件。
- 为首登、空数据、等待 AI、失败、撤回、冲突、离线恢复、权限拒绝写完整文案和恢复动作。
- 把“多消息、停顿、主动回访、记忆命中、共鸣解释”做成现场一眼可见的动态体验。
- 完成全局英文和中文资源、en-SG 时区/资源、WCAG 2.2 AA 目标、键盘/读屏、reduced motion、移动性能与视觉回归。

### 4.4 P0：数据权利还没有覆盖全部派生面

现有 FORGET 路径覆盖大量记忆、画像、关系、向量、授权和共鸣体数据，但尚未用统一机制证明缓存、prompt snapshot、异步队列、分析指标、通知/推送、导出副本、日志与备份的撤回传播。

必须完成：

- 建立“数据产品 / 派生物登记表”，记录来源、目的、同意、保留期、处理器和删除策略。
- 纠正/撤回通过可靠事件传播；各消费者幂等确认，失败进入可重放队列。
- 在线数据立即不可用；备份按声明的保留周期到期清除，不伪造物理即时擦除。
- 建立数据导出、删除、共鸣体撤回、画像纠正的端到端验收与审计收据。
- 日志、metrics 和 AI trace 禁止包含 P0 文本或可逆敏感数据。

### 4.5 P1：心理 Skill 还未形成可信、好用的产品能力

Skill runtime、manifest、同意和风险路径已经存在，但仍需：至少三项真正有用户价值的非诊断 Skill；权威来源、版本、适用/不适用范围；中英双语内容；可解释结果；反例和危机降级；专业人员审阅。Skill 应由 Aurora 在合适时机建议，也允许用户显式调用，但不得在用户脆弱时强迫量表或冒充诊疗。

### 4.6 P1：PWA、移动端、国际化与无障碍尚未闭环

- 仓库尚未形成可验证的 service worker / 离线缓存 / 安装升级策略，不能仅因 React + Capacitor 就宣称 PWA 完成。
- i18n 只覆盖部分组件，仍有大量硬编码中文。
- 真实 Android/iOS 设备、APNs/FCM、签名、生产域和 iOS Xcode 构建是人工/设备门禁。
- 缺全局 axe、键盘、读屏、对比度、字体缩放、低性能移动设备和视觉回归证据。

### 4.7 P1：可观测、成本与故障降级不完整

- Prometheus/Grafana 已有 JVM/HTTP 信号，但缺跨角色 OpenTelemetry trace、AI token/成本/质量、检索命中、主动唤醒、撤回传播和用户体验指标。
- `/actuator/metrics/**` 与 `/actuator/prometheus` 当前被 Web Security 公开放行；生产发布前需使用独立管理端口、网络策略、mTLS/授权或其组合收紧，同时保持探针和抓取可用。
- 缺 Provider 超时/限流/错误注入、模型降级、自动 canary/rollback、异地备份和非作者 runbook 演练。

### 4.8 P1：架构边界和 API 演化仍需治理

- “模块化单体”是正确方向，但 Spring Modulith/ArchUnit 的可执行边界尚未建立；当前按 controller/service/entity 横向分层容易形成跨领域依赖。
- OpenAPI 和 typed client 已有基础，但前端仍混合生成类型与手写编排。
- 接下来应先冻结领域契约、用架构测试阻止新增越界，再按热点渐进拆包；不需要为了指标一次性重写所有 Java 包。

### 4.9 P2：商业新加坡与真正生产尚未完成

- Academy EKS 位于 `us-east-1`，不能当作新加坡生产证明。
- 尚无可复现的 Terraform 新加坡基础设施、最小权限工作负载身份、托管数据库/缓存/对象存储、DR 与成本预算闭环。
- PDPA、跨境 LLM、商店、HSA 边界、主体、税务和支付均需要当前专业意见与所有者签字。
- 这些不阻止本地完全体 Demo，但阻止“已在新加坡公开商用”的表述。

### 4.10 容易被忽略的产品缺口

- 缺一套明确的北极星指标和事件字典：用户是否感到被理解、是否愿意纠正、主动回访是否有帮助、共鸣是否产生真实连接。
- 缺内容与运营系统：引导示例、空状态、通知节奏、反骚扰、举报闭环、实验开关、模型版本回滚。
- 缺成本预算：双核、embedding、critic 和主动唤醒在不同体验等级下的预算与缓存策略。
- 缺演示数据的“可重复故事”：现场不能依赖随机模型恰好表现好，应有真实 Provider 的受控剧本、预热数据、失败备份路径和明确的现场切换方式。
- 缺非作者可理解的架构/产品讲述：优秀技术必须被映射为用户价值，而非展示一串组件名称。

## 5. 完全体的可验证效果

完成后，项目至少应让非作者在不听开发者解释的情况下感受到：

1. Aurora 不只是回答一句话，而是会思考、分多条说、在被打断时调整，并在合理时间记得回来。
2. 它使用的是“我的”可纠正长期理解；用户能看见来源、纠正错误并撤回影响。
3. 共鸣体像一个授权生成的动态人物，而不是资料卡或换皮 Prompt；匹配理由既惊喜又可解释。
4. 复杂技术默认隐藏在自然交互之后，需要时才可展开审计。
5. 现有 `local-complete` Compose/PowerShell 资产可以从干净环境实际启动完整系统并运行真实 Provider；离线/Provider 故障不会让产品无响应或伪装成功。
6. 同一镜像可在 Academy EKS 展示 Kubernetes 的伸缩、恢复和观测；不把受限课程云伪装成商业生产。
