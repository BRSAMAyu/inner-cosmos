================================================================================
EXPAND-CONTRACT SCHEMA COMPATIBILITY (the other half of this gate)
================================================================================
The canary above ran two app revisions CONCURRENTLY (stable 64c697ddc7 + canary 677fcc9c96 both
Running and Ready at the same time during steps 1-4) - which is exactly the situation
expand-contract schema compatibility exists to make safe against a SHARED database. The contract:
- migrations are ADDITIVE only (new nullable columns / new tables), never drop/rename/retype during a
  canary window, so the old stable revision and the new canary revision can both read+write the same
  schema. The Flyway set under db/migration/postgresql + the init-container schema-version gate
  (scripts/academy/validate-schema-version.ps1, CN-CREDIBILITY-001 6.1-6) is the mechanism that
  bounds what a canary revision's schema may be.
- HONEST BOUNDARY: this local proof used per-pod H2 (emptyDir), so stable and canary did NOT share a
  single DB here - it proves the canary/rollback MECHANICS, not live schema-compat against shared
  Postgres. The deeper exact-match -> version-compatible-range gate redesign (CN-CREDIBILITY-001
  6.1-6) remains deferred; verifying two revisions against the real shared Postgres in inner-cosmos-w3
  is real remaining work (out of scope: it would disturb the live hero-gate workload).

================================================================================
ACADEMY FALLBACK (CRD controller may not run under limited LabRole)
================================================================================
If Academy EKS LabRole cannot run the Argo Rollouts controller / Rollouts CRD, the same progressive-
delivery guarantee is obtained with a SCRIPTED HTTPRoute-weighted canary using only Gateway API
(which Academy DOES support - deploy/k8s/overlays/academy-eks/http-route.yml):
- two Deployments (stable + canary) behind one Gateway/HTTPRoute; a Job or human bumps HTTPRoute
  weights stable90/canary10 -> 50/50 -> 0/100;
- a PromQL/health gate (the same aurora-canary-health query against the running Prometheus, or a
  kube_pod_status_ready check) MUST pass before each weight bump;
- on degradation the gate flips the route back to 0% canary (stable 100%) - the manual/scripted
  equivalent of progressDeadlineAbort. No CRD controller, no webhooks, just HTTPRoute + a gate.
This is documented as the Academy path in deploy/k8s/extensions/rollouts/api-rollout.yaml.
