> Item: `docs/audit/2026-07-23-gemini-master-audit-reconciliation.md` R2 row 2.5 (PARTIAL/P2).
> Contract: "先构造代表性数据并 `EXPLAIN (ANALYZE, BUFFERS)`；达到阈值后以新 Flyway 版本(不是已占用的
> V21)创建按模型/有效状态设计的 HNSW，并测 recall、P95、build/memory 成本和回退。" Forbidden:
> do not create the index speculatively "because it seems like good practice."
>
> Environment: Docker Desktop was available in this session (`docker info` succeeded), so real
> `pgvector/pgvector:0.8.1-pg16` Testcontainers benchmarking was possible (same image already used

## Decision: DO NOT create the ANN index in this pass

Real load evidence *does* show the growth-risk the audit flagged is genuine (see "Latency and
build-cost evidence" below) — but a second, independently-required piece of evidence, **recall**,
could not be honestly produced with the data available in this sandboxed environment, and the one
number I did measure was alarming enough that shipping blind to it would be reckless for a
privacy/quality-sensitive personal-memory feature. Per the task contract, "cannot produce genuine
justifying evidence → do not create it, write up why" is the correct, complete outcome here, not a
partial/unfinished one.

## What the query shape actually is (checked before benchmarking anything)

- `MemoryEmbeddingIndexServiceImpl.postgresScores` (`src/main/java/com/innercosmos/service/impl/MemoryEmbeddingIndexServiceImpl.java`):
  `WHERE e.user_id=? AND e.model_name=? AND e.model_version=? AND e.task_scope='GENERAL' AND
  e.status='ACTIVE' ... ORDER BY e.embedding_vector <=> ?::vector LIMIT 100`. This is a real
  top-K ANN candidate: filtered first by the existing `idx_memory_embedding_user_model(user_id,
  model_name, model_version, status)` B-tree, then an exact-distance sort over whatever survives
  that filter (in practice: *one user's own* embedding rows — this table's per-user row count is
  the real cost driver, not total table size across all users).
- `CapsuleEmbeddingIndexServiceImpl.postgresScores` (`src/main/java/com/innercosmos/service/impl/CapsuleEmbeddingIndexServiceImpl.java`):
  `WHERE (capsule_id=? AND content_hash=?) OR (...) ... ` — **no `ORDER BY`/`LIMIT` at all**. It
  computes an exact cosine score only for the explicit, already-narrowed candidate list the caller
  passes in (bounded by however many public capsules survived upstream safety/consent/block
  filters). There is no global nearest-neighbor search happening here, so an ANN index has **no
  applicable query to accelerate** on `tb_capsule_embedding` — adding one there would be exactly the
  forbidden "seems like good practice" speculative index. Confirmed by reading the method: excluded
  from consideration entirely.

So the only candidate for 2.5 is `tb_memory_embedding.embedding_vector`, and the real cost variable
is **one user's own embedding row count**, not global corpus size.

## Latency and build-cost evidence (real, Testcontainers, current schema)

Seeded `tb_memory_embedding` + `tb_memory_card` rows for a single user (matching the real
`source_version`/`version_no` join semantics), ran `ANALYZE`, then ran the exact
`postgresScores` SQL via `EXPLAIN (ANALYZE, BUFFERS)` and 10x repeated timed execution
(single connection, no network hop — a lower bound on real request cost):

| per-user embedding rows | avg exec time (10 runs, no index) | scan shape |
|---|---|---|
| 5,000  | ~27–38 ms  | Seq Scan over the user's rows, Sort, Limit 100 |
| 20,000 | ~135–188 ms | same shape, cost grows with row count |
| 100,000 | ~275–387 ms | same shape (partly Parallel Seq Scan) |

With an `HNSW (vector_cosine_ops)` index added at 100,000 rows:
- Build time: **~28–40 seconds** for 100k rows (background-job-safe; the existing
  `MemoryEmbeddingRebuildJob` already runs on a poll/delay cadence, not inline with user requests).
- Query time: **~2.1–2.7 ms avg** (vs. ~275–387 ms exact) — roughly **100–150x** faster on this
  synthetic dataset.

**Is 20k–100k per-user rows a plausible real scale, or an unrealistic edge case?** `tb_memory_card`
rows are created per settled dialog session (`uk_memory_card_user_session` — see
`V1__application_baseline.sql`), so per-user row count tracks roughly 1:1 with how many Aurora
conversation sessions a person has ever had, *plus* one additional embedding row per edited
version (memory-card edits bump `version_no`/`source_version`, and old-version embedding rows are
never marked non-ACTIVE or deleted — `MemoryEmbeddingIndexServiceImpl.indexIfMissing` has no
supersede step, unlike its capsule-embedding sibling). A genuinely engaged long-term user of a
"self-understanding" product having several conversations a day for several years, with normal
editing/revision of memory cards over time, is not an extreme edge case at the 20,000–100,000 row
mark — it is a plausible multi-year power-user outcome. **This part of the audit's PARTIAL verdict
is confirmed correct by real measurement: the growth-risk is genuine, not speculative.**

## Why this is still not enough to ship the index: recall could not be honestly measured

The contract explicitly requires recall evidence, not just latency. I attempted this with 5 random
probe vectors, comparing the exact top-100 (pre-index) against the same query's top-100 once the
HNSW index existed, at the 100,000-row scale, at both default (`ef_search=40`) and a much higher
(`ef_search=400`, 10x) search-list size:

```
ef_search=40  mean recall@100 over 5 random probes = 0.07   (first run)
ef_search=40  mean recall@100 over 5 random probes = 0.0    (second run, tighter measurement)
ef_search=400 mean recall@100 over 5 random probes = 0.0
```

Recall did **not** improve at all between `ef_search=40` and `ef_search=400` — ruling out "just
needs a higher search-list tuning knob" as the explanation. The real explanation is more
fundamental and makes the measurement itself uninformative: the seeded vectors were **uniformly
random floats in [-1, 1]^1536**, generated purely for realistic *volume/latency* testing, not as a
stand-in for real embedding semantics. In 1536 dimensions, uniformly random vectors concentrate
extremely tightly around cosine similarity ≈ 0 with almost no separation between the "top 100" and
the remaining 99,900 — there is no real nearest-neighbor manifold structure for either an exact
scan or an HNSW graph to agree on, so "top-100 identity" is essentially arbitrary noise, unstable
under even tiny floating-point/algorithmic differences. A 0% (or 7%) recall number here does not
mean "this HNSW index has bad recall on real data" any more than a 100% number would have meant
"this HNSW index has great recall on real data" — the test is simply not measuring what it needs
to measure. Recall can only be validated honestly against embeddings that actually carry semantic
structure, i.e. real output from `MemoryEmbeddingClient`'s real provider (GLM/MiniMax/etc.) at a
representative volume — which this sandboxed benchmarking session has no way to produce (no bulk
real-provider embedding budget/quota available here, and it would be irresponsible to spend real
API quota/money on a throwaway benchmark script rather than the product's actual users).

## Conclusion

- **Do not** add an ANN index migration in this pass. No `V22` (or later) migration was created;
  the next free Flyway version remains available for a future, properly-evidenced attempt.
- **Do not** add any index to `tb_capsule_embedding` — its query shape has no ANN search to
  accelerate at all.
- Record as a genuine, tracked follow-up (not silently dropped): once real-provider embeddings are
  available at a representative per-user volume (either from real production analytics data, or a
  deliberately budgeted real-provider benchmark run), re-run this exact latency+build-cost+recall
  methodology against `tb_memory_embedding` before creating an HNSW index. The latency risk is real
  and will keep growing with product usage; the recall gate is the only thing currently blocking
  action, and it is an honest environment/data limitation, not a decision to defer indefinitely.
- No code or schema change accompanies this item. This document is the closing receipt.
