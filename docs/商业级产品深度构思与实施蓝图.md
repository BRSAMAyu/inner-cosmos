# Inner Cosmos 内宇宙：商业级产品深度构思与实施蓝图

版本：v1.0  
日期：2026-05-26  
用途：交给后续 Coding Agent 继续实现。本文档是产品、内容、体验、架构与任务拆解的综合交接书。

## 0. 文档定位

Inner Cosmos 现在已经不是空项目，也不是单纯的 Java Web 骨架。当前仓库已经具备 Spring Boot 主架构、Mock-first LLM、ASR Mock、Aurora 对话、P1 记忆沉淀、情感重力、记忆星图、共鸣体、星海广场、慢信、基础安全、AI 日志、管理后台和暖雾 Morandi UI 的第一版。

但距离“可商业化推广的完整产品”仍有明显差距：目前大部分模块已经打通，但内容深度、长期记忆能力、真实社交质感、可控隐私、管理运营、移动端体验、模型可靠性和产品叙事仍需要大幅扩展。

本文档的目标不是重复已有愿景，而是把愿景转化为可执行的工程蓝图：

- 明确产品真正解决的用户问题。
- 明确商业级完成版的核心体验。
- 明确当前已经实现什么、还缺什么。
- 明确每个模块应该继续实现到什么程度。
- 明确 API、数据、UI、内容、测试与验收标准。
- 为后续 Coding Agent 提供足够具体的任务拆解。

后续 Agent 执行时，不应只做页面装饰，也不应只补 CRUD。必须围绕“用户核心旅程”和“长期个人内在世界系统”继续推进。

### 1.3 当前实现的核心矛盾

当前项目已经“能跑完整闭环”，但还没完全“让用户愿意留下来长期使用”。主要差距如下：

1. Aurora 仍偏模板式，尚未形成真正有记忆、有节律、有个性化的长期陪伴 Agent。
2. 记忆系统能抽取结构，但缺少长期主题、关系提及、事件卡、每周/月度叙事。
3. 星图已有视觉雏形，但还不是用户长期自我理解的核心操作台。
4. 共鸣体有创建和边界字段，但缺少真实脱敏预览、授权内容审查、可编辑授权包和回声衰减体验。
5. 慢信有状态流，但缺少回信链、投递延迟解释、信件安全过滤、收发信仪式感。
6. 安全边界已有基础，但不够系统：缺少分级响应、节律保护闭环、社交安全和后台审核链路。
7. 管理后台只展示基础数据，缺少运营、审核、模型配置、AI 日志筛选和风险事件处理能力。
8. 前端缺少真正的组件化组织，后续多端复用还不充分。
9. 真实模型接入还只是适配层，缺少 JSON 结构化输出、重试、降级、成本控制和 Prompt 版本管理。
10. 产品内容体系不够丰富：种子共鸣体、Aurora 模式、今日记录文案、星图解释、慢信引导都需要大量高质量内容。

## 2. 用户洞察与产品定位

### 2.1 核心用户不是“想聊天的人”

Inner Cosmos 的用户不是单纯来找 AI 闲聊的人，也不是只想写日记的人。真正核心用户是：

- 有表达欲，但不知道怎么把自己说清楚的人。
- 想写日记，但觉得日记像任务、难以坚持的人。
- 在压力、自责、拖延、关系拉扯中反复循环的人。
- 想被理解，但不想立刻暴露给现实社交的人。
- 对普通社交软件感到疲惫，但仍渴望真实连接的人。
- 希望长期看见自己变化、整理成长叙事的人。

他们真正需要的不是“AI 给答案”，而是：

- 先被接住。
- 再被整理。
- 再看见重复模式。
- 最后在合适时走向行动或连接。

产品必须避免变成一个“高级安慰话术生成器”。用户真正愿意付费或长期使用的价值来自：系统长期记得他、帮他看见模式、减少混乱，并在不冒犯边界的情况下带来真实共鸣。

### 2.2 用户核心任务

用户打开 Inner Cosmos 时，最常见的任务应该不是“管理数据”，而是以下几类：

1. 我现在很乱，想先倒出来。
2. 我今天过得不太好，想被慢慢听见。
3. 我想知道自己为什么总是重复同一种痛苦。
4. 我想把今天整理成一张我能看懂的记录。
5. 我想看看最近真正牵动我的主题是什么。
6. 我想把某个已经想清楚的片段，安全地放到星海里。
7. 我想遇到一个和我有相似精神频率的人，但不想被即时社交打扰。
8. 我想收到一封认真而不冒犯的信。

所有功能都应该服务这些任务，而不是服务内部数据模型。

### 2.3 产品一句话定位

商业推广时，Inner Cosmos 可以定义为：

> 一个由 AI 陪你把日常混乱整理成长期内在星图，并通过授权数字回声与慢信件连接真实共鸣的个人精神空间。

更短版本：

> 让你把自己说清楚，也让相似的人温柔相遇。

### 2.4 核心差异化

用户为什么不用通用大模型？

因为 Inner Cosmos 不只回复，还会沉淀：

- 通用大模型是一次性对话。
- Inner Cosmos 是长期记忆世界。

用户为什么不用日记 App？

因为 Inner Cosmos 不要求用户已经清晰：

- 日记 App 让用户自己整理。
- Aurora 陪用户从混乱走向清晰。

用户为什么不用陌生人社交？

因为 Inner Cosmos 不直接暴露真人：

- 陌生人社交先看人。
- Inner Cosmos 先遇见脱敏精神回声。

用户为什么不用心理测评？

因为 Inner Cosmos 不把用户标签化：

- 测评给结论。
- Inner Cosmos 给长期可回看的自我理解。

## 3. 商业级完成版的产品支柱

商业级版本必须围绕五个支柱建设。

### 3.1 支柱一：Aurora 是入口，不是普通聊天框

Aurora 要成为产品主体验。它应该具备：

- 多模式对话：今日倾诉、思维整理、睡前复盘、苏格拉底追问、行动拆解、关系复盘。
- 语气个性化：温柔安静、理性清晰、朋友式直接、哲学式追问、行动导向。
- 会话节律：识别过度倾诉、长对话疲劳、深夜低能量状态，主动建议沉淀。
- 语音倾诉：录音、转写、编辑、发送、语音元数据。
- 长期记忆调用：能透明引用高重力记忆，而不是每次从零开始。
- 结构化输出：每次对话都能沉淀成用户可理解的资产。

### 3.2 支柱二：P1 记忆系统是技术壁垒

真正壁垒不是模型，而是“用户内在世界模型”。它包括：

- MemoryCard：关键记忆。
- EventCard：具体事件。
- ThoughtFragment：事实、感受、担心、需求、信念、行动。
- EmotionTrace：情绪天气和趋势。
- TodoItem：从倾诉中转化的轻行动。
- RelationMention：关系线索。
- MemoryTheme：长期主题。
- NarrativeArc：周/月/季度成长叙事。

后续实现要把 P1 从“几张表”升级为“长期自我建模层”。

### 3.3 支柱三：记忆星图是可视化，不是装饰

星图必须让用户理解：

- 哪些记忆现在最亮。
- 哪些主题反复出现。
- 哪些情绪正在变轻或变重。
- 哪些关系影响自己。
- 哪些行动已被推进。
- 哪些记忆适合被脱敏放入星海。

星图应该逐步升级为一个可操作空间，而不是只展示点。

### 3.4 支柱四：共鸣体是社交前的安全缓冲

共鸣体不是 AI 分身，也不是用户替身。它是：

- 用户授权后的抽象精神片段。
- 有边界、有衰减、有关闭权的数字回声。
- 让陌生人先遇见主题，而不是直接遇见真人。
- 深聊后引导慢信，而不是无限陪聊。

共鸣体必须有强隐私结构：P0 原文永不进入共鸣体，P1 必须脱敏、授权、抽象后才能进入 P2。

### 3.5 支柱五：慢信件是高质量连接机制

慢信不是消息系统。它应该是一种郑重连接机制：

- 有写信引导。
- 有投递延迟。
- 有送达仪式。
- 有回信链。
- 有拒绝和拉黑。
- 有信件安全过滤。
- 不鼓励即时纠缠。

商业级版本要让“收到一封慢信”成为产品中的情感高点。

## 4. 总体用户旅程设计

### 4.1 首次使用旅程

目标：让用户在 3 分钟内完成第一次低压表达。

流程：

1. 首页说明产品不是诊断、不是任务管理、不是普通社交。
2. 用户进入 Dashboard，看到“今天可以从一句话开始”。
3. 用户进入 Aurora。
4. Aurora 只问一个低压力问题：
   - “今天你不用整理好再开口。可以先从最散的一句话开始。”
5. 用户输入一句话或使用语音。
6. Aurora 先承接，再澄清，不立刻建议。
7. 用户点击“沉淀今日记录”。
8. 系统生成 Daily Record。
9. 用户看到主题、情绪、认知碎片和一个轻行动。
10. 系统引导用户进入记忆星图。

验收标准：

- 用户不需要看说明也知道第一步该做什么。
- 第一次对话后必须产生“我被整理清楚了”的感受。

### 4.2 七日留存旅程

目标：让用户感到系统开始“记得我”。

第 1 天：完成一次 Aurora 倾诉，生成今日记录。  
第 2 天：Dashboard 显示昨天高重力主题，温柔提醒可继续观察。  
第 3 天：用户再次提到类似主题，系统识别重复。  
第 4 天：记忆星图出现一个更亮的主题星。  
第 5 天：系统生成“本周小结草稿”。  
第 6 天：用户完成一个从对话提取的小待办。  
第 7 天：系统展示“这周你反复提到的不是懒，而是压力下的自我评价”。  

验收标准：

- 系统不能像每天都是新用户。
- 至少有一次透明记忆引用。
- 至少有一次长期主题聚合。

### 4.3 共鸣连接旅程

目标：让用户安全地从私人表达走向精神连接。

流程：

1. 用户在星图中点击一颗高重力星。
2. 系统解释为什么这颗星较亮。
3. 系统提供“脱敏预览”。
4. 用户确认哪些信息可进入共鸣体。
5. 系统生成共鸣体。
6. 共鸣体进入星海，但不暴露原始日记。
7. 另一个用户与共鸣体有限对话。
8. 共鸣体在适当时引导慢信。
9. 用户收到慢信，可读、回、拒绝、拉黑。

验收标准：

- 用户能清楚看到“我授权了什么”。
- 其他用户不能通过共鸣体还原 P0 原文。
- 慢信不是即时聊天替代品，而是郑重连接。

## 5. 模块级深度设计

## 5.1 Aurora Agent

### 当前状态

已实现：

- `/api/aurora/message`
- `/api/aurora/stream`
- Aurora Chat 页面。
- 四种模式的前端切换。
- Mock LLM 回复。
- 对话存储。
- AI 日志。
- 语音 Mock 转写入口。
- 对话结束后跳转 DailyRecord。

不足：

- 模式主要是前端标签，Prompt 差异不够深。
- Aurora 个性化尚未接 UserProfile。
- 没有真正的上下文滑窗摘要。
- 没有透明记忆引用。
- 多段回复主要前端拆句，后端语义结构不足。
- 节律保护不够强。

### 目标状态

Aurora 应成为“对话式自我整理 Agent”。每次回复要做到：

1. 承接用户表达。
2. 识别表达层次。
3. 提出一个低压力澄清问题。
4. 必要时回忆相关高重力记忆。
5. 控制节律，避免无限倾诉。
6. 为对话结束后的结构化沉淀预埋信息。

### 后端任务

1. 新增或扩展 `UserProfile`：
   - `auroraTone`
   - `preferredInputType`
   - `reflectionDepth`
   - `allowMemoryRecall`
   - `quietHoursStart`
   - `quietHoursEnd`

2. 新增 `ConversationMode` 枚举：
   - `DAILY_TALK`
   - `THOUGHT_CLARIFY`
   - `SLEEP_REVIEW`
   - `SOCRATIC`
   - `ACTION_SPLIT`
   - `RELATION_REVIEW`

3. `ChatRequest` 增加：
   - `mode`
   - `clientMessageId`
   - `timezone`
   - `localTimeLabel`

4. `AuroraAgentServiceImpl` 改造：
   - 读取用户偏好。
   - 读取最近 3 轮对话。
   - 读取 `DialogSession.summaryAnchor`。
   - 读取用户 top 3 high-gravity memory summaries。
   - 注入语音元数据。
   - 调用 `SafetyService`。
   - 调用 `RhythmGuardService`。
   - 返回 `AuroraReplyVO`，而不是单纯字符串。

建议 `AuroraReplyVO`：

```java
public class AuroraReplyVO {
    public List<String> messages;
    public String replyTone;
    public String detectedTheme;
    public String nextQuestion;
    public Boolean suggestSettle;
    public Boolean memoryReferenced;
    public List<Long> referencedMemoryIds;
}
```

5. 新增 `DialogSummaryService`：
   - 当消息数超过阈值时，生成或更新 `summaryAnchor`。
   - Mock 模式使用规则摘要。
   - Remote 模式要求 JSON 结构化摘要。

6. 新增 `RhythmGuardService`：
   - 单会话消息数阈值。
   - 单会话时长阈值。
   - 深夜时段温柔提示。
   - 高频发送提示。
   - 返回建议：继续、放慢、沉淀、转安全资源。

### 前端任务

Aurora 页面应升级为主产品页面：

- 对话模式改为带说明的 segmented control。
- 添加“对话能量条”或“今日节律”提示。
- 输入区支持：
  - 文本。
  - Mock 录音状态。
  - 转写中。
  - 可编辑转写。
  - 发送后清空。
- 回复气泡支持多段自然出现。
- 结束对话时显示沉淀步骤。
- 若 `suggestSettle=true`，结束按钮视觉更突出。

### 内容模板

每个模式至少准备 12 个 Mock 模板。

今日倾诉模板：

- “我听到这里有两层，一层是事情本身，一层是你对自己的评价。”
- “我们先不急着解决。你愿意先说说最刺痛你的那一刻吗？”
- “你说得很轻，但它好像并不轻。”

思维整理模板：

- “我先帮你把这团东西分成事实、担心、需求和下一步。”
- “这里有一个结论跳得很快，我们可以把它拆开。”

睡前复盘模板：

- “今晚不适合再把问题拉得更大。我们先把它收成一个可以睡前放下的形状。”
- “今天先到这里也可以，不需要在睡前把人生想明白。”

苏格拉底模板：

- “这个判断成立需要哪些证据？”
- “有没有一种更温柔但仍然真实的解释？”

### 验收

- 一次对话至少产生 2 条以上 Aurora 自然消息。
- 对话超过阈值时出现节律提示。
- 结束后 DailyRecord 有主题、碎片、情绪、待办。
- 测试覆盖文字输入、语音 Mock 输入、结束沉淀、节律提示。

## 5.2 P1 记忆系统

### 当前状态

已实现：

- MemoryCard
- ThoughtFragment
- EmotionTrace
- TodoItem
- DailyRecordVO
- MemoryService.extractFromSession
- 基础类型推断
- 基础情绪与待办抽取

不足：

- ThoughtFragment 类型不完整。
- 没有 EventCard。
- 没有 RelationMention。
- 没有 MemoryTheme。
- DailyRecord 没有独立表或版本。
- 没有用户修正与确认机制。
- 记忆重复识别较弱。

### 目标状态

P1 记忆系统要成为产品壁垒。它不只是存储对话摘要，而是形成可持续增长的“个人内在结构”。

必须继续补充：

1. EventCard：今天发生了什么。
2. RelationMention：谁或哪类关系牵动了用户。
3. MemoryTheme：反复出现的长期主题。
4. DailyRecord：每天一次结构化总结。
5. UserCorrection：用户对 AI 抽取结果的修改。
6. WeeklyReview：周总结。

- 对方回信。

## 8. 技术壁垒路线

### 8.1 第一层壁垒：结构化记忆

把自由表达转成结构化资产：

- 事件。
- 情绪。
- 认知。
- 需求。
- 行动。
- 关系。
- 主题。

### 8.2 第二层壁垒：情感重力

用非线性公式和长期触发记录，让系统知道什么真正有重量。

### 8.3 第三层壁垒：长期主题聚合

从 MemoryCard 中抽象出长期主题，并持续更新。

### 8.4 第四层壁垒：授权数字回声

通过 P0/P1/P2/P3 分层和 DataMasking，让用户可安全分享精神片段。

### 8.5 第五层壁垒：慢社交机制

用共鸣体和慢信件替代即时陌生人社交，降低打扰和风险。

### 8.6 第六层壁垒：节律与安全

不是无限陪聊，而是知道何时沉淀、暂停、转现实支持。

## 14. 给后续 Coding Agent 的直接指令

后续 Agent 应按以下原则执行：

1. 不要推翻现有架构，沿用 Spring Boot MVC、Mock-first、P0/P1/P2/P3、Adapter/Observer/State/Builder/Strategy。
2. 不要只写文案或只做 UI，应优先补齐数据与 API。
3. 每次新增前端页面，都必须有对应真实 API 或明确 Mock 数据源。
4. 每次新增 AI 输出，都必须沉淀到至少一个结构化资产或 AI 日志。
5. 每次处理社交相关功能，都必须考虑隐私、拉黑、举报、边界和状态流。
6. 每次改核心页面，都必须做桌面和移动截图 QA。
7. 没有真实 API Key 时，Mock 模式必须完整可跑。
8. 不要为了代码行数写空代码。代码量必须来自真实功能、内容、测试和文档。
9. UI 继续保持暖雾浅色 Morandi，不要回到深色冷峻宇宙风。
10. 所有功能最终都要服务这句话：帮助用户更清楚、更自由、更安全地理解自己，并在合适的时候走向真实连接。

## 15. 最后判断

Inner Cosmos 的核心潜力不在“AI 聊天”，而在以下组合：

```text
朋友式 Aurora
  + 结构化长期记忆
  + 情感重力星图
  + 授权数字回声
  + 慢信件连接
  + 安全边界与节律保护
```

这个组合一旦做深，就会形成区别于通用大模型、普通日记 App、心理测评和陌生人社交的独特产品。

后续实现的关键不是再问“要不要做这个功能”，而是把每个功能都做成能服务用户核心需求的真实体验：

- Aurora 要让用户说清楚。
- 记忆要让用户看见自己。
- 星图要让用户理解长期模式。
- 共鸣体要让用户安全分享一部分自己。
- 慢信要让真实连接变得郑重。
- 安全系统要让产品值得信任。

当用户说出下面这三句话时，项目才算真正接近商业级完成：

```text
我终于把自己说清楚了一点。
我看见自己这段时间其实在变化。
原来不是只有我这样。
```
