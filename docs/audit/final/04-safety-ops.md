# Expert 4 — Safety, Security, Reliability & Ops

> Senior authority synthesis of three sub-aspect inspections (AuthN/AuthZ & session security; Safety system & crisis handling; Privacy & data protection). Every P0/P1 claim below was re-verified by reading the cited files directly. Severities were corrected where the subagents under- or over-rated; overlapping findings were merged; and gaps the subagents missed were added.

---

## Executive Summary

**What's strong.** The codebase shows real safety engineering maturity where it matters most for crisis handling. The two-tier crisis architecture is principled: explicit `CrisisKeywordRule` blocks synchronously, while `DistressSignalDetector` feeds a synchronous LLM recheck that deliberately avoids medicalizing ordinary venting. A deterministic acute-crisis floor (`looksLikeGenuineCrisis`) is enforced symmetrically across the live-LLM, fallback, and exception paths — a permissive or failing model can never silently clear a genuine crisis, and this is regression-tested (F1/F9/F12). Safety runs FIRST on both POST and SSE paths before any token is persisted. Passwords are BCrypt(12). Session-fixation is rotated on login. Account deletion is genuinely broad (30+ tables). Ownership checks are correct on the highest-traffic letter/persona-chat/dialog paths.

**What's broken (headline risks).** Underneath that safety layer, the **access-control and privacy perimeter is effectively theatrical** for three independent, compounding reasons, any one of which is a P0:

1. **`X-User-Id` header trust** (`JwtAuthenticationFilter`) grants full Spring Security authentication — including `ROLE_ADMIN` — to any client that sets a request header. The frontend never sends it, so in practice the app relies on the session path; but every controller built tomorrow that reads `SecurityContext` instead of the session is wide open, and `health`'s `when_authorized` can be satisfied by the header.
2. **P0 IDOR on the Aurora chat core** (`AuroraChatController` → `AuroraAgentServiceImpl.replyRich` → `AgentContextAssembler.recentMessages`): a client-supplied `sessionId` is read and written with no ownership check, so an attacker loads a victim's most-sensitive P0 raw dialog into context and corrupts their conversation + downstream memory extraction.
3. **Unauthenticated P0/P1 data egress at the public plaza**: `GET /api/plaza/capsules` (permitAll) serializes full `EchoCapsule` entities — `personaPrompt`, `ownerContextNote`, `styleProfileJson`, `contextPreviewJson`, `authorizedMemoryIds` — to the open internet.

Layered on top: **no PII redaction anywhere on the dialog→LLM path** (raw P0 dialog, names, phones, ID-card-shaped content reaches MiniMax/DeepSeek/GLM and is stored plaintext in `tb_ai_interaction_log` indefinitely); **CSRF globally disabled with cookie-session auth and `SameSite` unset**; and — the safety-domain P0 that sits apart from security — the **crisis funnel ends on a page that renders ZERO real hotline numbers**, and the **synchronous safety recheck can block for tens of seconds-to-minutes** (failover × JSON-repair retry × 20–30s provider timeouts) before a person in distress sees anything.

**Net read.** This is a product whose safety *intent* and crisis *logic* are genuinely excellent, but whose *trust boundary, privacy enforcement, and crisis last-mile* are not yet production-grade. The privacy P0–P3 model described in the vision is real in the data model but is not enforced on the read/egress side in several load-bearing places. Everything below is fixable without rearchitecting the core loop.

## Strengths

- **Crisis design is correct and tested.** `SafetyServiceImpl.check` blocks `CRISIS_KEYWORD` synchronously (lines 73–82); `DistressSignalDetector` feeds `SafetyReviewService.recheckSync`; the acute-crisis floor (`looksLikeGenuineCrisis`) is applied on the live-LLM success path (SafetyReviewService 83–86), the fallback (138), and the catch block (112–113). Red-team suite F1/F6/F9/F12 pins the false-positive boundary (casual venting must NOT be medicalized; genuine crisis must block even when the LLM clears it).
- **Safety runs first, on every path, before persistence.** `replyRich` (line 150) and `stream` both gate synchronously before `saveUserMessage`; a blocked turn still persists the user message + a sober safe-message (`saveUserAndBlockedAurora`) so the record is captured.
- **Conservative-on-failure philosophy** is applied consistently — LLM failure, parse failure, and timeout all fall back to the deterministic heuristic, never to "allow."
- **Non-medicalization positioning** is reinforced in three places: `PromptBuilder.withSystemBoundary`, `SafetyServiceImpl.resources()`, and the global `StructuredAiService` instruction.
- **Password storage is correct**: BCrypt(12), `matches()` verification, seeded admin/demo go through `register()` (no plaintext seeding).
- **Session-fixation rotation on login** (old session invalidated + recreated before storing the id).
- **Ownership is correctly enforced on the highest-exposure letter/persona-chat/dialog paths** (sender||receiver on letters; visitor ownership on persona chat; `session.userId` on dialog finish/verifyOwnership/settle).
- **Account deletion is genuinely broad** (30+ tables including aiInteractionLog, voice transcriptions, capsules) and data export exists.
- **The authorized-memory boundary concept is present**: persona chat reads only `capsule.authorizedMemoryIds`, not all owner memory — the P2 design intent is there even though enforcement is incomplete (sec-priv-006).

## Vision Gaps — what this area SHOULD become

The findings above are defects. The gaps below are *missing capabilities* a complete, delightful, trustworthy Inner Cosmos should have. Several directly realize stated vision values (安全 / 透明 / 隐私分层 / 不制造依赖).

1. **A real crisis funnel with a live, region-aware hotline layer (not a phone-less page).** Beyond S4-006's numbers: locale/region detection, one-tap dial on mobile, a "someone to call with you" hand-off, and an opt-in "check on me in 24h" passive follow-up. This is the single most important safety-capability gap; the architecture is ready, only the last-mile content is missing.

2. **A wired "gentle session rhythm" guardian (S4-026).** `SessionDurationLimiter` exists but is dead code. For a product explicitly committed to *not creating emotional dependency* (a hard boundary in the prompt), there is no runtime mechanism enforcing breaks, soft-warns on long sessions, or nudges the user to ground themselves. This should be a first-class, visible feature — a warmth-forward "你已经陪自己很久了" nudge — not a hidden constant.

3. **A single-source, config/db-driven safety taxonomy.** Today there are three divergent keyword lists (`CrisisKeywordRule`, `AbuseKeywordRule`, `LetterSafetyFilterImpl`) plus `DistressSignalDetector`, and they have already drifted. The product should own ONE curated, versioned, admin-editable crisis/abuse vocabulary (with English + homophone + means-planning tiers) that every surface — chat, letter, persona chat, slow letter — routes through, with a shared audit trail.

4. **A PII-redaction pipeline as a first-class LLM-egress boundary (S4-005/S4-010).** Not a patch, but a named subsystem: redact-at-egress for every `LlmClient`, reversible-free, with per-PII-class tests and a "what was redacted" transparency log. This is what makes the P0–P3 privacy promise real against third-party provider egress.

5. **Active safety observability & escalation (S4-028).** A safety ops console: micrometer counters + alerts on HIGH/CRISIS spikes, a moderator triage queue (paginated, filtered — S4-032), and an async post-crisis follow-up path. Right now HIGH events are write-only. For a mental-health product, an unobserved HIGH event is an operational risk.

6. **A user-facing privacy cockpit (S4-042, S4-014, S4-029).** Granular consent toggles (memory/portrait/relations/location/voice), a per-turn "what Aurora knew this turn" transparency view, location consent + clear-stored-data, and a "download everything the system & providers have on me" export that matches the export-before-delete flow. This realizes the stated 透明 value and is currently absent from the UX.

7. **Real token-based auth + defense-in-depth role rules (S4-001, S4-021).** Signed JWT (or opaque server-side tokens) as the sole identity source; framework-level `hasRole` rules so admin isolation cannot regress; a mandatory `OwnershipGuard` helper/aspect every path-variable-id controller routes through (S4-016), so IDORs are impossible to ship by omission. This is the trust-boundary foundation everything else rests on.

8. **A complete credential & account lifecycle (S4-033, S4-031).** Change-password, forgot/reset via one-time token, 2FA option for an AI companion holding P0 inner dialog, and a re-auth + soft-disable + export-first account-deletion flow — matching the product's privacy-first positioning rather than a single irreversible delete call.

9. **Encryption-at-rest + retention hygiene as policy, not incidentals (S4-013, S4-012, S4-041).** Column/DB-level encryption for intimate text, a real DB password + externalized key, a documented retention window that keeps analytics but drops raw utterance, and a residual-data policy covering logs/cache/free-space. This is the "your inner cosmos is yours" guarantee made operational.

10. **A reliability "budget" discipline for every synchronous safety/LLM call (S4-007).** Generalize the SAFETY_REVIEW fix into a module-deadline contract: every synchronous gate that stands between a user and a response has a hard, short budget and a deterministic fallback, so no external provider slowness/failure can ever hold a vulnerable user hostage.
