# Inner Cosmos 内宇宙：Codex 愿景实现推进书 v1.0

> 本文档用于指导 Codex 在完成项目初始骨架之后，继续把 Inner Cosmos 从“可运行的课程作业原型”推进为“高度完整、体验连续、接近真实产品形态的 AI 自我共鸣与慢社交平台”。  
> 本文档不是代码规格书，而是产品愿景落地任务书。Codex 应自行完成工程拆解、代码实现、测试与修复，但所有实现必须服务于本文档描述的最终体验。

---

## 0. 使用前提

本文档默认以下基础工程已经完成或正在完成：

1. Spring Boot 项目可启动。
2. MySQL 数据库可连接并初始化。
3. 用户注册、登录、Session 校验可用。
4. Mock LLM 模式可用。
5. Aurora 基础对话页面可用。
6. SSE 流式回复可用。
7. DialogSession / DialogMessage 可保存。
8. 对话结束后可生成基础 MemoryCard。
9. GravityService 可计算情感重力。
10. EchoCapsule 基础 CRUD 可用。
11. 星海广场可展示种子胶囊。
12. SlowLetter 基础状态流可跑通。
13. AI 日志可记录。
14. 至少 10 个物理 HTML 页面可访问。

如果上述能力尚未完成，Codex 应先补齐基础工程，再执行本文档后续任务。

---

## 1. Codex 的总目标

Codex 的任务不是继续堆页面，也不是简单填充 Service，而是把系统推进到以下状态：

> 用户打开 Inner Cosmos 后，能够获得一个连续、完整、有情绪、有记忆、有可视化、有慢社交、有安全边界的真实产品体验。

这意味着：

- 每个页面不再只是占位，而要能完成真实交互。
- 每个模块之间必须打通，不允许孤立功能堆叠。
- Aurora 不只是聊天机器人，而要成为对话式记录入口。
- 用户说出的内容必须能沉淀为记忆、待办、情绪、主题、共鸣体素材。
- 记忆星空不只是图表，而要能表达用户长期内在结构。
- 共鸣体不只是资料卡，而要能作为用户授权后的数字回声参与有限交流。
- 慢信件不只是消息表，而要有投递、送达、回信、拒绝、拉黑、过滤、状态流转。
- 安全边界不是免责声明，而要体现在输入过滤、节律保护、社交防护和隐私隔离中。

---

## 2. 最终产品形态总览

Inner Cosmos 的高度完成版应该呈现为一个完整的四层系统：

```text
第一层：我与 Aurora
用户通过文字或语音，与朋友式 Agent 自然对话，完成每日倾诉、记录、整理和反思。

第二层：我与记忆宇宙
系统将对话沉淀为情绪、事件、认知碎片、待办、关系线索和高重力记忆，并以星空方式可视化。

第三层：我与共鸣体
用户选择部分抽象、脱敏后的记忆与价值线索，生成有限、不完美、可衰减的数字回声。

第四层：我与他人
其他用户先与共鸣体有限交流，若产生共鸣，再通过慢信件郑重联系真实用户。
```

Codex 推进所有功能时，必须始终维持这四层结构。

---

## 3. 总体实现原则

### 3.1 愿景优先，但必须可运行

任何高级功能都不能破坏项目可启动、可演示、可回退。

必须保证：

- Mock 模式始终可运行。
- 远程模型失败时系统不崩溃。
- 页面加载失败时有友好提示。
- 数据为空时有冷启动内容。
- 所有重要操作有错误处理。

### 3.2 模块必须打通

不允许出现以下情况：

- Aurora 能聊天，但聊天内容不进入记忆系统。
- MemoryCard 能生成，但星空页面不读取真实数据。
- EchoCapsule 能创建，但星海广场无法对话。
- 慢信件能发送，但没有状态流转和收信体验。
- AI 日志存在，但不覆盖主要 AI 调用。

每个功能必须进入完整链路。

### 3.3 AI 功能必须有结构化沉淀

所有重要 AI 输出都应至少产生以下一种后续资产：

- DialogMessage。
- MemoryCard。
- ThoughtFragment。
- EmotionTrace。
- TodoItem。
- EchoCapsule 更新。
- SlowLetter 过滤结果。
- AiInteractionLog。

不要让 AI 输出只停留在页面上。

### 3.4 不依赖 Prompt 保护隐私

隐私边界必须由数据访问层和 Service 层保证。

禁止：

- 在共鸣体 Prompt 中直接拼接用户原始日记。
- 在 PersonaChat 中读取 DialogMessage 原文。
- 在慢信件推荐中暴露用户真实身份。
- 把未授权 MemoryCard 用于星海广场。

### 3.5 体验必须温柔，但不能制造依赖

Aurora 应该像一个清醒、温柔、长期熟悉用户的朋友式 Agent。  
但它不能伪装成人、不能承诺永远陪伴、不能成为用户唯一精神支柱。

---

## 4. 阶段推进路线

Codex 应按以下阶段推进。每一阶段完成后，应保证项目仍可运行，并尽量补充测试数据与页面入口。

# Phase A：产品体验整体化

## A.1 目标

把当前多个页面从“功能页面集合”升级为“统一产品空间”。

## A.2 必须完成

### A.2.1 统一视觉设计系统

建立统一前端风格：

- 深色星空背景。
- 柔和蓝紫/灰蓝/金色点缀。
- 玻璃拟态卡片。
- 微光按钮。
- 统一字体大小、行距、卡片圆角、阴影。
- 统一导航栏。
- 统一页面进入动画。
- 统一空状态组件。
- 统一错误状态组件。

Codex 应创建：

```text
/static/css/inner-cosmos.css
/static/js/common.js
/static/js/api.js
/static/js/auth.js
/static/js/ui.js
```

所有页面复用这些基础文件。

### A.2.2 统一导航结构

用户登录后应能在主要页面间自然跳转。

建议导航：

```text
今日对话
今日记录
记忆星空
思维碎纸机
待办
共鸣体
星海广场
慢信箱
AI 日志
个人设置
```

管理员额外看到：

```text
后台管理
举报处理
模型配置
系统日志
```

### A.2.3 Dashboard 升级

Dashboard 不应只是入口页，而应成为用户的“内宇宙首页”。

应展示：

- 今日一句欢迎语。
- 今日情绪天气。
- 最近一次 Aurora 对话摘要。
- 最近高重力记忆。
- 今日待办。
- 共鸣体状态。
- 未读慢信件。
- 最近星海推荐。
- 安全资源入口。

空状态时展示系统种子内容与引导用户开始第一次对话。

## A.3 验收标准

用户登录后，无需阅读说明，也能自然知道从哪里开始、如何记录、如何查看记忆、如何进入星海。

# Phase B：Aurora 朋友式主动对话完整化

## B.1 目标

把 Aurora 从“流式聊天接口”升级为“对话式日记与内在整理 Agent”。

## B.2 必须完成

### B.2.1 对话模式

Aurora 页面应至少支持 4 种模式：

```text
今日倾诉：自由说今天发生了什么。
思维整理：帮助用户拆解一团混乱想法。
睡前复盘：温柔总结今天，降低刺激。
苏格拉底追问：围绕一个信念进行连续澄清。
```

用户选择模式后，Prompt、追问风格、输出结构应不同。

### B.2.2 多条消息主动追问

Aurora 不应每次只给一大段回复。它应能在某些场景下拆成多条自然消息。

示例：

```text
第一条：我听到你说今天“没做成事”，这似乎让你很自责。
第二条：但我也注意到，你说的不是“事情没完成”，而是“我又不行了”。这两个结论不一样。
第三条：我们先不急着解决，你想先讲讲今天最让你卡住的那一刻吗？
```

实现上可以：

- 远程模型返回完整文本后按段落拆分。
- Mock 模式根据模板返回多段。
- SSE 分段输出。

### B.2.3 上下文滑窗与语义锚点

每个 DialogSession 必须维护：

- 最近 3 轮原始对话。
- summaryAnchor：前情摘要。
- highGravityMemoryHints：相关高重力记忆摘要。

当消息超过阈值时：

1. 自动更新 summaryAnchor。
2. 后续 Prompt 不再拼接全部历史。
3. AI 日志记录摘要过程。

### B.2.4 语气个性化

用户可设置 Aurora 的陪伴风格：

```text
温柔安静
理性清晰
朋友式直接
哲学式追问
行动导向
```

Aurora Prompt 应读取 UserProfile 中的偏好。

### B.2.5 对话结束体验

用户点击“结束今日对话”后，不应只关闭会话，而应进入整理流程：

```text
正在整理今天发生的事...
正在提取情绪线索...
正在寻找可能重复出现的主题...
正在生成今日记录卡...
```

整理完成后跳转到今日记录页。

## B.3 验收标准

用户能通过 Aurora 自然完成一次 5~10 分钟的对话，并在结束后看到结构化记录，而不是只留下聊天记录。

# Phase C：语音输入与双轨情绪线索

## C.1 目标

让用户能够用更自然、更低负担的方式倾诉，同时保留语音中的有限元数据。

## C.2 必须完成

### C.2.1 语音输入入口

在 Aurora Chat 页面提供：

- 按住说话 / 点击录音。
- 录音计时。
- 转写中状态。
- 转写文本可编辑。
- 发送前可修改。

V1 可使用浏览器能力或 Mock ASR，后续可接远程 ASR。

### C.2.2 不保存原始音频

默认策略：

- 音频只用于转写。
- 后端不落盘。
- 数据库只保存文本和元数据。

### C.2.3 语音元数据

至少记录：

```text
audioDurationSec
speechRate
pauseCount
longPauseCount
inputConfidence
```

### C.2.4 元数据注入 Aurora

Prompt 中不要做情绪诊断，只做辅助提示：

```text
系统观察到：本段语音较长，存在多次停顿。请在回应时保持温和，避免直接判断用户情绪，并优先帮助用户澄清表达。
```

## C.3 验收标准

用户可以用语音完成一次倾诉，系统能将其转为文字并进入 Aurora 对话与记忆沉淀流程。

---

# Phase D：P1 记忆系统深度化

## D.1 目标

让 Inner Cosmos 真正区别于普通聊天工具：每一次对话都能沉淀成长期可用的自我结构。

## D.2 必须完成

### D.2.1 今日记录卡

对话结束后生成 Daily Record 页面，至少包含：

- 今日主题。
- 今日事件。
- 核心情绪。
- 触发因素。
- 认知碎片。
- 关系提及。
- 待办事项。
- Aurora 总结。
- 是否建议生成共鸣体素材。

如数据库尚未单独建 DailyRecord，可用 MemoryCard + DialogSession summary 实现。

### D.2.2 MemoryCard 质量提升

MemoryCard 不应只是 title + summary，应包括：

- 类型。
- 情绪标签。
- 关键词。
- 相关人物。
- 来源对话。
- 情绪强度。
- 用户重要性。
- 重复次数。
- 最近触发。
- 情感重力。
- 可授权状态。

### D.2.3 ThoughtFragment 分类

把用户表达拆成：

```text
事实 FACT
感受 FEELING
担心 WORRY
需求 NEED
信念 BELIEF
行动 ACTION
```

页面应以卡片方式展示。

### D.2.4 TodoItem 转化

从对话中自动提取待办，用户可以：

- 接受。
- 修改。
- 删除。
- 标记完成。
- 推迟。

待办页面要能按：

```text
今日
本周
来自哪次对话
情绪关联
优先级
```

进行展示。

### D.2.5 关系人物线索

从用户提到的人中记录 RelationMention：

- 不要求做复杂社交图谱。
- 只需记录“某人/某类人/某种关系”与情绪的关联。

示例：

```text
人物：老师
关联情绪：紧张、压抑
关联主题：权威、认可、表达困难
```

### D.2.6 长期主题聚合

系统应能根据 MemoryCard 的关键词和类型，生成 MemoryTheme：

```text
自责与任务推进
面对权威的压抑
关系中的讨好
对未来的不确定感
被看见的渴望
```

如果暂不建表，可在 Service 层聚合展示。

## D.3 验收标准

用户一次对话后，至少生成 3 类结构化资产：MemoryCard、ThoughtFragment、TodoItem 或 EmotionTrace。多次对话后，系统能展示重复主题。

# Phase E：情感重力与记忆星空成熟化

## E.1 目标

让“记忆星空”从普通图表升级为产品核心体验。

## E.2 必须完成

### E.2.1 情感重力公式落地

必须使用非线性公式：

```text
G(t) = ln(1 + αI + βR + γU + δM) × e^(-λt)
```

并允许在配置中调整权重。

### E.2.2 重力重算机制

触发条件：

- 新 MemoryCard 生成。
- 用户标记重要性。
- 某主题再次被提及。
- 每日/每周结算。

### E.2.3 星空可视化

记忆星空页面至少支持：

- 按类型筛选。
- 按情绪筛选。
- 按时间筛选。
- 星体大小表示 gravity。
- 星体亮度表示 freshness。
- 点击星体查看详情。
- 显示相关 ThoughtFragment。
- 显示是否可编织为共鸣体。

### E.2.4 记忆详情页/弹窗

点击星体后展示：

- 标题。
- 摘要。
- 来源日期。
- 来源对话入口。
- 相关情绪。
- 相关待办。
- Aurora 观察。
- 用户重要性调节。
- “放入星海”按钮。

### E.2.5 高重力记忆提醒

Dashboard 应显示最近高重力记忆，但语气必须温和：

```text
最近这颗星比较亮，不一定要现在处理，只是它似乎值得被你看见。
```

## E.3 验收标准

用户能够直观看到自己的记忆星空，并理解每颗星为何重要，而不是只看到随机气泡图。

# Phase F：思维碎纸机完整体验

## F.1 目标

把 Thought Shredder 做成一个强体验模块：允许用户混乱表达，并把混乱转化为少量可沉淀的理性内核。

## F.2 必须完成

### F.2.1 输入体验

页面应支持：

- 大文本框。
- 情绪强度选择。
- 是否保存原文选项。
- “粉碎并沉淀”按钮。
- 粉碎动画或简化视觉反馈。

### F.2.2 输出结构

AI 输出至少包含：

```text
混乱内容中的核心感受
背后的可能需求
可以丢弃的噪声
值得留下的一句话
建议生成的 MemoryCard
建议生成的 TodoItem
```

### F.2.3 隐私模式

用户可以选择：

```text
保留原始输入
只保留整理结果
完全不保存，只显示一次
```

### F.2.4 沉淀流程

用户确认后，将结果写入：

- MemoryCard。
- ThoughtFragment。
- EmotionTrace。

## F.3 验收标准

用户输入一段混乱文本后，系统能生成清晰、温柔、可保存的整理结果，并进入记忆系统。

---

# Phase G：共鸣体系统成熟化

## G.1 目标

让 EchoCapsule 成为“授权数字回声”，而不是普通资料卡。

## G.2 必须完成

### G.2.1 共鸣体创建流程

用户可以从三种入口创建或更新共鸣体：

1. 手动进入共鸣体设置页。
2. 从 MemoryCard 详情页点击“放入星海”。
3. Aurora 在对话或总结后主动建议编织。

### G.2.2 DataMaskingService

生成共鸣体前必须脱敏：

- 删除真实姓名。
- 删除具体地点。
- 删除具体联系方式。
- 删除精确时间。
- 删除学校/公司等敏感上下文，除非用户明确授权。
- 将具体事件抽象为精神主题。

示例：

```text
原始：今天在北邮和某老师沟通项目时，我觉得自己完全不敢反驳。
抽象：在面对权威与评价时，容易感到表达受限。
```

### G.2.3 共鸣体资料页

展示：

- 昵称。
- 简介。
- 公开标签。
- 精神主题。
- 回声能量。
- 可聊话题。
- 不可聊边界。
- 写信入口。

### G.2.4 回声衰减

共鸣体展示应体现 freshness：

- 最近更新：明亮。
- 较久未更新：柔和。
- 长期未更新：安静。
- 用户关闭社交：隐藏。

### G.2.5 对话限制

每个访问者与同一共鸣体每日最多 3~5 轮。

超限后引导慢信。

### G.2.6 用户控制权

用户必须能：

- 查看自己的共鸣体包含哪些授权信息。
- 删除某条授权记忆。
- 关闭共鸣体。
- 修改可聊话题。
- 修改社交可达状态。

## G.3 验收标准

用户能从自己的记忆生成一个可展示、可对话、可关闭、可控制边界的共鸣体。

---

# Phase H：星海广场与冷启动体验

## H.1 目标

让用户第一次进入星海广场时就感到“这里有人、这里有精神气质、这里值得探索”。

## H.2 必须完成

### H.2.1 种子共鸣体

初始化至少 5 个 SEED_CAPSULE：

```text
斯多葛信使
苏格拉底之问
庄周之梦
存在主义旅人
热烈的画家
```

每个种子共鸣体要有：

- 独立头像/视觉符号。
- 独立简介。
- 独立 Prompt。
- 独立标签。
- 独立可聊话题。
- 独立限制语。

### H.2.2 广场展示

星海广场应支持：

- 推荐共鸣体。
- 最新共鸣体。
- 高能量共鸣体。
- 种子共鸣体。
- 按标签筛选。
- 搜索。

### H.2.3 空状态处理

如果没有真实用户共鸣体，页面仍然展示种子共鸣体与引导。

### H.2.4 推荐逻辑

V1 不要求复杂算法，但应有基础排序：

```text
优先展示：种子胶囊 + 最近更新 + 标签匹配 + 高 freshness + 未被用户拉黑
```

### H.2.5 广场安全

不展示：

- 被举报待处理的胶囊。
- 用户设置为 RED/CLOSED 的胶囊。
- 被当前用户拉黑的胶囊。

## H.3 验收标准

老师或新用户第一次打开星海广场时，不会看到空页面，而会立即理解“共鸣体慢社交”的概念。

# Phase J：慢信件系统高完成度化

## J.1 目标

让慢信件成为产品最有仪式感、最接近真实连接的社交模块。

## J.2 必须完成

### J.2.1 写信体验

## J.3 验收标准

# Phase K：安全、节律与现实拉回

## K.1 目标

保证系统不越界、不诱导依赖、不放大风险。

## K.2 必须完成

### K.2.1 SafetyBoundaryFilter

所有用户输入进入 AI 前先经过安全过滤。

风险等级：

```text
LOW：正常
MEDIUM：普通回复 + 支持资源提醒
HIGH：停止普通 Agent 回复，进入固定安全流程
```

### K.2.2 高风险固定流程

高风险场景不允许大模型自由发挥。

系统应：

- 温和回应。
- 鼓励联系现实可信任的人。
- 提供支持资源页。
- 避免继续沉浸式聊天。
- 记录 SafetyEvent。

### K.2.3 节律保护

触发条件：

- 单次对话超过时间阈值。
- 消息数超过阈值。
- Token 估算超过阈值。
- 用户短时间内大量发送。

回应：

```text
我们已经聊了很久了。为了让这些内容真正沉淀下来，我建议你先暂停一下。我会帮你整理今天的重要内容，你可以喝点水、走一走，稍后再回来。
```

### K.2.4 支持资源页

页面应温和呈现：

- 非医疗声明。
- 寻求现实帮助的建议。
- 可信任的人。
- 专业支持资源。
- 自我稳定小步骤。

### K.2.5 社交安全

必须支持：

- 拉黑。
- 举报。
- 隐藏共鸣体。
- 限制信件。
- 管理员审核。

## K.3 验收标准

系统在高风险输入、长时间对话、恶意信件、举报等场景下都有明确、可运行的处理流程。

## 8. 核心文案方向

Codex 在生成页面文案时，应遵循以下气质。

### 8.1 Aurora 文案

可以说：

```text
我听见你刚才把这件事说得很轻，但它好像并不轻。
```

```text
我们先不急着解决，先把它看清楚一点。
```

```text
这不一定说明你失败了，也许只是你把一个事件太快地解释成了自我评价。
```

避免说：

```text
你有焦虑症。
你应该立刻这样做。
我永远都在。
只有我懂你。
```

### 8.2 共鸣体文案

可以说：

```text
我只是这个人的一部分回声。
```

```text
如果这段对话让你停下来了，也许你可以写一封慢信。
```

避免说：

```text
我就是他本人。
我完全了解他的所有经历。
```

### 8.3 安全文案

可以说：

```text
如果你现在正在经历强烈痛苦，请优先联系现实中可信任的人。你不需要独自承受这一刻。
```

避免说：

```text
系统检测到你有严重心理问题。
```

## 9. 严禁实现的反方向

Codex 不应把产品做成以下形态：

1. 普通 ChatGPT 套壳。
2. 普通日记增删改查。
3. 心理诊断网站。
4. MBTI / 依恋风格标签机器。
5. 即时陌生人聊天软件。
6. AI 恋人或过度亲密陪伴。
7. 没有隐私边界的数字替身。
8. 全靠后台 Agent 自动互聊的烧钱系统。
9. 只有炫酷图表、没有结构化沉淀的展示页。
10. 只靠 Prompt 承诺安全的系统。

## 10. 最终验收口径

Codex 完成后，请按以下问题自检：

1. 用户是否能自然完成一次“说出今天”的体验？
2. 系统是否把这次表达沉淀成了结构化记忆？
3. 用户是否能在星空中看到这段记忆？
4. 这段记忆是否能经过授权进入共鸣体？
5. 其他用户是否能与共鸣体有限交流？
6. 交流是否能自然转向慢信？
7. 用户是否能控制隐私与社交边界？
8. 系统是否在高风险内容和过度使用时有保护机制？
9. 没有 API Key 时是否仍然能演示完整流程？
10. 这个产品看起来是否明显区别于普通 AI 聊天网站？

如果以上答案都是“是”，说明 Inner Cosmos 已经从课程原型走向了真正的产品雏形。
