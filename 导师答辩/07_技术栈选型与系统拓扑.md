# 导师答辩：系统技术栈选型与拓扑规范

> 本文档用于系统性介绍 Inner Cosmos 全栈的技术选型逻辑、前后端数据流向与系统运行拓扑。

---

## 1. 全栈技术选型与学术合理性 (Technology Choice)

项目在进行架构设计时，坚决避免使用未经工业界或学术界验证的试验性框架，而是采用了一套成熟度极高、契合高并发分布式标准的云原生技术栈：

*   **后端核心：Java 21 + Spring Boot 3.5.x**
    *   *选型逻辑*：Java 作为老牌的工程语言，具备极佳的强类型安全与丰富的企业级生态。使用 Java 21 LTS 保证了系统可以使用现代的协程/虚拟线程理念（虽然此处仍使用标准线程池），并且支持了诸如 `SequencedCollection`（简化顺序集合的操作如 `getFirst()`）等先进语法，让代码可读性极高。
    *   Spring Boot 3.5.x 提供了成熟的依赖注入、Spring Validation 参数校验以及统一的全局异常捕获（`@RestControllerAdvice`），缩短了开发周期。
*   **前端核心：React 19 + TypeScript + Vite**
    *   *选型逻辑*：React 19 是前端最前沿的声明式 UI 渲染框架。结合 TypeScript 强类型约束，保证了前端代码的零运行时崩溃，避免了 JavaScript 弱类型导致的“属性未定义”等致命错误。
    *   Vite 作为超快速的构建工具，提供秒级的热重载开发体验（HMR），并能将代码压缩打包为轻量级的静态资源，交由 Spring Boot 后台直接托管，大幅度降低部署架构的复杂性。
*   **持久化控制：PostgreSQL 16 + pgvector + Flyway**
    *   *选型逻辑*：PostgreSQL 16 是开源关系型数据库的王者。我们引入了 `pgvector` 插件，这意味着数据库天然支持保存 1536 维的向量数据，并能通过 HNSW 索引进行高速相似度（余弦夹角）匹配，直接为未来的“胶囊匹配”与“向量语义 RAG”留下了原生扩展接口。
    *   Flyway 作为持久化层版本控制系统，能自动从 `db/migration/postgresql` 下读取 SQL 脚本演进 Schema，使得任何组员拉下代码都能一键重建最新的数据库结构，杜绝了手动修改数据库引发的 Schema 漂移。
*   **分布式协同：Redis Cluster**
    *   *选型逻辑*：用于存储多节点分布式 Session（实现单点登录与状态共享）、进行全局 API 访问限流，并通过 Redis Stream 协调多节点 SSE 的 Token 异步分发，起到了系统通信总线（Message Bus）的作用。

## 2. 系统拓扑与流量流向 (Data Flow Topology)

```text
[ 用户 / 游客 / 访客浏览器 (React 19 Frontend) ]
                    │  (通过 Gateway API 进入集群)
                    ▼
       [ API 网关负载均衡 (Gateway API) ] (提供 HTTPS TLS 卸载与限流)
                    │
       ┌────────────┴────────────┐ (根据路由转发)
       ▼                         ▼
[ API 副本 Pod A ]        [ API 副本 Pod B ] (多 Pod api 角色集群)
       │                         │
       ├─────────────────────────┴─────────┐ (通过 Redis Stream 协调异步推送)
       ▼                                   ▼
 [ Redis 集群 (Session & Lock) ]   [ Redis Stream (SSE Fan-out) ]
       │                                   ▲
       ├───────────────────────────────────┤
       ▼                                   │ (触发 AI 处理)
[ PG 数据库 (pgvector) ]        [ Worker 副本 (AI 线程池) ] (负责 AI 生成)
```

1.  **用户注册/登录**：
    前端发送 HTTPS POST 密码采用 `BCryptPasswordEncoder` 强加密哈希校验。会话令牌写入 HttpSession，并在后台由 Spring Session 自动持久化同步至 **Redis** 集群。
2.  **Aurora 倾诉对话**：
    用户发起对话，连接发送到 `api` 节点，`api` 节点将对话事件写入数据库并丢给 `worker`。`worker` 驱动 AI 大模型（或 Mock 客户端）分词提取语义，将流式数据输出到 **Redis Stream**，最终由用户所在的 `api` 连接节点通过 **SSE (Server-Sent Events)** 管道泵送回浏览器。
3.  **定时慢信投递**：
    [scheduler](file:///d:/code/inner%20cosmos/deploy/k8s/overlays/academy-eks/scheduler-deployment.yml) 副本每分钟被 Cron 唤醒，运行 `LetterDeliveryJob`，扫描 `tb_slow_letter` 表，将处于飞越中（`FLYING`）且到达视差延迟时段的信件状态，修改为已送达（`DELIVERED`），并向用户发送全局 WebSocket/轮询通知。
