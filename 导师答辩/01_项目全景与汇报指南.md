# 导师答辩：Inner Cosmos 项目全景与汇报指南

> 本文档专为项目答辩和导师汇报设计，旨在系统性地向评审阐述 Inner Cosmos 的核心价值与系统架构。

---

## 1. 为什么做这个项目？(痛点与愿景)

*   **当前社交媒体的困境**：主流平台（WeChat, X, HelloTalk）主打“即时反馈（Instant Gratification）”，算法泛滥，导致用户产生严重的“快社交疲劳”、低效沟通以及隐私泄露。
*   **AI 伴侣系统的痛点**：现有的 AI 聊天机器人（如 Character.ai, Replika）多为“一语一答”的对话玩具，记忆浅且不具备数据所有权，无法将伴侣体验转化为真实的、有保护的社交人际资产。
*   **内宇宙 (Inner Cosmos) 的解决方案**：
    我们通过一个长期的 AI 伴侣 **Aurora** 与用户深度对话，将其精神特质沉淀为结构化且人机协同纠错的“记忆卡片（MemoryCard）”。在用户显式授权和数据隔离的前提下，将记忆编译为脱敏的“共鸣体（Echo Capsule）”。共鸣体替代用户在社交广场进行“灵魂 vibe 预匹配”，若契合则通过具有真实时差延迟的“慢信（SlowLetter）”构建真正的高质量人际连接。

---

## 2. 系统技术架构全景 (System Architecture)

项目采用高度内聚、健壮的**模块化单体（Modular Monolith）**设计，各层技术选型如下：

```text
[ 前端 AppShell ] React 19 + TS + Vite (单页五空间架构，适配 Capacitor 8 移动容器)
       │ (SSE 流式推送 / RESTful APIs)
       ▼
[ 后端 API & Worker ] Java 21 + Spring Boot 3.5.x + Spring Security
       │ (MyBatis-Plus ORM / Flyway 数据迁移版本控制)
       ├──► [ 数据存储 ] PostgreSQL 16 (带 pgvector 向量插件，用于胶囊匹配)
       └──► [ 缓存/分发 ] Redis Cluster (管理会话、Leases 锁、SSE Redis Stream 消息队列)
```

*   **Java 21 新特性支持**：充分利用了 `SequencedCollection`、`ExecutorService` 的 `AutoCloseable` 语法，代码规范高。
*   **双轨运行模式**：开发环境默认支持 H2 内存数据库与 Mock AI Client，实现零 API Key、零依赖极速启动；生产环境通过配置中心热插拔接入真实 PG 与 Redis 集群，并部署至 Kubernetes (EKS)。

## 3. 近期四大技术攻坚成果

1.  **体验层暖色系重构 (Phase 0)**：彻底告别冷色深绿“AI 控制台”主题，引入以暖褐色 (`#211A18`) 为基底的 Morandi 温暖色系，建立 CSS 变量与 `@layer tokens` 系统，支持七时段（dawn 至 deep-night）的平滑 lerp 渐变，且实现全局点击涟漪与星尘粒子动效。编写了 Playwright 自动化测试清剿所有交互死按钮。
2.  **事实提取与纠错闭环 (Claim Correction)**：开发了确定性精度优先事实提取器，在对话结束时自动生成 `ThoughtFragment`（分为事实、感受、信念、行动），暂存在 Staging 区供用户编辑和确认才生效。
3.  **共鸣体 Genome 编译器 v2 升级**：生成的共鸣体声线与场景数据中，全部载有 `voiceEvidence` 数组，精确溯源至用户的原始记忆卡片 ID，防止 AI 产生幻觉，并设立了 `GenomeCompilerGroundednessEvaluationTest` 物理隔离测试门禁。
4.  **防提示词注入安全硬化 (Security Redaction)**：为共鸣体对话的最后出站通道强力注入了脱敏拦截器，通过 `DataMaskingUtils.maskContact` 彻底阻断任何通过 Prompt Injection 窃取主人邮箱或联系方式的企图。
