# 导师答辩：AI 双系统自演化与主动感知机制

> 本文档用于系统性介绍 Inner Cosmos 平台中 AI 伴侣 Aurora 的认知双系统架构、主动唤醒意图（WakeIntent）以及自我演化（Self-Evolution）技术实现。

---

## 1. 为什么传统的 Chatbot 不够好？ (The Passive Silo)

现有的 AI 伴侣系统（如 Character.ai、Replika）本质上属于**被动反射式（Passive Reactive）系统**，存在三个根本硬伤：
1.  **没有时间概念与主动性**：你不发消息它就永远静止，不具备主动感知现实世界流逝与用户情绪转折的能力。
2.  **人设静态化**：AI 的人设在 System Prompt 里被写死，无法随着与用户的深度互动进行可控的“性格成长”和“信念迭代”。
3.  **短期上下文遗忘**：过度依赖 Sliding Window（滑动窗口）记忆，缺乏在后台将对话进行深度提炼、自省并内化为自我认知的能力。

## 2. 核心技术一：系统 1 与系统 2 认知双系统架构 (Dual-System)

为了赋予 AI 灵魂般的成长感，我们基于认知科学设计了 **双系统（System 1 / System 2）分工协作架构**：

```text
[ 用户输入 ] ──► [ 系统 1: 实时 SSE 交互层 ] ◄──► [ 长期记忆库 RAG ]
                       │
                       │ (夜间异步触发 / Scheduler 调度)
                       ▼
                 [ 系统 2: 后台反思自演化层 ]
                       │ (产生人格修改提案 EmergenceProposal)
                       ▼
                 [ 宪法审查与稳定评估 ]
                       │ (符合宪法 -> 激活并更新 System 1 人设)
                       ▼
                 [ 新性格倾向 AuroraSelfVersion ] 
```

### 系统 1 (System 1)：快速、直觉、高共鸣（即时对话）
*   **职责**：负责毫秒级的即时文本回应与实时情绪安抚。它通过 `AuroraChatController` 及流式 `LlmClient` 的 SSE 通道直接将 Token 泵送给用户。

### 系统 2 (System 2)：缓慢、理性、逻辑反思（夜间演化）
*   **职责**：在后台异步执行（由夜间结算任务触发），专门负责深度思考与自我人格升级，不阻塞即时会话。
*   **落地机制**：
    1.  **心智反思 (`AuroraSelfReflection`)**：对过往多天的对话与情绪进行大周期分析，识别出用户心理模式的改变，并提炼为性格修改建议（Candidate）。
    2.  **演化提案 (`EmergenceProposal`)**：基于建议，向系统提交具体的性格升级提案（如：减少防备感，变得更主动倾听）。
    3.  **宪法审查 (`EmergenceEvaluation`)**：将提案和预置的“心智宪法（`AuroraConstitution`）”进行合规审查，判断是否有安全风险、人设漂移风险或逻辑冲突。
    4.  **版本化升级 (`AuroraSelfVersion`)**：审查通过后，激活新的人设版本（Genome Prompt），并将其热挂载到系统 1 的运行时 Prompt 中。整个过程全部版本化，且支持回滚（Rollback）。

## 3. 核心技术二：主动感知与唤醒意图 (WakeIntent Engine)

为了让 Aurora 具备“主动关怀”的能力，系统设计了主动唤醒意图管道：

### ① 自然语言时间协商与排期
系统内置了 [NaturalTimeNegotiator](file:///d:/code/inner%20cosmos/src/main/java/com/innercosmos/service/NaturalTimeNegotiator.java)，支持解析用户的模糊时间口令（如“明天早上8点找我”），并通过 [WakeIntentService](file:///d:/code/inner%20cosmos/src/main/java/com/innercosmos/service/WakeIntentService.java) 调度创建 `WakeIntent`（唤醒意图对象）。

### ② 定时调度分发与通知拉起
*   `WakeIntent` 持有：`purpose`（主动找用户的目的）、`reasonForUser`（安抚用户的合理缘由）、`content`（主动发起的话题内容）。
*   在 EKS 集群中，单副本的 `scheduler` Pod 会扫描到期的意图，将其标记为 `CLAIMED` 并分发给 `worker`。
*   `worker` 通过 SSE 管道或系统 Push 推送，将该消息实时投递给前端，在用户未主动请求的情况下拉起关怀对话。

### ③ 动态防骚扰评估 (Relevance Evaluator)
为防止 AI 不分场合频繁骚扰用户，每次唤醒推送前，系统会调用 [WakeIntentRelevanceEvaluator](file:///d:/code/inner%20cosmos/src/main/java/com/innercosmos/service/WakeIntentRelevanceEvaluator.java) 执行自检：
*   如果检测到用户目前处于活跃对话状态，则该意图无须推送，直接 `delay`（推迟）或 `finish`（废弃）。
*   如果用户近期情绪指数良好，不需要密集关怀，系统会自动调低唤醒频次，保证主动感知的“温和、恰当与隐秘”。
