# 导师答辩：AI 系统与多模型适配架构

> 本文档用于系统性介绍 Inner Cosmos 后台 AI 系统的模块化设计、多模型热插拔适配以及流式 SSE 交互机制。

---

## 1. 适配器模式对接多 LLM 提供商 (LlmClient Adapter)

为了解决大模型市场上 API 多变、频繁切换的问题，系统在 [ai/client/](file:///d:/code/inner%20cosmos/src/main/java/com/innercosmos/ai/client/) 中采用了标准的**适配器模式 (Adapter Pattern)**，定义了统一的 [LlmClient](file:///d:/code/inner%20cosmos/src/main/java/com/innercosmos/ai/client/LlmClient.java) 接口。

*   **统一的客户端接口**：
    ```java
    public interface LlmClient {
        String chat(LlmRequest request);
        Flux<String> streamChat(LlmRequest request);
        boolean isHealthy();
    }
    ```
*   **支持的适配器实现**：
    1.  **`MockLlmClient` (默认)**：无需 API Key 即可启动，返回系统内置的哲学式、自省式的高质量模拟回复。这极大地保护了本地开发和测试环境，避免产生不必要的 API 调用开销，支持本地快速调试。
    2.  **`GlmLlmClient` (智谱 GLM)**：对接 GLM-4 相关的流式和非流式接口。
    3.  **`MiniMaxLlmClient` (稀宇 MiniMax)**：对接 MiniMax 模型，优化富情感伴侣交互。
    4.  **`DeepSeekLlmClient` (深度求索 DeepSeek)**：对接 DeepSeek 模型，实现低成本、高精度的推理和特征提取。
    5.  **`OpenAiCompatibleLlmClient`**：通用的 OpenAI 标准格式适配器，支持任何兼容厂商（如 One-API 等聚合网关）。

## 2. 动态提示词与策略模式 (PromptBuilder & Strategy)

在交互时，不同场景（如伴侣 Aurora、碎纸机、共鸣体）需要不同的人设和上下文。我们采用**策略模式 (Strategy Pattern)** 来管理回复：
*   **`AgentReplyStrategy` 接口**：定义了不同 AI 角色的回复机制。
*   **提示词构建器 `PromptBuilder`**：
    为了向模型提供清晰的上下文，`PromptBuilder` 会动态抓取用户的：
    *   `SystemPrompt`：设定的系统边界与人设（如 Aurora 伴侣设定）。
    *   `RecentMessages`：最近几轮的对话历史（实现短期记忆）。
    *   `AuthorizedMemories`：被编译授权的长期记忆卡片（实现长期记忆 RAG 检索）。
    *   `SafetyBoundaryRules`：高危安全边界准则。
    拼接后生成最终结构化的 JSON/Markdown 提示词发给 LlmClient。

## 3. SSE (Server-Sent Events) 实时流式推送机制

对于大模型这种长文本生成的应用，如果等待全部生成完毕再返回，会造成长达数十秒的白屏，极大地损害用户体验。
为了实现极致的顺滑对话，项目采用了 **Server-Sent Events (SSE)** 流式推送：
*   **非阻塞反应式生成**：
    在 `AuroraChatController` 中，使用 Spring WebFlux 的 `Flux<String>` 异步接收 `LlmClient` 吐出的 Token，并通过 `SseEmitter` 建立与 React 前端的长连接，将字词一个个实时推送到前端页面上。
*   **跨 Pod SSE Redis 消息分发 (SSE Fan-out)**：
    在 Kubernetes 多副本部署的分布式场景下，用户的 SSE 长连接可能建立在 Pod A 上，而 AI 异步生成的计算可能运行在 Pod B 上。我们通过 **Redis Stream** 构建了跨实例的 SSE 消息订阅发布系统（SSE Fan-out）。当 Pod B 的 AI 生成出一个 Token 时，它会发布到 Redis Stream，Pod A 收到订阅通知后，再通过本地的 SSE 管道推送给真实客户端，完美解决了分布式架构下的长连接一致性问题。

## 4. 情绪光谱衍生与 PseudoSemantic 情感引擎

在没有真实大模型 key 的 Mock 开发模式下，为了让伴侣的回答带有情绪波动，并让前端的“记忆星空”呈现出丰富的色彩，系统实现了一套 **轻量级中文情感分词与评估引擎**：
*   **`PseudoSemanticAnalyzer` (伪语义分析器)**：
    利用 `ChineseSentimentLexicon`（中文情感词典）和 `ChineseIntensifiers`（程度副词词典），通过对文本进行快速的子串包含扫描，评估出一段话中的**积极/消极情绪系数**。
*   **情绪光谱推导 (`EmotionSpectrumDeriver`)**：
    将评估的情感分值映射到包括**喜悦、悲伤、焦虑、愤怒、平静**在内的八维情绪光谱中，并将其推导为相应的 RGB 颜色代码，从而传递给前端，用于渲染记忆卡片的颜色深度和“情感重力”值，做到了 Mock 模式下高度还原的界面联动。
